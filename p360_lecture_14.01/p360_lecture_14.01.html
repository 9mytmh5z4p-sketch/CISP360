Reference Variables — Review

A reference and a pointer — same memory, different rules?

  • Both give access to another variable’s memory.
  • A reference is an alias — it is the variable. A pointer holds the variable’s address.

Total slides: 4

Reference as alias

int score = 88;
int& ref = score;   // ref is an alias for score

ref = 95;           // modifies score directly
cout << score;      // prints 95
  • Declared with & in the type, not the assignment
  • Must be initialized at declaration — cannot be reseated
  • No dereference syntax needed; just use the name

Reference vs. pointer at a glance

Reference Pointer
Syntax to access ref *ptr
Can be null no yes
Can be reseated no yes
Must initialize yes no (but should)
Use when alias for one thing optional, nullable, or reassignable

[Graphic suggestion: side-by-side memory diagram — ref points to the same box as the original variable; pointer box holds an address value that points to the variable]

Summary

  • A reference is an alias: no address arithmetic, no dereferencing, no null.
  • Use a reference when you need an alternate name for something that always exists; use a pointer when the target may change or may not exist.

What’s the Output?

Exercise 1 — What does this print?

  • Two variables, one reference, one modification. What does each cout print?

Total slides: 4 Give students 60 seconds to work this out on paper before revealing.

Exercise 1 — Code

#include <iostream>
using namespace std;

int main() {
    int x = 10;
    int& r = x;

    r = 20;
    cout << x << "\n";    // line A

    int y = 30;
    int* p = &x;
    p = &y;
    *p = 50;

    cout << x << "\n";    // line B
    cout << y << "\n";    // line C

    return 0;
}

Line A: 20 — r is an alias for x; assigning to r changes x. Line B: 20 — p was reseated to y; x is untouched. Line C: 50 — *p = 50 modifies y through the pointer.

Exercise 2 — What does this print?

  • A reference parameter and a pointer parameter side by side. Trace each call.

Exercise 2 — Code

#include <iostream>
using namespace std;

void doubleRef(int& n) { n *= 2; }
void doublePtr(int* n) { *n *= 2; }

int main() {
    int a = 5;
    int b = 5;

    doubleRef(a);
    doublePtr(&b);

    cout << a << "\n";    // line A
    cout << b << "\n";    // line B

    int c = 3;
    int* p = &c;
    doublePtr(p);
    cout << c << "\n";    // line C

    return 0;
}

Line A: 10 — doubleRef takes a reference; a is modified in place. Line B: 10 — doublePtr takes a pointer; n = 2 modifies b through the pointer. Line C: 6 — p holds the address of c; doublePtr(p) passes that address, n = 2 modifies c.

Heap vs. Stack

Where does memory come from after the function returns?

  • Every local variable lives on the stack — and disappears when its function ends.
  • The heap is a second pool of memory you manage yourself, with no automatic cleanup.

Total slides: 5

The stack

void foo() {
    int x = 42;   // allocated when foo() is called
    // ...
}                 // x is destroyed here — automatically
  • Memory is allocated and freed automatically as functions are called and return
  • Fast, but limited in size (~1–8 MB typical)
  • Lifetime is tied to scope — you cannot return a pointer to a local variable

The heap

int* p = new int(42);   // allocated on the heap
// ...
delete p;               // freed only when YOU say so
  • Memory persists until you explicitly release it with delete
  • Much larger than the stack — limited by available RAM
  • Lifetime is independent of scope — survives function return

Stack vs. heap at a glance

Stack Heap
Allocation automatic manual (new)
Deallocation automatic manual (delete)
Lifetime tied to scope until delete is called
Size limit small (~1–8 MB) large (RAM limit)
Speed fast slightly slower

[Graphic suggestion: two-region memory diagram — stack grows downward from top, heap grows upward from bottom, with arrows showing growth directions]

Summary

  • The stack handles memory automatically but is scope-bound and size-limited.
  • The heap lets you control lifetime and size — at the cost of managing it yourself.

new and delete

How do you ask for memory that outlasts the function?

  • new reserves heap memory and returns a pointer to it.
  • delete releases that memory back to the system — you are responsible for calling it.

Total slides: 6

new — allocating a single value

int* p = new int;       // allocate — uninitialized
int* q = new int(42);   // allocate and initialize to 42

cout << *q;             // 42
  • new returns a pointer to the allocated memory
  • The pointer itself is a local variable — the memory it points to is on the heap
  • Type determines how many bytes are reserved

delete — releasing a single value

int* p = new int(42);

// ... use p ...

delete p;       // release the heap memory
p = nullptr;    // prevent accidental reuse
  • delete frees the heap block the pointer points to
  • The pointer variable still exists after delete — set it to nullptr immediately
  • Calling delete on the same pointer twice is undefined behavior

Always check the allocation

int* p = new (nothrow) int(42);

if (p == nullptr) {
    cerr << "Allocation failed\n";
    return 1;
}

cout << *p << "\n";
delete p;
p = nullptr;

nothrow version returns nullptr instead of throwing std::bad_alloc. Good habit for systems code. In most course programs, allocation rarely fails — but the pattern is worth showing.

The complete pattern

#include <iostream>
using namespace std;

int main() {
    int* score = new int(0);

    cout << "Enter score: ";
    cin >> *score;

    cout << "You entered: " << *score << "\n";

    delete score;
    score = nullptr;

    return 0;
}

Walk through each step: allocate, use through dereference, delete, null the pointer. This is the skeleton every heap program follows.

Summary

  • new allocates heap memory and returns a pointer; delete releases it.
  • Always set the pointer to nullptr after delete — leaving a dangling pointer is a bug waiting to happen.

Dynamic Arrays

What if you don’t know the array size until the program runs?

  • Stack arrays require a compile-time size. Dynamic arrays let the size come from the user or a file.
  • new[] allocates the array on the heap; delete[] releases it.

Total slides: 5

Allocating a dynamic array

int n;
cout << "How many scores? ";
cin >> n;

int* scores = new int[n];   // size decided at runtime
  • n does not need to be a constant — any integer expression works
  • scores is a pointer to the first element, just like a regular array name
  • All the pointer arithmetic and subscript syntax from week 13 applies

Using and releasing a dynamic array

for (int i = 0; i < n; i++) {
    cout << "Score " << (i + 1) << ": ";
    cin >> scores[i];
}

// ... process ...

delete[] scores;    // must use delete[], not delete
scores = nullptr;
  • Use delete[] — not delete — for arrays allocated with new[]
  • Mixing them is undefined behavior
  • Set to nullptr after release

Complete example

#include <iostream>
using namespace std;

int main() {
    int n;
    cout << "How many scores? ";
    cin >> n;

    int* scores = new int[n];

    for (int i = 0; i < n; i++) {
        cout << "Score " << (i + 1) << ": ";
        cin >> scores[i];
    }

    int sum = 0;
    for (int i = 0; i < n; i++)
        sum += scores[i];

    cout << "Average: " << (sum / n) << "\n";

    delete[] scores;
    scores = nullptr;
    return 0;
}

Summary

  • Use new int[n] when the array size is not known at compile time.
  • Always pair new[] with delete[] — and set the pointer to nullptr after release.

Key Takeaways

  • A reference is an alias — same memory, no pointer syntax needed
  • Stack memory is automatic and scope-bound; heap memory is manual and persistent
  • new allocates heap memory; delete releases it — you own the cleanup
  • Use new[] and delete[] for arrays; set pointers to nullptr after release
