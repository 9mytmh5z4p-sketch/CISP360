File I/O

Data That Outlasts The Program

  • Variables disappear when the program ends.
  • Files let a program save data, load it later, and share it with other programs.

Slides 3-29

Files As Streams

  • cin reads from the keyboard.
  • cout writes to the terminal.
  • File streams use the same idea, but the stream is connected to a file.

The operators are familiar: >> extracts data, and << inserts data.

The File I/O Process

  1. Include <fstream>.
  2. Declare a file stream object.
  3. Open the file.
  4. Check that the open worked.
  5. Read or write data.
  6. Close the file.

Every file program follows this pattern.

The fstream Library

Type Direction Common Use
ifstream input read from a file
ofstream output write to a file
fstream input and output read and write with one stream

Use the narrowest stream type that matches the job.

Writing To A File

#include <fstream>
#include <iostream>
using namespace std;

int main() {
    ofstream outFile("scores.txt");

    outFile << "Alice " << 95 << endl;
    outFile << "Bob " << 87 << endl;

    outFile.close();
    return 0;
}

By default, ofstream creates the file if it does not exist and overwrites it if it already exists.

Checking That A File Opened

ifstream inFile("scores.txt");

if (!inFile.is_open()) {
    cerr << "Error: could not open scores.txt" << endl;
    return 1;
}

File code should check the stream before trusting it.

Where Is The File?

  • "scores.txt" is a relative path.
  • It is resolved from the program’s working directory.
  • That may not be the same folder as the .cpp file.

When a file does not open, check both the file name and where the program is running.

Overwrite And Append

ofstream fresh("log.txt");
fresh << "Program started" << endl;
fresh.close();

ofstream later("log.txt", ios::app);
later << "Program ended" << endl;
later.close();
Mode Behavior
default output overwrite the file
ios::app add new data at the end

Reading From A File

#include <fstream>
#include <iostream>
using namespace std;

int main() {
    ifstream inFile("scores.txt");
    string name;
    int score;

    while (inFile >> name >> score) {
        cout << name << ": " << score << endl;
    }

    inFile.close();
    return 0;
}

The stream advances automatically as each value is read.

Design The File Before The Loop

Before writing the read loop, write down a small example file.

Alice 95
Bob 87
Carol 91

The example answers two questions: what does one record look like, and how does the reader know when to stop?

The File Format Is A Contract

  • The writer and reader must agree on the record shape.
  • They must also agree on the stopping rule.
  • A correct read loop matches the file’s contract.

Changing the file format means changing the code that reads it.

Pattern 1: Read Until Extraction Fails

int score;
vector<int> scores;

while (inFile >> score) {
    scores.push_back(score);
}

Use this when every remaining token in the file is data.

This is usually better than while (!inFile.eof()) because it tests the read itself.

Why Not eof() First?

while (!inFile.eof()) {
    inFile >> score;
    scores.push_back(score);
}

eof() becomes true only after a read attempts to pass the end. Test the extraction instead of predicting it.

Pattern 2: First Value Is A Count

3
Alice 95
Bob 87
Carol 91
int count;
inFile >> count;

for (int i = 0; i < count; i++) {
    inFile >> name >> score;
    cout << name << ": " << score << endl;
}

Use this when the file declares how many records follow.

Pattern 3: Sentinel Value Ends The Data

91
78
84
-1
int score;
inFile >> score;

while (score != -1) {
    scores.push_back(score);
    inFile >> score;
}

The sentinel must be a value that cannot be valid data.

Pattern 4: One Record Per Line

string line;

while (getline(inFile, line)) {
    if (line != "") {
        cout << line << endl;
    }
}

Line-controlled files work well when each record may contain spaces or optional fields.

Choosing A Reading Pattern

File Design Loop Shape
all tokens are data read until extraction fails
first value is record count counted for loop
special value marks the end sentinel loop
one logical record per line getline loop

Choose the loop after you know how the file is constructed.

When Reading Stops Early

  • End of file is normal.
  • A wrong data type is a format error.
  • Both can make extraction fail.

If the loop stops before all expected records are read, inspect the file format before changing the loop.

Writing A Vector

vector<int> scores = {91, 78, 84, 95};
ofstream outFile("scores.txt");

for (int score : scores) {
    outFile << score << endl;
}

outFile.close();

One value per line is easy to write, easy to read, and easy to inspect.

Reading A Vector

vector<int> scores;
ifstream inFile("scores.txt");
int score;

while (inFile >> score) {
    scores.push_back(score);
}

inFile.close();

The vector grows as the file provides values.

Round Trip Pattern

  1. Build data in memory.
  2. Write the data to a file.
  3. Close the output stream.
  4. Open an input stream.
  5. Read the data back into memory.

Round trips are a useful way to test whether the write format and read loop agree.

Writing Records As Text

ofstream outFile("students.txt");

outFile << "Alice " << 95 << endl;
outFile << "Bob " << 87 << endl;
outFile << "Carol " << 91 << endl;

outFile.close();

A record is one grouped unit of data. Here, each record has a name and a score.

Reading Records As Text

ifstream inFile("students.txt");
string name;
int score;

while (inFile >> name >> score) {
    cout << name << " earned " << score << endl;
}

inFile.close();

The extraction order must match the file’s field order.

CSV Files

ofstream outFile("students.csv");

outFile << "name,score" << endl;
outFile << "Alice,95" << endl;
outFile << "Bob,87" << endl;

outFile.close();

CSV files are plain text files with comma-separated fields. Spreadsheet programs can read them.

Common Practices

  • Check that every file opened before reading or writing.
  • Design the file format before designing the loop.
  • Match the loop pattern to the file’s stopping rule.
  • Keep text formats simple enough to inspect by eye.

Summary

  • File I/O is stream I/O connected to persistent storage.
  • The read loop must match the file’s format contract.
  • Clear file formats make programs easier to test and debug.

Assertions

What Should Never Be False?

  • Some checks are not about normal client mistakes.
  • They mean the program reached a state the programmer did not intend.

Slides 30-45

Validation vs Assertion

Validation Assertion
Handles client mistakes Catches programmer mistakes
Gives a message and reprompts Stops the program during development
Part of normal behavior Documents an assumption

Use validation for data coming from outside the program.

What assert Does

  • assert checks a condition the programmer believes must be true.
  • If the condition is false, the program stops.
  • The failed condition points to a broken assumption.

Assertions turn assumptions into executable checks.

Basic assert Syntax

#include <cassert>

int index = 3;
int size = 5;

assert(index >= 0 && index < size);

The condition should describe a state that must already be true.

Assertions Express Preconditions

  • A precondition must be true before code runs safely.
  • A divisor should not be zero.
  • An index should be inside the valid range.

The assertion documents what the caller or earlier code must guarantee.

Asserting Inside A Function

#include <cassert>
#include <iostream>
using namespace std;

double divide(int numerator, int denominator);

int main() {
    cout << divide(10, 2) << endl;
    return 0;
}

double divide(int numerator, int denominator) {
    assert(denominator != 0);
    return static_cast<double>(numerator) / denominator;
}

This example intentionally omits input. User-entered denominators should be validated before the function call.

Assertions Express Invariants

An invariant is a condition that should stay true while the program runs.

vector<string> names;
vector<int> scores;

assert(names.size() == scores.size());

If parallel vectors represent the same records, their sizes should match.

Reading An Assertion Failure

  • Do not patch around the failure immediately.
  • Ask which assumption failed.
  • Trace where the bad state entered the program.

An assertion failure is evidence. Use it to find the earlier cause.

Do Not Use assert For Client Input

cin >> score;
assert(score >= 0 && score <= 100);

Bad client input is expected. Expected mistakes need validation and recovery.

Validation Handles Client Input

cout << "Enter a score from 0 to 100: ";
cin >> score;

while (score < 0 || score > 100) {
    cout << "Score must be 0 to 100. Try again: ";
    cin >> score;
}

Validation keeps the program running and tells the client how to recover.

Avoid Side Effects In Assertions

assert(count++ < limit);
  • Do not put assignments inside assertion conditions.
  • Do not rely on increments inside an assertion.
  • Do not place required function calls inside an assertion.

The program should behave correctly even if assertions are disabled.

Assertions Can Be Disabled

  • Some builds compile assertions out.
  • Required behavior cannot depend on assert.
  • Use assert to catch problems, not to implement normal logic.

Assertions are development checks, not a substitute for program behavior.

Assertions And File I/O

inFile >> expectedCount;

for (int i = 0; i < expectedCount; i++) {
    inFile >> score;
    scores.push_back(score);
}

assert(scores.size() == expectedCount);

Validate the file operation first. Assert internal assumptions after the program has built its state.

Common Practices

  • Assert programmer assumptions.
  • Assert preconditions before unsafe operations.
  • Assert invariants after state changes.
  • Validate client input and file data before trusting it.

Summary

  • Validate what can reasonably happen.
  • Assert what should never happen if the code is correct.

Key Takeaways

  • File I/O uses familiar stream operations, but the data persists beyond the program run.
  • A file’s format is a contract: record shape and stopping rule determine the read loop.
  • Use validation for expected bad input and assertions for programmer assumptions that should never be false.

Appendix

File I/O: Good Practices

  • Check that a file opened before reading from it or writing to it.
  • Write down a sample file before writing the read loop.
  • Keep text file formats simple enough to inspect by eye.

File I/O: Common Mistakes

  • Assuming the file is in the same folder as the source code.
  • Using while (!inFile.eof()) instead of testing extraction.
  • Changing the write format without updating the read loop.

File I/O: More Examples

  • A score file with one integer per line.

  • Assert that two parallel vectors have the same size after loading data.
