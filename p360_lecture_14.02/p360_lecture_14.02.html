Review: new and delete

Allocating a single variable on the heap

  • new requests memory from the heap at runtime and returns its address
  • Assign that address to a pointer — that pointer is your only way to reach the memory
  • delete releases the heap block and must be called exactly once per new
  • Always set the pointer to nullptr after delete to avoid a dangling pointer
int* pNum = nullptr;     // declare a pointer — no allocation yet

pNum = new int;          // request 1 int from the heap
*pNum = 42;              // dereference to store a value

cout << *pNum << "\n";   // output: 42

delete pNum;             // release the heap memory
pNum = nullptr;          // pointer is now safe — not dangling

Emphasize: new returns an address, not a value. The pointer holds that address. delete frees the block but does NOT zero the pointer variable — only the assignment to nullptr does that.

Allocating an array on the heap

  • Use new[] to allocate an array on the heap — the size can be a variable
  • Use delete[] (with square brackets) to release it — mismatching delete and delete[] is undefined behavior
  • The pointer can be used exactly like a regular array name with [ ] subscript notation
int* pArr = nullptr;
int  size  = 5;

pArr = new int[size];          // allocate — size known at runtime, not compile time

for (int i = 0; i < size; i++)
    pArr[i] = (i + 1) * 10;   // use just like a regular array: 10 20 30 40 50

for (int i = 0; i < size; i++)
    cout << pArr[i] << " ";
cout << "\n";

delete[] pArr;                 // brackets required for arrays!
pArr = nullptr;

Drive home the delete[] rule — this is one of the most common mistakes. Also note: pArr lives on the stack; only the array block it points to is on the heap.

Key rules — don’t skip these

Situation Allocate Deallocate
Single variable new int delete p
Array new int[n] delete[] p
  • Never mix delete and delete[] — mismatching is undefined behavior
  • Every new needs exactly one delete — more is a double-free, fewer is a leak
  • A pointer does not become nullptr after delete — you must set it yourself
  • Always initialize pointers to nullptr on declaration

The table is the key reference. Quiz students on which delete form to use.

Algorithm: using new and delete correctly

  1. Declare a pointer variable and initialize to nullptr.
  2. Use new (or new[]) to allocate memory — store the returned address in the pointer.
  3. Use the pointer to read/write the heap memory as needed.
  4. When finished, call delete (or delete[]) to free the memory.
  5. Set the pointer to nullptr immediately after delete.

The five steps are identical whether it’s a single variable or an array — only the keyword changes (new vs new[], delete vs delete[]).

14-02a.cpp — putting it together

// 14-02a.cpp  --  new and delete review
#include <iostream>
using namespace std;

int main(){
    // --- Single variable on the heap ---
    int* pNum = nullptr;           // pointer to int     (#6)
    pNum = new int;                // allocate on heap   (#7)
    *pNum = 42;

    cout << "Value:   " << *pNum << "\n";
    cout << "Address: " << pNum   << "\n";

    delete pNum;                   // free heap memory   (#13)
    pNum = nullptr;                // remove dangling pointer

    // --- Array on the heap ---
    int* pArr = nullptr;
    int  size  = 5;
    pArr = new int[size];          // allocate array     (#19)

    for (int i = 0; i < size; i++)
        pArr[i] = (i + 1) * 10;

    for (int i = 0; i < size; i++)
        cout << pArr[i] << " ";
    cout << "\n";

    delete[] pArr;                 // free heap array    (#27)
    pArr = nullptr;
    return 0;
}

Walk through both sections: single int and then the array. Note the difference in delete vs delete[]. Ask students: what would happen if we wrote delete pArr instead of delete[] pArr?

Quick check

What is wrong with this code?

int* p = new int[10];
p[0] = 1;

delete p; 
p = nullptr;

Answer: Should be delete[] p — using delete instead of delete[] on a heap array is undefined behavior. The compiler will not warn you. Always match new[] with delete[].

Pseudo Dynamic Arrays

Arrays on the heap — the key insight

  • Arrays are created on the stack by default — size must be a compile-time constant
  • We can also create them on the heap using new
  • This allows the size to be determined at runtime — e.g., by asking the user
// Stack array — size must be a literal or compile-time constant
int stackArr[10];          // fine
int n = 10;
int stackArr2[n];          // ❌ not legal in standard C++

// Heap array — size can be any integer expression at runtime
int* heapArr = new int[n]; // ✅ perfectly fine

Total slides: 6 The word “pseudo” is intentional — the size is chosen at runtime, which is an improvement, but once allocated it is still fixed. If we need to grow or shrink, we need to do real dynamic resizing (next topic).

14-02b.cpp — pseudo-dynamic sales array

// 14-02b.cpp  Pseudo Dynamic Memory
#include <iostream>
#include <iomanip>
using namespace std;

int main(){
    double* sales = nullptr,     // set to no address     (#7)
            total = 0.0,
            average;
    int numDays;                 // number of sales days  (#10)

    cout << "How many days of sales figures do you wish ";
    cout << "to process? ";
    cin >> numDays;                                     // (#15)

    sales = new double[numDays]; // allocate memory       (#18)

    cout << "Enter the sales figures below.\n";
    for (int count = 0; count < numDays; count++){
        cout << "Day " << (count + 1) << ": ";
        cin >> sales[count];
    }

    for (int count = 0; count < numDays; count++)
        total += sales[count];
    average = total / numDays;

    cout << setprecision(2) << fixed << showpoint;
    cout << "\n\nTotal Sales: $"  << total   << "\n";
    cout <<   "Average Sales: $" << average << "\n";

    delete[] sales;              // Free Memory!           (#38)
    sales = nullptr;
    return 0;
}

Walk through key line numbers. Pointer declared at #7. User provides size at #15. Memory allocated at #18 using the user-provided size — this is the key line. Freed at #38.

Algorithm: creating a pseudo-dynamic array

  1. Declare a pointer variable and initialize to nullptr (#7).
  2. Prompt the user for the array size.
  3. Allocate memory using new[] with the user-provided size (#18).
    • The pointer can now be used just like a regular array variable.
  4. Use the array as needed — read, write, loop over it.
  5. When done, call delete[] and set the pointer to nullptr (#38).

14-02b.cpp

This is essentially the same as the new/delete algorithm from Topic 1 — the only new idea is that the size comes from the user at runtime.

This is a step closer, but not there yet.

Transition: we can set the size at runtime — good. But once new double[numDays] runs, the size is locked in. What if we need to add another day’s sales? We’d have to allocate a bigger array from scratch. That’s what a real dynamic array does — and it’s the next topic.

Why is this only “pseudo” dynamic?

  • The size is chosen at runtime — that’s a real improvement over a stack array
  • But once new double[numDays] executes, the size is fixed
  • If we need to add a new element, we must:
    1. Allocate a whole new, larger array
    2. Copy all existing data into it
    3. Free the original array
    4. Point our pointer at the new array
  • That process — done correctly — is what a real dynamic array does

This is a good place to mention that std::vector does exactly this internally, automatically, every time it needs to grow.

How do we create a real dynamic array?

The trick

  • Create new memory space, either larger or smaller, than what we are already using
  • Copy our existing data into that new space
  • Replace the address of our existing array with the address of the new array

Total slides: 12 The three-step trick is the heart of all dynamic resizing — including what std::vector does internally every time it grows. Once students grasp this, they understand how every resizable container works under the hood.

14-02c.cpp — initial array and increase

// 14-02c.cpp  --  real dynamically allocated array
#include <iostream>
using namespace std;

int main(){                    // build the initial array
    int* pInt = nullptr;                            // (#6)
    int  size  = 1;                                 // (#7)

    pInt = new int[size];                           // (#9)
    pInt[0] = 10;                                   // (#10)

    for (int i = 0; i < size; i++)
        cout << pInt[i] << "  ";
    cout << "\n";

    // increase by 1
    int* pTemp = nullptr;                           // (#17)
    size++;                                         // (#18)
    pTemp = new int[size];                          // (#19)

    pTemp[0] = pInt[0];   // copy old data          (#21)
    pTemp[1] = 20;        // add new element        (#22)

    delete[] pInt;        // free original          (#24)
    pInt  = pTemp;        // swing pointer          (#25)
    pTemp = nullptr;                                // (#26)

Walk through the initial array creation: pInt points to a 1-element heap array holding 10. Then walk the increase: size goes from 1→2, temp array created, data copied, old array freed, pointer swung.

    for (int i = 0; i < size; i++)

    // decrease by 1 from the right

    pInt  = pTemp;        // swing pointer          // (#39)
    pTemp = nullptr;                                // (#40)

        cout << pInt[i] << "  ";

The decrease mirrors the increase exactly — same 6-step pattern, just size– instead of size++, and we copy only the elements we want to keep.

Algorithm: building the initial array

  1. Create the main array pointer variable (#6).
  2. Set the array size variable (#7).
  3. Allocate new memory on the heap (#9).
  4. Put initial values in it.

Just like a pseudo-dynamic array so far …

14-02c.cpp

The first four steps are identical to pseudo-dynamic. The difference comes when we need to grow or shrink — that’s the new material.

Algorithm: increasing a dynamic array

  1. Increment the size variable (#18).
  2. Declare a temp pointer and allocate a new array with the updated size (#17, #19).
  3. Copy the old array values into the temp array (#21).
  4. Add the new value(s) to the temp array (#22).
  5. Free the memory for the original array (#24).
  6. Swing the original pointer to point at the temp array (#25).
  7. Set pTemp to nullptr (#26).

14-02c.cpp

Algorithm: decreasing a dynamic array

  1. Decrement the size variable (#34).
  2. Create a temp array with the new (smaller) size (#35).
  3. Copy the values you want to keep — skip the ones you are removing (#36).
  4. Free the memory for the original array (#38).
  5. Swing the original pointer to point at the temp array (#39).
  6. Set pTemp to nullptr (#40).

14-02c.cpp

The pattern — always the same

Whether increasing or decreasing, the resize steps are always:

  1. Adjust size
  2. Create pTemp with the new size
  3. Copy what you want to keep into pTemp
  4. delete[] the old array
  5. Swing: pInt = pTemp
  6. pTemp = nullptr

Once students internalize this 6-step pattern, resizing is mechanical. Stress that the ORDER matters — you must copy before you delete.

Why not just reallocate the original pointer?

// ❌ This destroys your data before you can save it:
delete[] pInt;
pInt = new int[size + 1];   // old data is gone — you can't copy it anymore
// ✅ Correct: the temp array preserves data during the transition:
pTemp = new int[size + 1];
for (int i = 0; i < size; i++)
    pTemp[i] = pInt[i];     // copy first ...
delete[] pInt;              // ... then free
pInt  = pTemp;
pTemp = nullptr;

This is the most common mistake students make when first attempting dynamic resizing. The temp array is not optional — it is the mechanism that makes the copy possible.

Programming Exercise

Create a dynamic array for pet ages using the short type. Start with 1 element and populate it.

  • Ask the user how many new ages to add. Grow the dynamic array and populate the new elements.
  • Ask the user how many ages to delete. Shrink the dynamic array by that count.
  • Adding appends to the right end of the array; deleting removes from the right end — for this exercise only. In reality you can add or delete anywhere.
  • Print the full array after each activity: after initial population, after adding, after deleting.

Total slides: 2 Give students ~12–15 minutes. They have all the pieces: new[], delete[], the 6-step resize algorithm. The challenge is applying it cleanly twice in a row (grow, then shrink) while managing the size variable consistently throughout.

Remind: use short, not int. Remind: the resize algorithm is exactly the same each time — only the direction (++ vs –) and copy logic changes.

Requirements checklist

Key Takeaways

  • Stack arrays have a size fixed at compile time — heap arrays can have a size set at runtime
  • A pseudo-dynamic array is allocated with new[] once — its size is then fixed until you explicitly resize it
  • To truly resize, use the 6-step pattern: create temp → copy → delete[] old → swing pointer → null temp
  • Always match new with delete and new[] with delete[] — mismatching is undefined behavior
  • Always set a pointer to nullptr immediately after delete or delete[]
