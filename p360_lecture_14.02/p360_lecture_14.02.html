new and delete

How Can a Program Request Memory While It Is Running?

  • What if the amount of data is known only after the program starts?
  • What if the data must remain alive beyond the block that creates it?

Slides 3–18 [Graphic suggestion: a program requesting a new labeled block from a pool of available memory.]

Why This Matters

  • Runtime decisions can determine whether and when an object is needed.
  • Some data must remain alive after the creating function returns.
  • Automatic storage alone cannot express every required lifetime.

Automatic Storage Has a Boundary

#include <iostream>
using namespace std;

void showScore();

int main() {
    showScore();
    return 0;
}

void showScore() {
    int score = 92;
    cout << score << "\n";
} // score's lifetime ends here

[Graphic suggestion: a stack frame appearing for showScore and disappearing at the closing brace.]

Dynamic Storage Has a Different Lifetime

int* score = new int{92};

// The allocation remains alive until delete.
cout << *score << "\n";

delete score;
score = nullptr;
  • The pointer has scope; the allocation has a separately controlled lifetime.

Allocating One Value

int* score = new int;
  • new int reserves enough dynamic storage for one int.
  • new returns the address of that storage.
  • score stores the returned address.

[Graphic suggestion: stack variable score containing an arrow to one int-sized heap block.]

Initializing During Allocation

int* a = new int;      // indeterminate value
int* b = new int();    // value-initialized to 0
int* c = new int{42};  // initialized to 42
  • Braced initialization makes the intended starting value explicit.

Using the Allocation

int* score = new int{92};

cout << *score << "\n";
*score = 97;
cout << *score << "\n";
  • Dereferencing accesses the dynamically stored object.

A Complete Scalar Example

#include <iostream>
using namespace std;

int main() {
    double* temperature = new double{98.6};

    cout << "Temperature: " << *temperature << "\n";
    *temperature = 101.3;
    cout << "Updated: " << *temperature << "\n";

    delete temperature;
    temperature = nullptr;
    return 0;
}

What Does delete Do?

  • delete ends the dynamically allocated object’s lifetime.
  • The storage becomes available for reuse.
  • The pointer variable still exists and still contains the old address.
delete temperature;
temperature = nullptr;

Following the Addresses

Moment Pointer Dynamic object
After new Holds allocated address Alive
After delete Still holds old address Dead
After nullptr assignment Holds no address Dead

[Graphic suggestion: three memory-state panels corresponding to the table rows.]

Allocation Inside a Function

#include <iostream>
using namespace std;

double* readTemperature();

int main() {
    double* value = readTemperature();
    cout << *value << "\n";
    delete value;
    value = nullptr;
    return 0;
}

double* readTemperature() {
    double* result = new double;
    cin >> *result;
    return result;
}

Who Owns the Allocation?

  • readTemperature creates the allocation.
  • Returning the pointer transfers cleanup responsibility to main.
  • The owner must eventually call the matching deallocation operation.

[Graphic suggestion: ownership label moving from readTemperature’s result pointer to main’s value pointer.]

The Scalar Lifetime Pattern

Step C++ operation
Allocate and initialize int* value = new int{5};
Access the object cout << *value;
Release the object delete value;
Mark the pointer empty value = nullptr;

The Ownership Contract

For every dynamic allocation, identify:

  • The pointer that owns the allocation
  • Any pointers that merely borrow access
  • The exact point where the owner releases it

Summary

  • new creates dynamic storage whose lifetime is controlled explicitly.
  • delete releases one scalar allocation.
  • Runtime flexibility comes with an ownership and cleanup obligation.

Pointer Errors, Memory Leaks, and Dangling Pointers

What Happens When the Pointer Outlives Its Target?

  • A pointer can retain an address after the object at that address is gone.
  • The address looks ordinary, but dereferencing it is no longer valid.

Slides 19–42 [Graphic suggestion: a pointer arrow ending at a faded, crossed-out object.]

A Map of Pointer Failures

Failure area Question
Initialization Does the pointer hold a known address?
Lifetime Is the target still alive?
Ownership Who must release the allocation?
Deallocation Is the release operation valid and correctly matched?

Leaks and Dangling Pointers Are Opposites

  • Memory leak: live allocation, but no usable pointer can reach it.
  • Dangling pointer: pointer remains, but its target is no longer alive.

[Graphic suggestion: side-by-side panels—an unreachable live block and an arrow aimed at a dead block.]

Why Results Seem Unpredictable

  • Undefined behavior removes the language’s guarantees.
  • A broken program may crash, print plausible output, or appear to work.
  • Successful output does not prove that the access was valid.

Three Questions Before Dereferencing

  1. Was the pointer initialized to a known address?
  2. Is the pointed-to object still alive?
  3. Does the pointer refer to the object the program intends to access?

Wild Pointers

#include <iostream>
using namespace std;

int main() {
    int* score;
    cout << *score << "\n"; // undefined behavior
    return 0;
}

Pause before revealing the final comment. Ask which of the three dereference checks fails. [Graphic suggestion: score containing question marks instead of a known address.]

Null Pointer Dereference

int* score = nullptr;

if (score != nullptr) {
    cout << *score << "\n";
}
  • nullptr represents a deliberate no-object state.
  • It must be checked before dereferencing.

Returning a Local Address

#include <iostream>
using namespace std;

int* makeScore();

int main() {
    int* score = makeScore();
    cout << *score << "\n"; // dangling pointer
    return 0;
}

int* makeScore() {
    int localScore = 95;
    return &localScore;
} // localScore dies here

Use After Scope

int* selected = nullptr;

{
    int score = 88;
    selected = &score;
} // score's lifetime ends

cout << *selected << "\n"; // undefined behavior

[Graphic suggestion: nested scope box disappearing while selected retains its address.]

Use After delete

int* score = new int{95};
delete score;

cout << *score << "\n"; // target is dead

score = nullptr;
  • delete invalidates access through every pointer to that allocation.

Aliasing Spreads Dangling Pointers

int* owner = new int{95};
int* alias = owner;

delete owner;
owner = nullptr;

cout << *alias << "\n"; // alias still dangles

[Graphic suggestion: owner and alias both point to one allocation; delete crosses out the allocation, then owner becomes null while alias remains.]

Why nullptr Does Not Solve Everything

  • Setting the deleting pointer to nullptr prevents reuse through that variable.
  • Other aliases are not changed automatically.
  • The real requirement is that no pointer access an object after its lifetime ends.

Double Deletion

int* score = new int{95};

delete score;
delete score; // undefined behavior
  • One allocation must be released exactly once.

Deleting Non-Dynamic Memory

int score = 95;
int* pointer = &score;

delete pointer; // score was not created by new
  • delete requires an address produced by a matching new expression.

[Graphic suggestion: stack object labeled automatic contrasted with heap object labeled dynamic.]

What Is a Memory Leak?

  • A dynamic allocation remains alive.
  • The program has lost every pointer that could release it.
  • Repeating the loss consumes more memory over time.

[Graphic suggestion: a live heap block with no incoming arrows.]

Overwriting the Only Pointer

int* score = new int{80};

score = new int{95}; // first allocation is leaked

delete score;
score = nullptr;

Pause before revealing the comment. Ask which allocation remains reachable.

Leaks Across Repeated Calls

for (int i = 0; i < 1000; i++) {
    int* sample = new int{i};
    cout << *sample << "\n";
    // No delete: one leaked int per iteration.
}
  • Small leaks become large when placed on repeated paths.

Leaks Through Early Returns

int* score = new int{95};

if (!(cin >> *score)) {
    return 1; // cleanup is skipped
}

delete score;
score = nullptr;
  • Every exit path must preserve the cleanup contract.

Conditional Cleanup Is Fragile

int* score = new int{95};

if (!(cin >> *score)) {
    delete score;
    return 1;
}

cout << *score << "\n";
delete score;
score = nullptr;
  • Manual ownership duplicates cleanup across control paths.

Ownership Must Be Singular and Visible

Role Responsibility
Owner Releases the allocation exactly once
Borrower Uses the object only while it remains alive
Alias Never assumes ownership merely because it has the address

Repairing a Broken Program

#include <iostream>
using namespace std;

int main() {
    int* score = new int{95};
    int* observer = score;

    cout << *observer << "\n";

    delete score;
    score = nullptr;
    observer = nullptr;
    return 0;
}

Safety Checklist

  • Initialize every pointer to a known address or nullptr.
  • Dereference only while the intended target is alive.
  • Keep ownership unambiguous and distinguish owners from observers.
  • Match each allocation with exactly one correct release.

Summary

  • Pointer validity depends on initialization, lifetime, and bounds.
  • Leaks lose access to live storage; dangling pointers retain access to dead storage.
  • Manual ownership makes every control path responsible for correct cleanup.

Smart Pointers

Can Ownership Clean Up After Itself?

  • Manual cleanup fails when an exit path forgets delete.
  • Smart pointers make ownership part of an object’s automatic lifetime.

Slides 43–75 [Graphic suggestion: a scope closing and a managed allocation being released automatically.]

Smart Pointers Are Objects

  • A smart pointer stores an address and manages an ownership policy.
  • Dereference syntax still provides access to the pointed-to object.
  • The smart-pointer object’s lifetime controls its cleanup behavior.

The Smart-Pointer Map

Type Ownership model Default use
unique_ptr One owner Preferred dynamic ownership
shared_ptr Multiple owners Genuine shared lifetime
weak_ptr Non-owning observer Observe shared storage safely

Required Header

#include <iostream>
#include <memory>
#include <utility>
using namespace std;
  • Smart pointers and their creation helpers are declared in <memory>; move is declared in <utility>.

What Is RAII?

  • Resource Acquisition Is Initialization ties a resource to an object’s lifetime.
  • Construction acquires or receives the resource.
  • Destruction releases it automatically when the object leaves scope.

[Graphic suggestion: timeline labeled construction, use, scope exit, automatic release.]

Raw Ownership vs. Managed Ownership

int* raw = new int{42};
cout << *raw << "\n";
delete raw;
raw = nullptr;
unique_ptr<int> managed =
    make_unique<int>(42);
cout << *managed << "\n";

Empty Smart Pointers

unique_ptr<int> score;

if (score) {
    cout << *score << "\n";
}

if (score == nullptr) {
    cout << "No score\n";
}

Introducing unique_ptr

unique_ptr<int> score(new int{95});
  • score is the sole owner of the dynamic int.
  • Destruction of score releases the allocation.
  • Direct use of new is valid here, but a helper is preferred.

Creating with make_unique

unique_ptr<int> score = make_unique<int>(95);
  • make_unique creates the object and returns its owner.
  • The allocation is never exposed as an unowned raw pointer.

Using a unique_ptr

unique_ptr<int> score = make_unique<int>(95);

cout << *score << "\n";
*score += 5;
cout << *score << "\n";
  • Dereferencing accesses the managed object, not the smart pointer itself.

Automatic Cleanup at Scope Exit

void showScore() {
    unique_ptr<int> score = make_unique<int>(95);
    cout << *score << "\n";
} // score is destroyed; the int is released

[Graphic suggestion: owner and allocation both disappear at the closing brace.]

A Complete unique_ptr Example

#include <iostream>
#include <memory>
using namespace std;

int main() {
    unique_ptr<double> temperature =
        make_unique<double>(98.6);

    cout << *temperature << "\n";
    *temperature = 101.3;
    cout << *temperature << "\n";

    return 0;
} // automatic cleanup

move Means Ownership Transfer

  • A move transfers a resource instead of duplicating it.
  • The destination becomes the owner.
  • The source becomes empty but remains a valid smart-pointer object.

[Graphic suggestion: ownership badge moving from source to destination while one allocation remains.]

Why Copying Is Disabled

unique_ptr<int> first = make_unique<int>(42);

// unique_ptr<int> second = first;
// Error: copying would create two unique owners.
  • The compiler prevents a double-ownership state.

Transferring Ownership with move

unique_ptr<int> first = make_unique<int>(42);
unique_ptr<int> second = move(first);

cout << *second << "\n";

Pause before the move. Ask which variable should own the allocation afterward.

State After a Move

unique_ptr<int> first = make_unique<int>(42);
unique_ptr<int> second = move(first);

cout << boolalpha;
cout << "first owns: " << static_cast<bool>(first) << "\n";
cout << "second owns: " << static_cast<bool>(second) << "\n";

Passing unique_ptr to Functions

void printValue(const unique_ptr<int>& value);
void consume(unique_ptr<int> value);

int main() {
    unique_ptr<int> score = make_unique<int>(95);
    printValue(score);       // borrow the owner
    consume(move(score));   // transfer ownership
}
  • The parameter style reveals whether ownership stays or moves.

Returning a unique_ptr

#include <iostream>
#include <memory>
using namespace std;

unique_ptr<int> makeScore(int value);

int main() {
    unique_ptr<int> score = makeScore(95);
    cout << *score << "\n";
    return 0;
}

unique_ptr<int> makeScore(int value) {
    return make_unique<int>(value);
}

Replacing a Managed Object

unique_ptr<int> score = make_unique<int>(80);

score.reset(new int{95});
cout << *score << "\n";

score.reset(); // release and become empty
  • reset releases the previous managed object before taking a new one.

Borrowing the Raw Address

unique_ptr<int> owner = make_unique<int>(95);
int* observer = owner.get();

cout << *observer << "\n";
  • get() does not transfer ownership.
  • The observer is valid only while the owner keeps the object alive.

Releasing Ownership Manually

unique_ptr<int> owner = make_unique<int>(95);
int* raw = owner.release();

delete raw;
raw = nullptr;
  • release() returns the address without deleting the object.
  • Manual cleanup responsibility immediately returns.

Introducing shared_ptr

shared_ptr<int> first = make_shared<int>(95);
shared_ptr<int> second = first;
  • Both smart pointers own the same object.
  • A reference count records how many shared owners remain.

Creating with make_shared

shared_ptr<double> temperature =
    make_shared<double>(98.6);

cout << *temperature << "\n";
cout << temperature.use_count() << "\n";
  • Prefer make_shared when shared ownership is required.

Copying Shared Ownership

shared_ptr<int> first = make_shared<int>(95);
cout << first.use_count() << "\n"; // 1

shared_ptr<int> second = first;
cout << first.use_count() << "\n"; // 2
cout << second.use_count() << "\n"; // 2

[Graphic suggestion: two owners pointing to one allocation and one control block labeled count 2.]

When Shared Storage Is Released

shared_ptr<int> first = make_shared<int>(95);

{
    shared_ptr<int> second = first;
    cout << first.use_count() << "\n"; // 2
}

cout << first.use_count() << "\n"; // 1
  • The allocation is released when the final shared owner is destroyed.

The Cost of Shared Ownership

  • A control block stores ownership information and the reference count.
  • Copies must update that count.
  • Use shared_ptr only when the design truly requires multiple owners.

Shared-Ownership Cycles

  • Owner A can keep Owner B alive while Owner B keeps Owner A alive.
  • Neither reference count reaches zero.
  • The cycle leaks even though every link uses shared_ptr.

Keep this conceptual because classes have not yet been developed in depth. [Graphic suggestion: two objects joined by circular owning arrows, each count held above zero.]

Why weak_ptr Exists

  • weak_ptr observes an object managed by shared_ptr.
  • Observation does not increase the shared reference count.
  • A weak link can break an ownership cycle.

[Graphic suggestion: replace one owning arrow in the cycle with a dashed observer arrow.]

Locking a weak_ptr

shared_ptr<int> owner = make_shared<int>(95);
weak_ptr<int> observer = owner;

if (shared_ptr<int> access = observer.lock()) {
    cout << *access << "\n";
}
  • lock() returns temporary shared ownership if the object still exists.

Choosing a Pointer Type

Need Prefer
No dynamic lifetime Direct object
Temporary guaranteed access Reference
Nullable non-owning access Raw pointer
One dynamic owner unique_ptr
Genuine shared ownership shared_ptr
Observe shared storage weak_ptr

Modern Ownership Rewrite

int* score = new int{95};
if (!(cin >> *score)) {
    delete score;
    return 1;
}
cout << *score << "\n";
delete score;
unique_ptr<int> score =
    make_unique<int>(95);
if (!(cin >> *score)) {
    return 1;
}
cout << *score << "\n";

Summary

  • Prefer unique_ptr for dynamic ownership and explicit ownership transfer.
  • Use shared_ptr only for genuine shared lifetime.
  • Use weak_ptr to observe shared storage without extending its lifetime.

Key Takeaways

  • Dynamic allocation separates an object’s lifetime from the scope that creates it.
  • Raw ownership requires valid access, correct bounds, and exactly one matching release.
  • unique_ptr makes single ownership automatic; shared ownership requires deliberate use of shared_ptr and weak_ptr.

Appendix

new and delete: Good Practices

  • Initialize allocations immediately when possible.
  • Preserve one clear owning pointer for each scalar allocation.
  • Place the matching release near the end of the owner’s responsibility.

new and delete: Common Mistakes

  • Calling delete more than once for the same allocation
  • Forgetting cleanup on an early return
  • Treating the pointer’s scope as the allocation’s lifetime

new and delete: More Examples

    return new char{value};
}

char* grade = makeGrade('A');
cout << *grade << "\n";
delete grade;
grade = nullptr;

Pointer Errors: Good Practices

  • Check initialization, lifetime, and bounds before dereferencing.
  • Keep owners distinct from borrowers.
  • Test failure and early-return paths, not only the successful path.

  • Assuming a non-null address must still be valid
  • Nulling one alias and assuming every alias changed
  • Confusing an observing alias with a second owner

Smart Pointers: Good Practices

  • Prefer make_unique and single ownership by default.
  • Transfer unique_ptr ownership explicitly with move.
