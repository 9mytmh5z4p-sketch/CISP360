Parts of a C++ Program

Two parts, every time

  • Every C++ program — from a five-line hello world to a million-line operating system — has the same two structural parts.
  • Before you can read or write any program, you need to know what those two parts are and what each one does.

Total slides: 6 This section establishes the mental model students will use for the rest of the semester. Do not show the code yet — name the two parts first (top matter, main function), then reveal them on the next slides. The two-part model is the frame; the syntax fills it in.

Part 1: Top Matter

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    int age = 25;
    string name = "Pat Smith";

    cout << "I'm " << name << ", age: " << age << endl;

    return 0;
}

Top matter (lines 1–6) sets up the programming environment before execution begins. It is the context that makes the rest of the program possible. Students should learn to write it first — before main — and to write it the same way every time. Cover each element briefly; the next topic (Top Matter) will go deeper.

Common Elements

The vocabulary of C++

  • Before you can read or write code, you need the vocabulary.
  • Every C++ program — from the smallest to the largest — uses the same three categories of words.

Total slides: 12 Frame this as vocabulary acquisition before grammar instruction. Students who mix up these three categories will write code that does not compile and will not understand why. Spend time here — the payoff is fewer beginner errors.

Three categories, every program

Category Who controls it Example
Identifiers You age, name, main
Reserved words The compiler int, return, if
Literals Fixed in the source 25, "Hello", 0

Show the taxonomy before explaining each category in depth. Students should be able to look at any line of code and classify every word into one of these three buckets. Comments are a fourth category worth mentioning briefly: they belong to neither the programmer’s logic nor the compiler — they are human notes that the compiler ignores entirely.

Identifiers

  • Words you get to name — C++ does not care what they are called.
  • Used for variables, functions, and other programmer-defined things.
int age = 25;           // 'age' is an identifier
string name = "Pat";    // 'name' is an identifier

Identifiers follow rules: must start with a letter or underscore, can contain letters, digits, and underscores, and are case-sensitive. The important point for day one is simpler: if you named it, it is an identifier. Examples like ‘age’, ‘myArray’, ‘GoHome’, and even ‘yo’ are all valid identifiers.

Reserved words

  • Words the compiler owns — they carry fixed meaning and cannot be repurposed.
  • Using them as identifier names will prevent your program from compiling.
int age = 25;       // 'int' is a reserved word — declares integer type
return 0;           // 'return' is a reserved word — sends a value back

Common reserved words students will encounter immediately: int, return, if, else, while, for, using, namespace. There are about 95 reserved words in C++. Students do not need to memorize them — they will learn them by using them. The important concept is that these words are off-limits as names.

Error: reserved word used as a variable name

// ERROR: 'int' is a reserved word — cannot be used as a variable name.
int int = 5;
cout << int << endl;
error: expected unqualified-id before 'int'
     int int = 5;
         ^~~

Show the actual compiler output so students recognize this error when they encounter it. The phrase “expected unqualified-id” is the compiler’s way of saying “I expected a name here, but you gave me a reserved word.” This exact error is one of the first many students produce.

Error: C++ is case-sensitive

// ERROR: 'Include' is not '#include' — directives are case-sensitive.
#Include <iostream>
using namespace std;

int main()
{
    cout << "Hello!" << endl;
    return 0;
}
error: invalid preprocessing directive #Include

Reserved words and directives are all lowercase. A capital letter anywhere in a reserved word makes it an unrecognized token. This catches beginners who autocorrect or type quickly. The same issue applies to ‘INT’, ‘Return’, ‘While’, etc.

Error: undeclared identifier

// ERROR: 'messge' is not declared — a one-character typo makes it unrecognizable.
int main()
{
    string message = "Hello!";
    cout << messge << endl;    // typo: 'messge' instead of 'message'
    return 0;
}
error: 'messge' was not declared in this scope

This is the most common identifier error students will make. A single typo — transposed letters, wrong case, extra character — creates a name the compiler has never seen. The fix is always the same: compare the name where it is declared to the name where it is used. [Note: if a screenshot of the actual compiler output in the student’s IDE would be clearer than the text block above, substitute it here.]

Literals

  • Data that is fixed in the source code — it does not change while the program runs.
  • Not under program control: once you write 25, it is always 25.
int age = 25;                    // 25 is a literal
cout << "I love to code!\n";     // "I love to code!\n" is a literal
myTax = salary * myRate + 8345.11;  // 8345.11 is a literal (magic number)

The magic number point is worth flagging: hard-coded values like 8345.11 buried in an expression are literals that make code hard to read and maintain. C++ provides constants to give literals meaningful names — that comes later in the course. For now, students should be able to identify literals when they see them.

Comments

// This is a single-line comment — everything after // is ignored.

/*
   This is a multi-line comment.
   The compiler ignores everything between the delimiters.
*/

int age = 25;   // end-of-line comment

Two forms: single-line (//) and block (/* */). The compiler ignores both completely. Uses: explain intent, document logic, temporarily disable code during debugging. Students should write comments from day one — not because they are required, but because they will thank themselves later.

Write comments for intent, not for code

// Helpful — explains why, not what
float age;   // store months after decimal; that's why not an int

// Unhelpful — restates what the code already says
float age;   // this is a float

The unhelpful comment adds no information — the code already says it is a float. The helpful comment explains a non-obvious decision. The best comments answer the question a future reader would ask: “Why was this done this way?” Encourage students to write comments as if explaining the code to a classmate who is not familiar with it.

Whole-line vs. end-of-line comments

// Whole-line comment — occupies an entire line
// Use for explaining blocks of logic

int age;    // End-of-line comment — use only with variable declarations
  • Whole-line comments are preferred because code changes frequently and they are easy to move.
  • End-of-line comments are appropriate only next to variable declarations.

The practical reason to prefer whole-line comments: if the code on a line changes or moves, an end-of-line comment may become detached from what it was explaining. A whole-line comment directly above the relevant code stays associated with it even after edits.

Summary

  • Identifiers are names you invent — C++ does not care what you call them.
  • Reserved words belong to the compiler — they have fixed meaning and cannot be used as names.
  • Literals are fixed data hardcoded into the program.
  • Comments are human notes the compiler ignores entirely — write them for intent.

Top Matter

The setup section

  • The lines before main() prepare the programming environment — without them, the rest of the program cannot compile or run.
  • You will write top matter the same way at the start of every program you produce in this class.

Total slides: 11 Top matter is not optional overhead — it is the scaffolding that makes the program possible. Frame it as: “before you write a single statement, you write top matter.” Students who skip or corrupt top matter will get confusing errors. Establishing the habit early pays dividends all semester.

Three components of top matter

  • Source File Header (SFH) — documents the file: name, author, date, description.
  • Preprocessor directives — bring in external code the program needs (e.g., #include).
  • Namespace specification — controls how you refer to standard library names.

All three components appear in every program students write this semester. They appear in this order, at the top of the file, every time. The next several slides cover each component individually.

The Source File Header

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    ...
}

Four lines, always in this order: program name, author name, date written, brief description. These are comments — the compiler ignores them — but they document who wrote the file and what it does. The author line: the author is the person who wrote the code. If a student uses AI to write code, they should understand what they are putting their name on.

The SFH template

// program name
// author name
// date written
// brief description
  • Every program submitted in this class must include these four lines.
  • Fill in the actual values — do not submit the template placeholders.

Make the requirement explicit. Students who skip the SFH lose points and also produce code that is harder for anyone (including themselves) to identify and maintain. The brief description should be one sentence — what does this program do?

The #include directive

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    ...
}
  • #include tells the preprocessor to insert the contents of a header file into your source before compilation begins.
  • Without it, the compiler does not know what cout or cin are.

The preprocessor runs before the C++ compiler. Think of #include as a copy-and-paste: the contents of are inserted at that line before the compiler sees any of the code. Students do not need to understand the preprocessor in depth today — they need to understand that #include is mandatory and that forgetting it is a common error.

#include <iostream>

#include <iostream>
  • iostream provides the tools for input and output.
  • Without it, cout, cin, and endl are not available.
Name Purpose
cout Output to the terminal
cin Input from the keyboard
endl End the current line

Students will use cout immediately. cin comes up in the next few weeks. endl is used alongside cout. The important concept: these names come from the iostream header; without the #include, the compiler does not know they exist.

Angle brackets vs. quotes

#include <iostream>    // standard library header — use angle brackets
#include "myfile.h"    // user-defined header — use double quotes
  • Angle brackets < > — search the standard library locations.
  • Double quotes " " — search the current project directory first, then standard locations.

In this class, students will only use angle brackets. The double-quote form is for headers students write themselves — that happens in later courses. The distinction is worth stating once so students know why the brackets exist and do not mix them up.

C-style headers and .h files

#include <stdio.h>     // old C header — do not use in C++
#include <cstdio>      // C++ version of the same header — acceptable
#include <iostream>    // modern C++ header — preferred in this class
  • Old C headers use the .h extension: <math.h>, <stdio.h>.
  • Modern C++ provides updated versions without .h: <cmath>, <cstdio>.
  • In this class, use only modern C++ headers — no .h files.

Students may encounter .h headers in older tutorials or examples online. It is worth flagging that those are C-style headers, not C++ headers, and should not be used in this class. The C++ standard library headers have no extension at all: , , , .

using namespace std

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    ...
}
  • Without this line, every standard library name requires the std:: prefix: std::cout, std::cin, std::endl.
  • With this line, you write cout, cin, and endl directly.

A namespace is a named container that prevents name collisions between large codebases. The standard library lives in the ‘std’ namespace. ‘using namespace std’ brings all of those names into the current scope. In this class, always include it — it keeps code concise. Students will learn about the tradeoffs of namespace pollution in later courses.

  • SFH documents the file — four lines: name, author, date, description.
  • #include <iostream> imports the input/output tools (cout, cin, endl).

The main() Function

Where execution begins

  • Every C++ program starts in exactly one place: the main() function.
  • No matter how large or complex the program, execution always begins at main() and the OS always receives a result when main() returns.

Total slides: 8 Students have now seen main() as part of the program structure. This section explains what main() actually is and why it is the way it is. The goal is not syntax memorization — it is building a mental model of program execution.

The main() declaration

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    int age = 25;
    string name = "Pat Smith";

    cout << "I'm " << name << ", age: " << age << endl;
    cout << "I love to code!\n";

    return 0;
}

Three parts on line 8: ‘int’ is the return type, ‘main’ is the function name, ‘()’ is the parameter list (empty here). Every function in C++ has these three parts. Students will write many functions later in the course — main() is the first one they see.

Why main returns int

  • The return value of main is sent to the operating system when the program finishes.
  • 0 means the program ran successfully.
  • Any non-zero value signals an error condition.
return 0;    // tell the OS: everything worked
return 1;    // tell the OS: something went wrong

Students may wonder why a function that “just runs the program” needs to return a value. The OS uses the return value for scripting and automation — if a program exits with a non-zero code, a shell script can detect the failure and respond. In this class, students always return 0. The deeper significance becomes clear when they write programs that interact with other programs.

Curly braces define a block

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    int age = 25;
    string name = "Pat Smith";

    cout << "I'm " << name << ", age: " << age << endl;
    cout << "I love to code!\n";

    return 0;
}

The opening brace on line 9 and closing brace on line 17 define the block — everything inside belongs to main(). Blocks control scope and lifetime: variables declared inside this block exist only while the block is executing. Not all languages use braces; Python uses indentation. C++ uses braces.

Block scope and variable lifetime

  • Everything between { and } belongs to the block.
  • Variables declared inside a block exist only while that block is executing.
  • Blocks can be nested — a block inside a block inside a block.

The concept of scope — where a variable is accessible — is one of the most important in C++. Blocks are the mechanism. Students will encounter scope errors within the first few weeks. The mental model: a block is a container; when the container closes, everything inside it is gone.

Statements inside main()

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.
#include <iostream>
using namespace std;

int main()
{
    int age = 25;
    string name = "Pat Smith";

    cout << "I'm " << name << ", age: " << age << endl;
    cout << "I love to code!\n";

    return 0;
}

Statements execute top to bottom, one at a time. Each statement ends with a semicolon — the semicolon is the terminator, not a separator. Forgetting a semicolon is one of the most common beginner errors, and the compiler error message it produces (‘expected’;’ before…’ ) is one of the first students will see.

return 0

// intro.cpp
// Dr. Fowler
// 1/1/26
// CISP360 introductory program.

    int age = 25;

return 0 is the last statement in main(). It closes the function and sends 0 to the OS. In this class, every main() ends with return 0. Students should write it reflexively — as part of the template, not as an afterthought.

Summary

  • main() is the entry point — execution always starts here.
  • Statements inside main() run top to bottom; each ends with a semicolon.
  • return 0 closes main() and tells the operating system the program finished successfully.

  • Every C++ program has two parts: top matter (environment setup) and main() (where execution begins)
  • Every word in a program is an identifier, a reserved word, a literal, or a comment — and the difference matters
  • Top matter is the same every time: SFH, #include <iostream>, using namespace std
  • main() is the entry point; statements run top to bottom; return 0 closes the program

Parts of a C++ Program: Good Practices

  • Always write top matter before main() — establish the habit from the first program.
  • Leave a blank line between the end of top matter and the opening of main() — it is a convention that aids readability.
  • If a program has two parts and one is missing or corrupt, the compiler will produce errors that seem unrelated to the actual problem.

Parts of a C++ Program: Common Mistakes

  • Writing main() content before top matter — the compiler processes files top to bottom; headers must come first.
  • Thinking the two-part structure changes with the problem — it does not; only the content inside main() changes.
  • Confusing the two parts: top matter does not execute; only the code inside main() executes.

Parts of a C++ Program: More Examples

// temperature.cpp
// Student Name
// Today's Date
// Converts Celsius to Fahrenheit.
#include <iostream>
using namespace std;

int main()
{
    double celsius = 100.0;
    double fahrenheit = (celsius * 9.0 / 5.0) + 32.0;

    cout << celsius << "C = " << fahrenheit << "F" << endl;

    return 0;
}

Common Elements: Good Practices

  • When you cannot figure out why code does not compile, classify each word on the problem line: identifier, reserved word, or literal. The error is often a category confusion.
  • Name identifiers after what they represent, not how they are used — age is better than x; totalPrice is better than tp.
  • Write comments that explain intent. If the code is clear, a comment may not be needed. If the code is not clear, a comment is required.

Common Elements: Common Mistakes

  • Using a reserved word as a variable name — the compiler error message is confusing until you recognize the pattern.
  • Assuming C++ is case-insensitive — Int, INT, and int are three different things; only int is a reserved word.
  • Writing comments that restate the code — “this is an int” next to int age adds no value.

Common Elements: More Examples

// Reserved words: int, string, cout, endl, return, using, namespace
// Identifiers: age, name, main
// Literals: 25, "Pat Smith", 0, "\n"

int age = 25;
string name = "Pat Smith";

cout << "I'm " << name << ", age: " << age << endl;

return 0;

Top Matter: Good Practices

  • Write top matter before you write a single statement — it is the foundation everything else compiles against.
  • Fill in the SFH completely: a description that says “does stuff” is not a description.
  • Include only the headers you actually use — adding #include <vector> to a program that uses no vectors is unnecessary.

Top Matter: Common Mistakes

  • Forgetting #include <iostream> — produces errors like 'cout' was not declared in this scope that seem unrelated to the problem.
  • Writing #Include instead of #include — preprocessor directives are case-sensitive; this produces invalid preprocessing directive.
  • Omitting using namespace std — produces errors like 'cout' is not a member of 'std' or requires std:: in front of every standard name.

Top Matter: More Examples

// A program using multiple headers
// calculator.cpp
// Student Name
// Today's Date
// Basic arithmetic calculator.
#include <iostream>
#include <cmath>     // provides sqrt(), pow(), abs()
using namespace std;

int main()
{
    double x = 16.0;
    cout << "sqrt(" << x << ") = " << sqrt(x) << endl;
    return 0;
}

The main() Function: Good Practices

  • End every main() with return 0 — always, without exception, until you have a reason not to.
  • Keep main() focused on the high-level logic of the program; detailed work belongs in separate functions.
  • Match every opening brace { with a closing brace } — mismatched braces are a common source of confusing errors.

The main() Function: Common Mistakes

  • Forgetting the semicolon at the end of a statement — produces expected ';' before errors that point to the next line, not the actual problem.
  • Misspelling main — mian(), Main(), or MAIN() are all undefined; the linker error message is undefined reference to 'main'.
  • Putting statements outside the braces of main() — executable statements must be inside a function.

The main() Function: More Examples

int main()
{
    // Sequence: three statements run top to bottom
    cout << "Step 1" << endl;
    cout << "Step 2" << endl;
    cout << "Step 3" << endl;

    return 0;
}

Coding Exercise 1-1: Good Practices

  • Type the program by hand — do not copy and paste. Typing forces you to see every character.
  • Compile frequently as you type; don’t wait until the whole program is written to check for errors.
  • When the compiler gives an error, read it carefully and try to understand it before making a change.

Coding Exercise 1-1: Common Mistakes

  • Forgetting a closing quote " on a string literal — the compiler error is confusing because it points to the wrong line.
  • Typing Cout or COUT instead of cout — C++ is case-sensitive; cout is always lowercase.
  • Forgetting a semicolon at the end of a cout statement — the error message points to the next line.
