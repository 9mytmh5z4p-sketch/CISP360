Very Brief Review of Static

What Does Static Remember?

  • If local variables disappear after a function ends, how can a function remember anything next time?
  • static changes lifetime without making the name global.

Slides 3-8

Local Scope, Longer Lifetime

  • A normal local variable is created when the function runs.
  • A static local variable is created once and keeps its value between calls.
  • The variable name is still usable only inside that function.

Why This Matters

  • Static locals can give a function memory.
  • That memory can be useful for counters, cached setup, or one-time initialization.
  • Hidden memory also makes behavior depend on previous calls.

Bridge to Patterns

  • static is a mechanism, not the main design idea.
  • Some patterns use static storage to enforce a larger design rule.
  • Singleton is one example: one shared object, reached in a controlled way.

Summary

  • Static local variables preserve state between calls.
  • The value lasts, but the name stays local.
  • That combination makes static useful and easy to misuse.

Design Patterns

Why Name Solutions?

  • Programmers run into the same design problems again and again.
  • A shared name lets a team discuss the shape of a solution without redrawing it from scratch.

Slides 9-20

Christopher Alexander

  • Architect Christopher Alexander wrote about recurring design problems in buildings and towns.
  • His patterns described problems, forces, and reusable solution shapes.
  • The idea was not “copy this building”; it was “recognize this design situation.”

Pattern Language

  • Alexander’s larger idea was a connected vocabulary of design choices.
  • Small patterns and large patterns could work together.
  • The value came from communication as much as construction.

From Buildings to Software

  • Software also has recurring design problems.
  • The exact code changes from program to program.
  • The design shape can still be recognized, discussed, and reused.

The Gang of Four Book

  • Design Patterns: Elements of Reusable Object-Oriented Software popularized software design patterns.
  • The authors are Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides.
  • They are often called the Gang of Four.

What Is a Design Pattern?

  • A design pattern is a reusable approach to a recurring design problem.
  • It names the problem, the structure, and the tradeoff.
  • The pattern is about intent, not memorized code.

What Patterns Are Not

  • Patterns are not syntax rules.
  • Patterns are not libraries.
  • Patterns are not guaranteed best practices.
  • Patterns are design vocabulary with consequences.

Pattern, Not Recipe

  • A recipe gives exact steps.
  • A pattern gives a shape that must be adapted to the program.
  • The same pattern may look different in small, medium, and large programs.

The Three Questions

  • What problem does this pattern solve?
  • What structure does it use?
  • What tradeoff does it introduce?

Common Practices

  • Start with the problem, not the pattern name.
  • Use the smallest structure that makes the design clearer.
  • Avoid adding pattern machinery when ordinary code is enough.

Summary

  • Design patterns name recurring design choices.
  • They help programmers recognize, explain, and maintain structure.
  • Every pattern also has a cost.

Singleton Pattern

What If There Should Only Be One?

  • Some program resources should not have many independent copies.
  • A settings manager, logger, or shared configuration may need one controlled access point.

Slides 21-31

Singleton Pattern

  • Singleton controls object creation so a program uses one shared instance.
  • The class decides when the instance is created.
  • Other code asks for access instead of creating its own copy.

One Instance, One Access Point

Rule Meaning
one instance the program should not create several copies
controlled access other code reaches the instance through one function
shared state callers see the same stored data

A Simple Singleton Shape

class Settings {
private:
    Settings();

public:
    static Settings& getInstance();
};
  • The constructor is private.
  • The access function is static.
  • Callers use the access function instead of constructing directly.

Why Static Shows Up Here

Settings& Settings::getInstance() {
    static Settings instance;
    return instance;
}
  • The local static object is created once.
  • Each later call returns the same object.
  • The object is stored inside the access function’s control.

Example: Program Settings

  • Several functions may need the same display mode, file path, or course setting.
  • A Singleton can keep those values in one shared place.
  • The pattern says the shared place is intentional, not accidental.

[Graphic suggestion: a central Settings object with several program areas pointing to the same access function.]

The Tradeoff

  • Singleton makes access convenient.
  • It also creates shared state.
  • Shared state can make testing and order-of-calls harder to reason about.

When Singleton Is a Bad Fit

  • The data could be passed as a parameter.
  • The program may need multiple versions later.
  • The object changes state in ways callers cannot easily see.

Common Practices

  • Use Singleton only when one instance is part of the problem.
  • Keep the object small and focused.
  • Do not use Singleton as a disguised global variable.

Summary

  • Singleton controls access to one shared instance.
  • It is useful when the one-instance rule is intentional.
  • It should be chosen for a design reason, not convenience alone.

Strategy Pattern

What If Behavior Changes?

  • A program may need to do the same job using different rules.
  • The caller should not have to be rewritten every time the rule changes.

Slides 32-44

Conditional Logic Smell

  • A long if/else if or switch chain can be a warning sign.
  • If each branch is a different behavior, the function may be doing too much.
  • Strategy moves the varying behavior into separate units.

Strategy Pattern

  • Strategy separates a behavior from the code that uses it.
  • The caller works through a common function shape.
  • Different strategies provide different versions of the behavior.

Same Job, Different Rules

Job Possible strategies
calculate a discount student, employee, seasonal
choose a score adjustment late work, extra credit, dropped quiz
sort data ascending, descending, custom priority

The Strategy Shape

  • The caller receives or stores a behavior choice.
  • Each strategy follows the same expected call shape.
  • The caller uses the behavior without knowing every rule inside it.

[Graphic suggestion: one checkout function pointing to three different discount-rule boxes.]

Example: Choosing a Discount Rule

#include <iostream>
#include <iomanip>
using namespace std;

double noDiscount(double price);
double studentDiscount(double price);
void printTotal(double price, double discountRule(double));

int main() {
    printTotal(40.00, studentDiscount);
    return 0;
}

Example: Strategy Functions

double noDiscount(double price) {
    return price;
}

double studentDiscount(double price) {
    return price * 0.90;
}

void printTotal(double price, double discountRule(double)) {
    cout << fixed << setprecision(2);
    cout << discountRule(price) << endl;
}

Why This Helps

  • printTotal does not need a long chain of discount cases.
  • Each discount rule has one focused job.
  • New behavior can be added without rewriting the caller’s main logic.

The Tradeoff

  • Strategy adds extra structure.
  • That structure pays off when behavior changes or grows.
  • For one simple rule, a direct function may be clearer.

Common Practices

  • Use Strategy when the variation is meaningful.
  • Name each strategy by the behavior it provides.
  • Keep the caller focused on the task, not the rule details.

Strategy vs Singleton

Pattern Main question
Singleton How many instances should exist?
Strategy Which behavior should be used?
  • Singleton controls access to one shared thing.
  • Strategy swaps the rule behind a stable task.

Summary

  • Strategy is useful when behavior should be interchangeable.
  • It replaces tangled choice logic with separate named behaviors.
  • The caller stays stable while the rule can change.

Key Takeaways

  • Design patterns name recurring design choices.
  • Static local storage can support a larger design rule, but it is not the design itself.
  • Singleton controls access to one shared instance.
  • Strategy makes behavior interchangeable behind a stable calling pattern.

Very Brief Review of Static: Good Practices

  • Use static local variables when a function intentionally needs memory between calls.
  • Keep the remembered state small and easy to explain.
  • Prefer visible parameters and return values when state does not need to persist.

Very Brief Review of Static: Common Mistakes

  • Treating static as a shortcut for avoiding parameters.
  • Forgetting that previous calls affect the next call.
  • Assuming local scope means short lifetime.

Very Brief Review of Static: More Examples

int nextId() {
    static int id = 1000;
    id++;
    return id;
}

Each call returns a new value because id keeps its previous value.

Singleton Pattern: Good Practices

  • Use Singleton only when the one-instance rule is part of the problem.
  • Keep the shared object focused on one responsibility.
  • Make shared state easy to find and explain.

Singleton Pattern: Common Mistakes

  • Using Singleton as a disguised global variable.
  • Storing unrelated program data in one shared object.
  • Ignoring the testing cost of hidden shared state.

Singleton Pattern: More Examples

class Logger {
private:
    Logger();

public:
    static Logger& getInstance();
    void write(string message);
};

A logger is a common example because the program may want one coordinated output path.

Strategy Pattern: Good Practices

  • Use Strategy when behavior changes independently from the caller.
  • Give each strategy a clear behavior name.
  • Keep the caller’s main task stable.

Strategy Pattern: Common Mistakes

  • Creating strategies for behavior that never changes.
  • Hiding simple logic behind too many layers.
  • Giving strategies vague names that do not describe the rule.
