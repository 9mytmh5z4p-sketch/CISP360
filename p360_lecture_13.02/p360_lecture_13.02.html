Pointers as Function Parameters

Pass-by-reference without the reference syntax

  • A pointer parameter gives a function the address of the caller’s variable
  • The function can modify the caller’s data by dereferencing that address

Total slides: 5

The mechanism

#include <iostream>
using namespace std;

void doubleIt(int* ptr);

int main() {
    int x = 5;
    doubleIt(&x);         // pass the address of x
    cout << x << "\n";    // 10
    return 0;
}

void doubleIt(int* ptr) {
    *ptr = *ptr * 2;      // modify through the pointer
}
  • &x passes the address; *ptr inside the function reaches back to x
  • Without the dereference, you’d change a local copy of the pointer — not the original value

Pointer parameter vs. reference parameter

void byPointer(int* p) { *p = 99; }   // caller must pass &x
void byRef(int& r)     { r  = 99; }   // caller passes x directly
Pointer parameter Reference parameter
Call site f(&x) f(x)
Inside function *p = ... r = ...
Can be null Yes No
Reassignable Yes No
  • References are cleaner syntax; pointers expose the mechanism and allow nullptr

Passing an array (review)

void fill(int arr[], int size, int val);
// equivalent — int arr[] decays to int*:
void fill(int* arr, int size, int val);
  • Array parameters are always pointers — the caller’s array is always modified
  • This is why const int arr[] (or const int* arr) protects the caller’s data

Summary

  • A pointer parameter accepts an address; the function modifies the caller’s variable through *ptr
  • Pointer parameters and reference parameters achieve the same effect — pointers are more explicit and can be nullptr

Pointers to Constants and Constant Pointers

Where does const go — and what does it protect?

  • const can restrict what a pointer points to, or restrict the pointer itself
  • Position relative to * determines which one is locked

Total slides: 5

Three combinations

int x = 10;
int y = 20;

const int* p1 = &x;    // pointer to const int
int* const p2 = &x;    // const pointer to int
const int* const p3 = &x;  // const pointer to const int
  • Read right-to-left from the variable name: p1 is a pointer (*) to a const int
  • p2 is a const pointer (* const) to a regular int
  • p3 is a const pointer to a const int — nothing can change

Pointer to const — data is read-only

const int* p = &x;

*p = 99;    // error — cannot modify through this pointer
p  = &y;    // OK — the pointer itself can point elsewhere
  • Use this in function parameters to promise “I will not change what you passed me”
  • The underlying variable is not truly const — x itself can still be modified directly

Const pointer — address is locked

int* const p = &x;

*p = 99;    // OK — the data can be modified
p  = &y;    // error — the pointer cannot be reassigned
  • Useful when you need a fixed alias to a variable — the address never changes
  • Less common than pointer-to-const

Summary

  • const int* p locks the data (read-only); int* const p locks the address — position of const relative to * determines which
  • Use const int* in function parameters to protect caller data — it is the most common application

Returning Pointers from Functions

Can a function hand back an address?

  • Yes — but only if what it points to still exists when the caller uses it
  • Returning the address of a local variable is the most common pointer mistake

Total slides: 5

The dangling pointer trap

int* broken() {
    int x = 42;
    return &x;     // x is destroyed when broken() returns
}                  // the returned pointer now points to dead memory
  • x lives on the stack — it is destroyed when the function exits
  • The caller gets an address that refers to memory the program no longer owns
  • Dereferencing it is undefined behavior — it may appear to work, then crash unpredictably

Valid: returning a pointer to static storage

#include <iostream>
using namespace std;

int* getCounter();

int main() {
    int* c = getCounter();
    (*c)++;
    cout << *c << "\n";   // 1
    return 0;
}

int* getCounter() {
    static int count = 0;   // persists across calls
    return &count;
}
  • static local variables live for the entire program — their address is always valid

Valid: returning a pointer to a caller-provided buffer

#include <iostream>
using namespace std;

int* findFirst(int arr[], int size, int target);

int main() {
    int scores[5] = {91, 78, 84, 95, 62};
    int* found = findFirst(scores, 5, 84);
    if (found != nullptr)
        cout << *found << "\n";   // 84
    return 0;
}

int* findFirst(int arr[], int size, int target) {
    for (int i = 0; i < size; i++)
        if (arr[i] == target) return &arr[i];
    return nullptr;
}
  • The array belongs to main — the returned pointer is still valid after the function returns

Summary

  • Never return the address of a local variable — it is destroyed when the function exits
  • Safe returns: address of a static variable, address of heap memory, or address of caller-provided storage

Lab — Multiple Output Parameters

Returning more than one value from a function

  • A function can only return one value — pointer parameters let it write back multiple results
  • This is the pointer-parameter version of the pass-by-reference technique students already know

Total slides: 8

The problem

A function that finds both the minimum and maximum of an array needs to produce two results. Options:

  • Return a struct — requires defining a type
  • Use output parameters — write results directly into caller-provided variables via pointers

Output parameters are the classic C approach and appear throughout legacy code and system libraries.

Function signature

void minMax(const int arr[], int size, int* outMin, int* outMax);
  • outMin and outMax are output parameters — the caller passes addresses of variables to receive the results
  • const int arr[] protects the array — it is read-only inside the function

Task — implement minMax

void minMax(const int arr[], int size, int* outMin, int* outMax) {
    // your code here
}

Rules: - Do not return anything — write results through outMin and outMax - Assume size >= 1

Five minutes.

Solution

void minMax(const int arr[], int size, int* outMin, int* outMax) {
    *outMin = arr[0];
    *outMax = arr[0];
    for (int i = 1; i < size; i++) {
        if (arr[i] < *outMin) *outMin = arr[i];
        if (arr[i] > *outMax) *outMax = arr[i];
    }
}

Calling minMax

#include <iostream>
using namespace std;

void minMax(const int arr[], int size, int* outMin, int* outMax);

int main() {
    int scores[6] = {91, 78, 84, 95, 62, 88};
    int lo, hi;

    minMax(scores, 6, &lo, &hi);
    cout << "Min: " << lo << "\n";   // 62
    cout << "Max: " << hi << "\n";   // 95
    return 0;
}
  • &lo and &hi give minMax the addresses it needs to write the results back

Task — minMaxAvg

Extend minMax to also compute the average:

void minMaxAvg(const int arr[], int size,
               int* outMin, int* outMax, double* outAvg);

Five minutes.

Summary

  • Output pointer parameters let one function produce multiple results without a struct or global
  • The caller declares variables, passes their addresses, and reads the results after the call

Smart Pointers

What if the pointer cleaned up after itself?

  • Raw pointers require manual memory management — easy to forget, easy to double-free
  • Smart pointers wrap a raw pointer and automatically release memory when done

Total slides: 5

unique_ptr — sole ownership

#include <memory>
#include <iostream>
using namespace std;

int main() {
    unique_ptr<int> p = make_unique<int>(42);
    cout << *p << "\n";    // 42 — dereference like a raw pointer
    // no delete needed — memory released when p goes out of scope
    return 0;
}
  • Only one unique_ptr can own a given object at a time — ownership cannot be copied
  • When p goes out of scope, the destructor runs and the memory is freed automatically

shared_ptr — shared ownership

#include <memory>
#include <iostream>
using namespace std;

int main() {
    shared_ptr<int> p1 = make_shared<int>(99);
    shared_ptr<int> p2 = p1;     // both share ownership

    cout << *p1 << "\n";         // 99
    cout << p1.use_count() << "\n";  // 2 — two owners
    // memory freed when the last shared_ptr goes out of scope
    return 0;
}
  • A reference count tracks how many shared_ptrs own the object
  • Memory is released when the count reaches zero

When to use each

Situation Use
One clear owner, no sharing unique_ptr
Multiple owners, shared lifetime shared_ptr
C API, legacy interface, performance-critical raw pointer
  • Default to unique_ptr — it has zero overhead over a raw pointer
  • Prefer make_unique and make_shared over calling new directly

Summary

  • Smart pointers automate memory management using RAII — the destructor frees memory when the pointer goes out of scope
  • Use unique_ptr for sole ownership and shared_ptr for shared lifetime; both live in <memory>

Key Takeaways

  • A pointer parameter gives a function access to the caller’s variable — dereference to read or write
  • const int* protects data; int* const locks the address — position of const relative to * determines which
  • Never return the address of a local variable — it is destroyed when the function exits
  • Smart pointers (unique_ptr, shared_ptr) release memory automatically — prefer them over raw new/delete
