Closer Look — Arrays & Memory

Where does an array actually live?

  • Every variable occupies a location in memory with a numeric address
  • Array elements are stored contiguously — one right after the next, no gaps

Total slides: 4

Contiguous storage

int scores[5] = {91, 78, 84, 95, 62};
Address:  0x1000  0x1004  0x1008  0x100C  0x1010
Value:      91      78      84      95      62
Index:     [0]     [1]     [2]     [3]     [4]
  • Each int occupies 4 bytes — so addresses advance by 4
  • The array’s address is the address of its first element

[Graphic suggestion: five labeled boxes side by side, each 4 units wide, with the hex address below each box and the value inside; an arrow labeled “arr” points to the left edge of box 0]

What an index really does

scores[2]   →   go to scores' start address, skip forward 2 × 4 bytes, read
  • scores[i] is shorthand for “start + i elements forward”
  • This is why out-of-bounds access doesn’t crash immediately — it just reads the wrong memory location

Summary

  • Array elements occupy consecutive memory addresses; each element is one element-size apart
  • The array’s address equals the address of element zero — everything follows from that

The Address Operator

How do we find out where a variable lives?

  • The address operator & returns the memory address of a variable
  • This is the same & used in pass-by-reference — now we’re seeing what it actually does

Total slides: 4

Getting and printing an address

#include <iostream>
using namespace std;

int main() {
    int x = 42;
    cout << x    << "\n";   // 42 — the value
    cout << &x   << "\n";   // 0x... — the address
    return 0;
}
  • &x produces the address of x — a number representing its location in memory
  • cout prints addresses in hexadecimal by default

Addresses of array elements

int scores[5] = {91, 78, 84, 95, 62};

cout << &scores[0] << "\n";   // base address
cout << &scores[1] << "\n";   // base + 4 bytes
cout << &scores[2] << "\n";   // base + 8 bytes
  • Each address is 4 bytes past the previous one — one int apart
  • &scores[0] equals the array’s own address: scores and &scores[0] print the same value

Summary

  • &variable returns the memory address of that variable
  • Array element addresses are spaced by element size — &arr[i] is i elements past the base address

Pointer Variables

Storing an address for later use

  • A pointer variable holds a memory address as its value
  • Two new operators: * in a declaration means “pointer to”; * in an expression means “go to that address”

Total slides: 6

Declaration

int   x   = 42;    // x holds an int value
int*  ptr;         // ptr holds the address of an int
  • int* is read as “pointer to int” — it is a type
  • ptr is a variable whose value is a memory address
  • An uninitialized pointer holds a garbage address — do not dereference it

Assigning an address

int  x   = 42;
int* ptr = &x;     // ptr now holds the address of x
  • The types must match: int* holds the address of an int, double* holds the address of a double
  • ptr and x are separate variables — ptr stores where x lives, not a copy of its value

[Graphic suggestion: two boxes side by side — box labeled “x” containing 42, box labeled “ptr” containing an arrow pointing to x’s box; the arrow is labeled with x’s address]

Dereferencing

int  x   = 42;
int* ptr = &x;

cout << ptr  << "\n";    // the address (e.g. 0x7ff...)
cout << *ptr << "\n";    // 42 — the value at that address

*ptr = 99;               // changes x through the pointer
cout << x    << "\n";    // 99
  • *ptr means “follow the pointer and access what’s there”
  • Writing to *ptr modifies the original variable

Complete example

#include <iostream>
using namespace std;

void show(int* ptr);

int main() {
    int score = 88;
    int* p    = &score;

    show(p);
    *p = 95;
    show(p);
    return 0;
}

void show(int* ptr) {
    cout << "address: " << ptr  << "\n";
    cout << "value:   " << *ptr << "\n";
}

Summary

  • A pointer variable stores a memory address; int* ptr declares a pointer to int
  • & gets an address; * follows it — assign with ptr = &x, read or write with *ptr

Array-Pointer Relationship

Why does an array name look like a pointer?

  • An array name evaluates to the address of its first element
  • This “decay” explains why arrays lose their size when passed to functions

Total slides: 4

Decay in action

int scores[5] = {91, 78, 84, 95, 62};

cout << scores    << "\n";    // address of scores[0]
cout << &scores[0]<< "\n";    // same address
  • scores and &scores[0] produce identical values
  • The array name is not a pointer variable — you cannot reassign it — but it evaluates to an address wherever a pointer is expected

[Graphic suggestion: the five-element array from topic 1, with two arrows both pointing to element 0 — one labeled “scores” and one labeled “&scores[0]” — converging at the same cell]

Subscript is pointer arithmetic

int scores[5] = {91, 78, 84, 95, 62};

scores[2]       // standard subscript notation
*(scores + 2)   // identical — add 2, then dereference
Subscript Pointer form
scores[0] *(scores + 0)
scores[1] *(scores + 1)
scores[i] *(scores + i)
  • The compiler treats both forms identically — subscript is syntactic sugar

Summary

  • An array name decays to a pointer to its first element; arr == &arr[0]
  • arr[i] and *(arr + i) are the same operation — subscript notation is shorthand for pointer arithmetic

Pointer Arithmetic

What happens when you add 1 to a pointer?

  • Adding 1 to a pointer advances it by one element — not one byte
  • The step size is sizeof the pointed-to type

Total slides: 4

Arithmetic scales by element size

int scores[5] = {91, 78, 84, 95, 62};
int* ptr = scores;           // points to scores[0]

cout << *ptr       << "\n";  // 91
cout << *(ptr + 1) << "\n";  // 78
cout << *(ptr + 2) << "\n";  // 84
  • ptr + 1 adds sizeof(int) bytes (4) to the address — it points to the next int
  • ptr + n points to the element n positions forward

Traversal with a pointer

int scores[5] = {91, 78, 84, 95, 62};

for (int* ptr = scores; ptr < scores + 5; ptr++) {
    cout << *ptr << "\n";
}
  • ptr++ moves to the next element each iteration
  • scores + 5 points one past the end — a valid sentinel address but never dereferenced

Summary

  • Pointer arithmetic scales by element size — ptr + 1 steps forward one element, not one byte
  • A pointer can traverse an array the same way an index does; ptr++ is equivalent to i++ in an index loop

Initializing Pointers

What’s in a pointer before you assign it?

  • An uninitialized pointer holds a garbage address — dereferencing it is undefined behavior
  • Always initialize a pointer, even if you don’t have an address to give it yet

Total slides: 3

nullptr and wild pointers

int* p1;             // wild pointer — garbage address, never safe to dereference

int* p2 = nullptr;   // initialized to "no address" — safe to test

if (p2 != nullptr) {
    cout << *p2 << "\n";   // only dereference after confirming it's valid
}
  • nullptr (C++11) is a typed null pointer constant — it compares equal to no valid address
  • A pointer that once pointed to a deleted or out-of-scope variable is called a dangling pointer; it has the same problem as a wild pointer

Summary

  • Always initialize pointers: use nullptr when no target is available yet, or assign a valid address immediately
  • Never dereference an uninitialized or dangling pointer — check for nullptr first

Comparing Pointers

Can we use < and > on pointers?

  • Yes — pointer comparison checks which address comes first in memory
  • Meaningful only when both pointers point into the same array

Total slides: 3

Range-check traversal

int scores[5] = {91, 78, 84, 95, 62};
int* ptr   = scores;          // points to scores[0]
int* limit = scores + 5;      // one past the end — a sentinel

while (ptr < limit) {
    cout << *ptr << "\n";
    ptr++;
}
  • ptr < limit is true as long as ptr has not reached the sentinel
  • == and != are useful for checking whether two pointers refer to the same element

Summary

  • Pointer comparisons (<, >, ==, !=) compare memory addresses numerically
  • The one-past-the-end sentinel pattern (arr + SIZE) gives a clean loop boundary without ever dereferencing an invalid address

VLAs & G++

Runtime array size without a vector?

  • A variable-length array (VLA) uses a runtime value as the array size
  • Legal in C99; a non-standard GCC extension in C++ — not guaranteed to compile on other compilers

Total slides: 3

VLA syntax and trade-offs

int n;
cin >> n;
int arr[n];        // GCC compiles this — MSVC and Clang may not
  • Allocated on the stack — no new, no delete, no heap overhead
  • No .size(), no range-for, no STL compatibility
  • Undefined behavior if n <= 0; stack overflow risk for large n
  • Prefer vector<int> arr(n); — same runtime size, portable, safe

Summary

  • VLAs allow a runtime-determined stack array in G++, but they are a non-standard extension not available in all compilers
  • Use vector for any dynamically sized collection — it is portable, bounds-safe, and STL-compatible

Key Takeaways

  • Every variable has an address; & gets it; a pointer variable stores it; * follows it
  • An array name decays to a pointer to its first element — arr[i] and *(arr + i) are identical
  • Pointer arithmetic scales by element size — ptr + 1 steps one element forward, not one byte
  • Always initialize pointers to nullptr or a valid address; never dereference an uninitialized pointer
