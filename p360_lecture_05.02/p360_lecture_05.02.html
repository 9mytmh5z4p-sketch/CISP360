Function Overloading

Same Name, Different Inputs?

  • When should related functions share one name?
  • Function overloading lets the parameter list explain which version is needed.

Slides 3-17

What Is Function Overloading?

  • Function overloading means defining multiple functions with the same name.
  • Each version must have a different parameter list.
  • The compiler chooses the version that matches the call.

Function Signatures

void display(int value);
void display(double value);
void display(string value);
  • A signature uses the function name and parameter list.
  • Parameter type and count matter.
  • Return type and parameter names do not distinguish overloads.

Example Program: Displaying Values

#include <iostream>
#include <string>
using namespace std;

void display(int value);
void display(double value);
void display(string value);

int main() {
    display(42);
    display(3.75);
    display("Overloaded display");
    return 0;
}

void display(int value) {
    cout << "int: " << value << endl;
}

void display(double value) {
    cout << "double: " << value << endl;
}

void display(string value) {
    cout << "string: " << value << endl;
}

Different Parameter Types

  • display(42) calls the int version.
  • display(3.75) calls the double version.
  • display("Overloaded display") calls the string version.

The function name stays the same because the operation is the same: display one value.

Example Program: Calculating Area

#include <iostream>
using namespace std;

double area(double side);
double area(double length, double width);
double area(double base, double height, char shapeCode);

int main() {
    cout << area(4.0) << endl;
    cout << area(5.0, 3.0) << endl;
    cout << area(6.0, 2.5, 't') << endl;
    return 0;
}

double area(double side) {
    return side * side;
}

double area(double length, double width) {
    return length * width;
}

double area(double base, double height, char shapeCode) {
    if (shapeCode == 't') {
        return base * height / 2.0;
    }
    return base * height / 2.0;
}

Different Number of Parameters

  • area(4.0) computes a square area.
  • area(5.0, 3.0) computes a rectangle area.
  • area(6.0, 2.5, 't') computes a triangle area.

The number of arguments helps the compiler choose the intended version.

Return Type Is Not Enough

int getValue();
double getValue();  // error
  • These functions have the same name and no parameters.
  • The compiler cannot choose between them from the call getValue().
  • Overloads need different parameter lists.

How the Compiler Chooses

void show(int value);
void show(double value);

show(10);
show(10.5);
  • The compiler first looks for an exact match.
  • 10 is an int literal, so it matches show(int).
  • 10.5 is a double literal, so it matches show(double).

Example Program: Formatting Reports

#include <iostream>
#include <iomanip>
#include <string>
using namespace std;

void printReport(string label, int value);
void printReport(string label, double value);
void printReport(string label, int low, int high);

int main() {
    printReport("Count", 42);
    printReport("Average", 87.5);
    printReport("Range", 10, 100);
    return 0;
}

void printReport(string label, int value) {
    cout << label << ": " << value << endl;
}

void printReport(string label, double value) {
    cout << fixed << setprecision(1);
    cout << label << ": " << value << endl;
}

void printReport(string label, int low, int high) {
    cout << label << ": " << low << " to " << high << endl;
}

Ambiguous Calls

void convert(long value);
void convert(double value);

convert(10);
  • The call may require a conversion either way.
  • If no single overload is clearly best, the compiler reports an error.
  • C++ does not guess when overload resolution is ambiguous.

Overloading vs. Different Names

Use overloading when… Use different names when…
the job is conceptually the same the jobs are meaningfully different
parameter lists communicate the case the shared name would hide intent
callers expect one operation family callers need separate actions

Common Practices

  • Use overloading to make related calls read naturally.
  • Keep overloads focused on one shared operation.
  • Avoid overloads that surprise the caller or depend on subtle conversions.

Summary

  • Overloaded functions share one name.
  • The parameter list distinguishes each version.
  • Good overloads make a function family easier to call and easier to read.

Choosing a Parameter Style

What Should Cross the Boundary?

  • Should the function receive a copy?
  • Should it work with caller-owned storage?
  • Should it return a new result?

Slides 18-32

The Four Main Styles

Style What it communicates
pass by value the function receives a copy
return value the function sends back one result
pass by reference the function may update caller data
pass by const reference the function reads larger data without copying

Style 1: Pass by Value

void showAdjusted(int score);
  • Use pass by value when the function needs data but should not change the caller’s variable.
  • The parameter is local to the function.
  • Changes inside the function stay inside the function.

Example Program: Value Parameters

#include <iostream>
using namespace std;

void showAdjusted(int score);

int main() {
    int quizScore = 72;
    showAdjusted(quizScore);
    cout << "Original: " << quizScore << endl;
    return 0;
}

void showAdjusted(int score) {
    score = score + 5;
    cout << "Adjusted copy: " << score << endl;
}

Style 2: Return a New Value

double addTax(double price);
  • Use a return value when the function computes one clear answer.
  • The caller decides whether to store, display, or ignore that answer.
  • The original argument does not need to change.

Example Program: Returning a Result

#include <iostream>
#include <iomanip>
using namespace std;

double addTax(double price);

int main() {
    double price = 20.00;
    double total = addTax(price);
    cout << fixed << setprecision(2);
    cout << "Price: " << price << endl;
    cout << "Total: " << total << endl;
    return 0;
}

double addTax(double price) {
    return price * 1.0825;
}

Style 3: Pass by Reference

void applyDiscount(double& price);
  • Use non-const reference when the function is supposed to update caller-owned data.
  • The & makes the parameter an alias for the caller’s variable.
  • The prototype warns that the argument may change.

Example Program: Updating Caller Data

#include <iostream>
#include <iomanip>
using namespace std;

void applyDiscount(double& price);

int main() {
    double price = 40.00;
    applyDiscount(price);
    cout << fixed << setprecision(2);
    cout << "Discounted price: " << price << endl;
    return 0;
}

void applyDiscount(double& price) {
    price = price * 0.90;
}

Style 4: Pass by Const Reference

double average(const vector<int>& scores);
  • Use const reference for read-only larger data.
  • The function avoids copying the whole object.
  • const prevents accidental changes inside the function.

Example Program: Reading a Vector

#include <iostream>
#include <vector>
using namespace std;

double average(const vector<int>& scores);

int main() {
    vector<int> scores = {90, 84, 96};
    cout << "Average: " << average(scores) << endl;
    return 0;
}

double average(const vector<int>& scores) {
    int total = 0;
    for (int score : scores) {
        total = total + score;
    }
    return static_cast<double>(total) / scores.size();
}

Decision Table

Need Style
read a small value pass by value
compute one answer return value
update caller data pass by reference
read large data safely pass by const reference

Reading the Prototype

double totalCost(double price, double taxRate);
void resetBalance(double& balance);
void displayScores(const vector<int>& scores);
  • The return type shows whether an answer comes back.
  • The & shows whether caller storage may be involved.
  • The const marker shows read-only intent.

Common Practices

  • Prefer return values for one computed answer.
  • Use non-const references only when mutation is part of the contract.
  • Use const reference for read-only large objects such as vectors and strings.

Summary

  • Parameter style is part of the function’s promise.
  • Choose the style that makes data movement obvious.
  • A clear prototype helps the caller trace what can change.

Function Side Effects

What Else Changed?

  • After a function call, what parts of the program might be different?
  • A return value is only one kind of result.
  • Some functions also perform actions or change existing data.

Slides 33-46

What Is a Side Effect?

  • A side effect is a change or action beyond returning a value.
  • Printing, reading input, and changing caller data are side effects.
  • Side effects are not automatically bad, but they must be intentional.

Common Side Effects

  • Displaying output with cout.
  • Reading input with cin.
  • Modifying reference parameters.
  • Changing global or static data.
  • Writing to files or stopping the program.

Return Value vs. Side Effect

double addTax(double price);
void addTax(double& price);
  • The first function returns a new value.
  • The second function changes caller-owned data.
  • The names are similar, but the contracts are different.

Example Program: Returning a Value

#include <iostream>
#include <iomanip>
using namespace std;

double addTax(double price);

int main() {
    double price = 25.00;
    double total = addTax(price);
    cout << fixed << setprecision(2);
    cout << "Original: " << price << endl;
    cout << "Total: " << total << endl;
    return 0;
}

double addTax(double price) {
    return price * 1.0825;
}

Example Program: Modifying by Reference

#include <iostream>
#include <iomanip>
using namespace std;

void addTax(double& price);

int main() {
    double price = 25.00;
    addTax(price);
    cout << fixed << setprecision(2);
    cout << "Updated: " << price << endl;
    return 0;
}

void addTax(double& price) {
    price = price * 1.0825;
}

Visible Side Effects in Prototypes

void updateScore(int& score);
int adjustedScore(int score);
  • A non-const reference parameter signals possible mutation.
  • A return type signals that a value comes back.
  • Read the prototype before assuming what a function changes.

Hidden Side Effects

int totalCalls = 0;

void recordCall() {
    totalCalls++;
}
  • The function changes data that is not in the parameter list.
  • The call site does not show what data may change.
  • Hidden side effects are harder to trace.

Example Program: Hidden State

#include <iostream>
using namespace std;

int nextCallNumber();

int main() {
    cout << nextCallNumber() << endl;
    cout << nextCallNumber() << endl;
    cout << nextCallNumber() << endl;
    return 0;
}

int nextCallNumber() {
    static int callNumber = 0;
    callNumber++;
    return callNumber;
}

When Side Effects Are Useful

  • Output functions should display something.
  • Input functions should read something.
  • Menu actions often update caller-owned program state.
  • Reference parameters are useful when mutation is the point.

When Side Effects Cause Trouble

  • A function changes data the caller did not expect.
  • The same call gives different results because hidden state changed.
  • One function calculates, prints, and mutates data at the same time.

Common Practices

  • Keep calculation functions return-based when possible.
  • Make mutation visible with reference parameters.
  • Avoid hidden state unless the function clearly owns that state.

Summary

  • A side effect is a change outside the return value.
  • Useful side effects make program actions possible.
  • Good function design makes side effects intentional and traceable.

Key Takeaways

  • Function overloading lets related functions share one name when their parameter lists are different.
  • A function signature is the function name plus the parameter types and count.
  • Parameter style is part of the function contract.
  • Side effects are changes outside a returned value; good function design makes them intentional and traceable.
