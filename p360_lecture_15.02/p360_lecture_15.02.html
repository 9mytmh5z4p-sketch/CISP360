Finding the Largest Member Value

int highestPaid(const Employee staff[], int size) {
    int highest = 0;

    for (int i = 1; i < size; i++) {
        if (staff[i].hourlyRate >

            highest = i;

    return highest;

Why Track the Index?

  • The index preserves access to the employee’s ID, name, and rate.
  • The invariant: highest identifies the best record seen so far.

Fixed Size Is the Key Invariant

Safety Before Raw Serialization

  • Confirm that the record is trivially copyable.
  • Define fixed-size text fields and failure behavior.
  • Plan for partial records, deletion, and format changes.

Binary Does Not Mean Secret

  • Binary mode reads and writes bytes without text translation.
  • It does not encrypt the data.
  • It does not guarantee compression.
  • Anyone who knows the layout can interpret the records.

Only Trivially Copyable Records

    "EmployeeRecord cannot be written as raw bytes"

        sizeof(record.name) - 1);

string restoredName = record.name;
  • Reserve one byte for the null terminator.
  • Overlong source names are intentionally truncated.

Short and Corrupted Records

           sizeof(worker));

    cout << "Incomplete record\n";
}

Deleting a Record Is Different

  • Rewrite the remaining records into a replacement file; or
  • set an active member to false and reuse the slot later.
  • A tombstone preserves fixed offsets but leaves unused slots.

Changing the Structure Changes the Format

  • Adding, removing, reordering, or resizing a member changes the record layout.
  • Store a format version when files must outlive one program build.
  • Convert old records rather than interpreting them with a new layout.

When Should We Use Binary Records?

  • Use them for fixed-layout local data under controlled conditions.
  • Use an explicitly defined format when files must survive program changes or move between systems.
  • Never assume raw serialization is universally portable.

Binary Records Summary

  • Equal-sized records make byte offsets and direct access possible.
  • Raw serialization requires a fixed, trivially copyable layout.
  • Portability, validation, deletion, and format evolution remain explicit design responsibilities.

Abstract Data Types — Part 1

What Have We Really Been Building?

  • Are structures only grouped variables, or are they the beginning of a new programming vocabulary?

Slides 107–135

A Type Answers Two Questions

  1. What values can exist?
  2. What operations can be performed on those values?
  • Representation is one possible way to support those answers.

Abstract Data Type

  • An abstract data type defines a set of values and the permitted operations on those values.
  • Its definition emphasizes observable behavior rather than one implementation.
  • “Abstract” means deliberately focused, not vague.

ADT Is Not C++ Syntax

  • struct and class are C++ language mechanisms.
  • Arrays and vectors are storage mechanisms.
  • An ADT is a design description that can have multiple implementations.
  • A data structure implements storage; an ADT specifies behavior.

Interface and Implementation

Interface Implementation
Available operations Stored members
Required inputs Algorithms
Results and failure behavior Supporting data structures

The Client’s View

Time meeting = makeTime(14, 30, 0);
advanceSecond(meeting);
printTime(meeting);
  • Client code asks for operations without managing the algorithm itself.

The Implementer’s View

struct Time {
    int hour;
    int minute;
    int second;
};

void advanceSecond(Time& time);
  • The implementer chooses storage and preserves valid transitions.

A Time ADT

  • Values represent times from 00:00:00 through 23:59:59.
  • Operations create, inspect, display, compare, and advance times.
  • Every operation must leave the value valid.

Representation with a Structure

struct Time {
    int hour = 0;
    int minute = 0;
    int second = 0;
};

Operations with Free Functions

Time makeTime(int hour, int minute, int second);
bool isValid(const Time& time);
void advanceSecond(Time& time);
void printTime(const Time& time);
  • Related function names form an informal interface.

Invariant, Precondition, and Postcondition

  • An invariant must hold for every valid value.
  • A precondition must hold before an operation begins.
  • A postcondition states what the operation guarantees afterward.

Preserving a Valid State

void advanceSecond(Time& time) {
    time.second++;
    if (time.second == 60) {
        time.second = 0;
        time.minute++;
    }
    if (time.minute == 60) {
        time.minute = 0;
        time.hour = (time.hour + 1) % 24;
    }
}

Changing the Representation

struct Time {
    int secondsSinceMidnight;
};
  • Which client statements should need to change if the operations remain stable?

Stable Operations, Replaceable Representation

  • Clients depend on operation names and contracts.
  • Implementers can improve storage or algorithms behind those operations.
  • Direct member access makes representation changes spread through the program.

A Stack as an ADT

  • push adds one value to the top.
  • pop removes the top value.
  • top observes the top value.
  • isEmpty reports whether a top value exists.

Same ADT, Different Implementations

struct ArrayStack {
    int values[100];
    int size;
};

struct VectorStack {
    vector<int> values;
};
  • Both can provide the same stack behavior.

Public Members Break the Boundary

Time meeting = makeTime(14, 30, 0);

meeting.hour = 83;
meeting.minute = -12;
  • Client code can bypass every validating operation.

Free Functions Cannot Enforce Exclusive Access

  • Free functions can encourage correct transitions.
  • Public structure members remain directly writable.
  • Documentation describes the boundary but cannot enforce it.

What We Can Do Before Classes

  • Give operations clear names and contracts.
  • Validate proposed state transitions.
  • Minimize direct member access in client code.
  • State invariants beside the type definition.

The Exposed-State Problem

  • A public structure cannot prevent clients from bypassing its operations.
  • Part 1 can describe the boundary but cannot enforce it.

“Abstract” Does Not Mean Vague

  • Abstraction keeps the details needed to use a value correctly.
  • It postpones details that clients do not need.
  • The interface must still specify precise behavior.

Data Structure Versus ADT

Data Structure Abstract Data Type
Organizes stored data Specifies values and behavior
Array or vector Stack or queue
Implementation choice Client-facing contract

You Already Use ADTs

  • string provides text operations without exposing its character storage strategy.
  • vector provides indexed, resizable sequence behavior.
  • Streams provide formatted input and output without exposing device details.

Kinds of Operations

  • Creators produce new valid values.
  • Observers report information without changing a value.
  • Modifiers replace one valid state with another.

The Contract Includes Failure Behavior

  • What happens when pop is called on an empty stack?
  • What happens when makeTime receives hour 99?
  • Rejecting, reporting, or substituting a value are different contracts.

A Contract for makeTime

  • Preconditions: the supplied hour, minute, and second are in range.
  • Postcondition: the returned Time represents exactly those values.
  • Failure behavior must be chosen and documented.

What Classes Will Add

  • Private representation
  • Controlled public operations
  • Automatic initialization
  • A language-enforced abstraction boundary

ADT Part 2 in 16.01 begins from this exposed-state problem.

ADT Part 1 Summary

  • An ADT is defined by its values, operations, and contracts.
  • Structures and free functions can model an ADT but cannot enforce its boundary.
  • ADT Part 2 will use classes to protect representation and valid state.

Key Takeaways

  • Structures define complete record types from related members.
  • Arrays of structures keep each record intact through searching, sorting, storage, and resizing.
  • Fixed-size binary records enable direct access but depend on a controlled file layout.
  • An ADT is defined by values, operations, and contracts rather than one representation.

Binary Records: Good Practices

  • Use fixed-size, trivially copyable disk-record types.
  • Validate indexes before calculating offsets.
  • Check every read and write operation.
  • Store a format version when records must survive program changes.

Binary Records: Common Mistakes

  • Writing string objects or pointers as raw bytes
  • Confusing human record numbers with zero-based indexes
  • Seeking while the stream remains in a failed state
  • Assuming binary files are encrypted or portable
