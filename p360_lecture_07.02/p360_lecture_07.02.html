Input Validation - Why?

Input Validation - Why?

  • What happens when a program receives bad input?
  • How do we prevent garbage data from corrupting our results or crashing our application?

Total slides: 6

Consider showing a visual comparison of good vs. bad data flow.

GIGO: Garbage In, Garbage Out

Unvalidated input is the most common source of program failures. If bad data enters your program, bad output (or a crash) follows.

Consequences of Unvalidated Input

  1. Crash program
  2. Wrong output
  3. Security vulnerabilities

The Programmer’s Responsibility

  • You control what your program accepts.
  • Trusting the user to provide correct data is naive—they may mistype, misunderstand, or (rarely) act maliciously.
  • Validation is not optional; it is part of correct program design.
  • Your program must check data before using it.

Summary

Input validation prevents crashes, wrong output, and security holes. As a programmer, you are responsible for ensuring that only sensible data enters your program’s logic.

Common Practices

Always check input immediately after reading it from the user. Validate at the boundary—right where data enters your program. Use clear error messages that tell the user exactly what was wrong and what is expected.

Separate validation logic from business logic. Write reusable validation functions rather than scattering checks throughout your code. Document what “valid” means for each input field.

Common Mistakes

Assuming the user will enter correct data is the root of most validation bugs. Never skip validation for “trusted” sources; corners of your code that seemed safe can become entry points later. Vague error messages (“Invalid input”) frustrate users and make debugging harder.

Validating too late (after the data has been used in calculations) allows bad data to corrupt results. Validating in too many places creates inconsistency and maintenance headaches.

More Examples

A program asks for a percentage (0–100). Without validation, entering “150” silently calculates wrong results. With validation, you reject 150 and ask again. A program asks for an age in years. Entering “-5” should be caught immediately, not stored in the database and discovered months later.

A program reads a file path from the user. Validation ensures the path is not empty and exists before attempting to open it.

Input Validation - What?

Input Validation - What?

  • What exactly should we check when validating input?
  • Are there categories of checks that apply across different programs and domains?

Total slides: 7

Consider showing a flowchart with the seven validation types as decision points.

Catagories of Validation Checks

  1. Presence (is there any data at all?)
  2. Data type (is it an int, string, etc.?)
  3. Range / bounds (is a number within min/max?)
  4. Format (does it match a pattern like a zip code or phone number?)
  5. Set membership (is it one of a specific list of allowed values?)
  6. Business logic (does it make sense in context — e.g., end date after start date?)
  7. Consistency (does it agree with other fields already entered?)

1. Presence

  • Presence validation checks whether data exists at all.

  • Example: a form field that requires a name. An empty string fails presence validation. This is the first gate: no data means no valid input.

2. Data Type

  • Data type validation ensures the data matches the expected type (integer, string, floating-point, etc.).

3. Range and Bounds

  • Range validation ensures a numeric value falls between acceptable minimum and maximum boundaries.
  • Prevents overflow, underflow, and nonsensical calculations. For example, validating that an array index is within the array’s actual size.

4. Format

  • Format validation checks whether data matches an expected pattern.
  • Phone numbers follow “123-456-7890” or similar patterns.
  • Zip codes are five digits.
  • Email addresses contain an @ symbol.

5. Set Membership

  • Set membership validation checks whether the input is one of a specific list of allowed values. A menu might accept only “A”, “B”, or “C”; a response field might accept only “Yes” or “No”.

  • This is a whitelist approach: only explicitly approved values are accepted. Everything else is rejected.

6. Business Logic and Consistency

  • Whether the input makes sense in the context of your program’s rules.

  • An end date must be after a start date.

  • A discount percentage cannot exceed 100%.

  • A withdrawal amount cannot exceed account balance.

  • Consistency validation ensures new input agrees with data already in the system (e.g., a zip code matches the city entered earlier).

Summary

The seven validation types—presence, data type, range, format, set membership, business logic, and consistency—cover all common checks. Apply them in order: first verify data exists, then its type, then its bounds, format, and allowed values, then logic and consistency rules.

Common Practices

Apply validation checks in layers, starting with presence and type, then range and format, then business logic. This order prevents crashes before logic errors. Use helper functions for each validation type—one function for range, one for format, etc.—to keep code readable and reusable.

Document what valid input means for each field. Include constraints in comments or assertions. When validation fails, report which type of validation failed (e.g., “Range error: value must be 0–100”) so the user knows how to correct it.

Common Mistakes

Mixing multiple validation types in a single function makes code hard to debug and reuse. Validating inconsistently (checking format in one place, range in another) allows invalid data to slip through.

Forgetting presence and type checks and jumping straight to range checks causes crashes when data doesn’t exist or is the wrong type. Validating too strictly (rejecting valid input due to overly narrow rules) frustrates users.

More Examples

Presence: checking that a user’s name field is not empty. Data type: ensuring an age input can be parsed as an integer. Range: ensuring a month value is 1–12. Format: checking that a phone number matches “XXX-XXX-XXXX”. Set membership: allowing only “M” or “F” for gender. Business logic: checking that a loan end date is after the start date. Consistency: checking that a zip code matches the state the user entered.

Input Validation Architectures

Input Validation Architectures

How should we structure validation in a real program? Are there different patterns for organizing validation logic, each suited to different scenarios?

Total slides: 7

Consider showing a comparison table or flowchart of the eight architectures.

Overview of Eight Architectures

  1. Inline Validation
  2. Validation Layer / Gateway
  3. Loop-Until-Valid (Input Loop)
  4. Whitelist vs. Blacklist Validation
  5. Type-First Validation
  6. Exception-Based Validation
  7. State Machine Validation
  8. Schema / Contract Validation

1. Inline Validation

  • Inline validation embeds checks directly in the code where data is used.
    • After reading an integer, you immediately check if it’s in range before using it in a calculation.
  • This approach is simple and works well for small programs. However, it scatters validation logic throughout your code, making it hard to maintain or reuse checks across multiple locations.

2. Validation Layer / Gateway

  • Validation layer abstracts checks into a separate module.

3. Loop-Until-Valid: The Input Loop

  • Repeatedly asks the user for input until a valid value is received.
    • After reading a value, you validate it; if invalid, display an error and loop back to ask again.
  • This pattern ensures you never proceed with invalid data.
  • It’s the most practical approach for interactive programs in C++ and is your primary tool at this stage.

4. Whitelist vs. Blacklist Validation

  • Whitelist validation explicitly approves only known-good values; everything else is rejected.
    • More secure and correct but requires knowing all valid values upfront.
  • Blacklist validation rejects known-bad values; everything else is accepted.
    • Easier but incomplete—new attack vectors or edge cases slip through.
  • Always prefer whitelist when possible.

5. Type-First Validation

  • Checks the data type before applying other checks.
    • In C++, this means testing whether cin successfully read an integer before checking its range or format.
  • This architecture makes type errors visible immediately.

6. Exception-Based Validation

  • Exception-based validation throws exceptions for invalid data.

7. State Machine Validation

  • State machine validation models valid input sequences.
  • It assumes there is a valid path through the program.
  • Each stop on the path has a limited number of allowable things to do there.
  • The program only allows those limited things to be performed.

8. Schema / Contract Validation

  • Schema validation (contracts) declares what valid data looks like.

Summary

  • There are a number of different architectural designs that we can use for input validation.
  • We can also combine designs (ex: Validation Layer + Loop-Until-Valid)

Common Practices

Use loop-until-valid as your default architecture for interactive input. Always check cin.fail() after reading, and clear the input buffer with cin.clear() and cin.ignore() on failure. Build a library of reusable validation functions (e.g., isValidAge(), isValidPhoneNumber()) and call them consistently.

Document which architecture you’re using and why. For inline validation, keep checks close to where data is used. For loop-until-valid, keep the loop condition clear and the error message helpful.

Common Mistakes

Forgetting to clear the input buffer after a cin failure causes the failed input to remain, blocking all subsequent reads. Mixing inline and loop-based validation in the same program creates inconsistency and confusion.

Using blacklist validation when whitelist is safer (e.g., accepting any string that doesn’t look like a SQL injection, rather than accepting only known-good values). Leaving validation logic scattered throughout the code instead of consolidating it into reusable functions.

More Examples

Loop-until-valid example: ask for an age. Read a value. If it’s out of range, print “Age must be 0–150” and loop back. Type-first example: attempt to read an integer. If cin.fail() is true, clear the buffer, print “Please enter a number”, and ask again.

Whitelist example: accept only menu choices “A”, “B”, or “C”. Reject anything else. Inline validation: after reading a test score, immediately check if it’s 0–100 before using it in a GPA calculation.

Inline Validation Design

Inline Validation Design

What does it mean to validate input right where it’s used, rather than in a separate validation function?

Total slides: 6

Consider a visual diagram showing the flow: Input → if condition → respond (accept or reject).

The Basic Structure

  • Checking input in place.
    • Read a value,
    • Test it with a condition, and
    • React immediately.
  • This is the clearest approach for single-point checks that don’t require re-prompting.

Range Checking Inline

Range checking verifies that a value falls between a minimum and maximum. Use AND (&&) to test both boundaries at once.

const int MIN_AGE = 0;
const int MAX_AGE = 120;

int age;
cin >> age;

if (age < MIN_AGE || age > MAX_AGE) {
    cout << "Invalid age." << endl;
}

Type Failure Check and cin Recovery

When cin >> x fails the input stream enters a failed state. You must clear the error flag and discard bad input.

int count;
if (!(cin >> count)) {
    cout << "Error: expected an integer." << endl;
    cin.clear();                    // Clear the fail flag
    cin.ignore(10000, '\n');        // Skip to next line
}

The isDirty Flag Pattern

When multiple conditions can invalidate input, use a boolean flag to track validity. Set the flag to true if any check fails, then use it to decide how to proceed.

bool isDirty = false;

if (name.length() == 0) {   // test for errors here
    isDirty = true;
}
if (age < 0) {              // and here
    isDirty = true;
}

if (isDirty) {              // display error message here
    cout << "Input rejected." << endl;
}

Summary

Inline validation is the simplest approach: read input, test it with a condition, respond immediately in an if block.

Common Practices

Always use named constants for range boundaries (MIN_VALUE, MAX_VALUE) instead of hardcoding numbers. This makes code self-documenting and simplifies maintenance.

When cin >> fails, always call both cin.clear() and cin.ignore(10000, '\n') to reset the stream and skip the bad input. The 10000 parameter is a large number that safely discards any remaining characters on the line.

Common Mistakes

Forgetting to clear the input stream — After a type mismatch, if you don’t call cin.clear(), the stream remains in a failed state and all subsequent reads will fail silently.

Using magic numbers for bounds — Hardcoding if (age > 120) instead of if (age > MAX_AGE) makes code harder to maintain and understand. Always use constants.

Mixing validation logic — Don’t scatter checks across multiple unrelated if statements. Group related validation checks together or use a isDirty flag to keep logic coherent.

More Examples

Example 1: Email string length check

string email;
cin >> email;

if (email.length() == 0) {
    cout << "Email cannot be empty." << endl;
}

Example 2: Character classification inline

char grade;
cin >> grade;

if (!isupper(grade)) {
    cout << "Grade must be uppercase A–F." << endl;
}

Loop-Until-Valid Design

Loop-Until-Valid Design

  • Inline validation is one and done code.
  • What if we need to keep asking the user for input until they provide a valid answer?

Total slides: 6 Users tend to hate their programs ending because of an incorrect input. Consider a visual flowchart: Prompt → Read → Test condition → if invalid, loop back; if valid, exit.

The Concept: Loop Condition as Validation

  • Keep re-prompting until the user enters valid data.
    • The loop condition IS your validation test.
    • When the condition becomes false, the loop exits and you have a guaranteed valid input.

What kind of loop?

  1. while loop
  2. do while loop
  3. for loop
  4. for-each loop

What kind of loop?

  1. while loop
  2. do while loop
  3. for loop

While Loop Validation

  • Runs AS LONG AS the input is invalid.
    • The key difference from other loops: the condition is “input is bad,” not “input is good.”
  • Requires a priming read before the loop
    • get input once,
    • then check it.
    • if invalid, the loop body re-prompts and checks input again.

Example: Range Validation

const int MIN = 1, MAX = 10;        // const eliminates magic numbers
int val;
cout << "Enter a number (1-10): ";
cin >> val;   //....................... Priming read
while (val < MIN || val > MAX) {
    cout << "Invalid. Enter a number (1-10): ";
    cin >> val;         // ............ another read needed here
}

The loop condition val < MIN || val > MAX means “keep looping if the value is out of range.” Once the user enters a valid value, the condition becomes false and the loop exits.

Example: Input Type Mismatches

When cin fails to read the expected type, the stream enters a fail state. You must clear this state and discard the invalid input before trying again.

#include <limits>   // required for numeric_limits

int val;
cout << "Enter an integer: ";
while (!(cin >> val)) {
    cin.clear();        // ............. reset fail bit
    cin.ignore(numeric_limits<streamsize>::max(), '\n');    // discard leftover input
    cout << "Not an integer. Try again: ";
}

Do While Loop Validation

  • executes the body first, then checks the condition.
  • a do-while never needs a priming read. The prompt and cin are inside the loop and always execute at least once.

Example: Simple Do-While Validation

const int MIN = 1, MAX = 10;
int val;
do {
    cout << "Enter a number (1-10): ";
    cin >> val;
} while (val < MIN || val > MAX);

The loop body runs first, gets the input, and then checks the condition. If invalid, it loops back to the beginning of the do block and re-prompts. No priming read needed.

Example: Handling cin Failures

Stream error handling inside a do-while works the same as in a while loop. Clear the fail state and discard bad input before looping.

int val;
do {        // .............................. prompt loop
    cout << "Enter an integer (1-100): ";
    if (!(cin >> val)) {        // ......... check the type
        cin.clear();
        cin.ignore(numeric_limits<streamsize>::max(), '\n');
        val = -1;  //........................Force invalid to loop again
    }
} while (val < 1 || val > 100);     //.......numeric range validation

While vs Do-While

int val;
cout << "Enter (1-10): ";
cin >> val; //..................priming read
while (val < 1 || val > 10) {
    cout << "Enter (1-10): ";
    cin >> val;
}
int val;
do {    
    cout << "Enter (1-10): ";
    cin >> val;
} while (val < 1 || val > 10);
  • Do-While is more concise.
    • One cin statement instead of two

Summary

Use a while loop with a priming read: get input once before the loop, then loop while the input is invalid. Always include error handling with cin.clear() and cin.ignore() for type mismatches.

A do-while loop validates input without a priming read. The body executes first, then the condition at the bottom decides whether to repeat. Use do-while when the prompt and input must run at least once.

Common Practices

Always include the re-prompt message in the loop body so the user sees it after an invalid entry. The initial prompt appears before the loop; the reprompt appears inside it.

Use named constants for range bounds (MIN_SCORE, MAX_SCORE) in both the output message and the loop condition. This keeps the bounds consistent and self-documenting.

Choose whitelist when you have a small set of acceptable values and blacklist when you want to reject only a few specific inputs. Avoid long lists of whitelisted values; use a range loop instead.

Common Mistakes

Forgetting the reprompt inside the loop — The loop condition repeats, but if you don’t print a new prompt inside the loop body, the user won’t know they need to re-enter data.

Using AND in a whitelist when you need OR — The condition response == "yes" && response == "no" is always false (one variable cannot equal two different strings). Use OR: response != "yes" && response != "no".

Mixing whitelist and blacklist logic — Don’t test response == "yes" && response != "no" in the loop condition. Pick one approach and stick with it for clarity.

Not handling type failures in the loop — If the user enters text when you expect a number, cin >> fails and the loop repeats on old input. Clear the stream inside the loop if needed.

More Examples

Example 1: Do-while reprompt (guarantees at least one prompt)

int age;
do {
    cout << "Enter age (0-150): ";
    cin >> age;
} while (age < 0 || age > 150);

The do-while structure ensures the prompt appears at least once before the condition is checked.

Example 2: Whitelist with multiple acceptable responses

string suit;
cout << "Enter card suit (hearts/diamonds/clubs/spades): ";
cin >> suit;

while (suit != "hearts" && suit != "diamonds" &&
       suit != "clubs" && suit != "spades") {
    cout << "Invalid suit. Try again: ";
    cin >> suit;
}

Increment and Decrement Operators

Increment and Decrement Operators

What is the difference between ++x and x++?

Total slides: 3

x++ and ++x Shortcuts

  • Shorthand for incrementing or decrementing
  • In a loop, the difference rarely matters:
    • for (int i = 0; i < 10; i++) and
    • for (int i = 0; i < 10; ++i) behave identically.
// x++
int x = 0;
x = x + 1;

// --x
x = x - 1;

Pre-Increment vs. Post-Increment

  • Pre-increment (++x): Increment first, then use the value.
  • Post-increment (x++): Use the value first, then increment.
int x = 5;

// Pre-Increment
cout << ++x;  // prints 6, x is now 6

// Post-Increment
cout << x++;  // prints 6, x is now 7

cout << x;    // prints 7

Summary

Pre-increment (++x) and post-increment (x++) both add 1 to the variable, but pre-increment returns the new value while post-increment returns the old value. For simple loop counters, the difference is negligible.

Common Practices

  • Use ++i in loops when possible. Pre-increment is marginally more efficient (though modern compilers optimize this away) and signals intent clearly.
  • Use whichever is clearer in context. In a standalone statement like x++;, the choice doesn’t matter.
  • Know the difference when using in expressions. Only worry about pre vs. post when the return value is used (e.g., assigned or printed).

Common Mistakes

  • Thinking they behave the same everywhere. In y = x++, y gets the old value of x; in y = ++x, y gets the new value.
  • Using post-increment when pre-increment is intended. If you want to increment and then use the value, pre-increment is clearer.
  • Overthinking it in simple loops. In for (int i = 0; i < 10; i++), don’t waste mental energy on the distinction—it doesn’t matter.

Breaking Out of a Loop

Breaking Out of a Loop

How do you exit a loop before the loop condition becomes false?

Total slides: 4

The break Statement

The break statement exits the nearest enclosing loop immediately, skipping any remaining iterations.

while (true) {
    cout << "Enter a value (-1 to quit): ";
    int val;
    cin >> val;

    if (val == -1) break;  // Exit the loop with -1

    cout << "You entered: " << val << endl;
}

cout << "Program ended." << endl;

break vs. Loop Condition

You can achieve the same result by adjusting the loop condition, but break is often clearer when the exit condition is a special value (sentinel).

// Using break with sentinel value
int val;
while (true) {
    cin >> val;             // the -1 is called a sentinel value
    if (val == -1) break;   // break makes intent obvious  
    cout << val << endl;
}

// Using condition (more awkward)
int val = 0;
while (val != -1) {
    cin >> val;
    if (val != -1) cout << val << endl;
}

Summary

Use break to exit a loop early when a sentinel value or stopping condition is detected. It’s clearer and more direct than restructuring the loop condition.

Tracing the Loop

#include <iostream>
using namespace std;

int main() {
    int x = 0;
    while (x < 5) {
        x++;
        if (x == 3) break;
        cout << x << " ";       // prints 1
    }
    cout << endl;               // prints 2
    return 0;                   // output: 1 2
}

  • Iteration 1: x becomes 1. Is x == 3? No. Print 1.

  • Iteration 2: x becomes 2. Is x == 3? No. Print 2.

  • Iteration 3: x becomes 3. Is x == 3? Yes. break executes. Exit loop (nothing printed this iteration).

  • The key insight: the break happens after x is incremented but before the print statement.

Summary

Trace loop execution step-by-step: increment, check the break condition, then execute the remaining statements. The break exits immediately, skipping any code after it in that iteration.

Common Practices

  • Trace one iteration at a time. Don’t try to predict the output mentally—write down the state after each line.
  • Mark the break point. Highlight where the break statement is in the code so you don’t forget to check it.
  • Simulate on paper. Use a table: Iteration | x value | Condition check | Output.

Common Mistakes

  • Forgetting the break happens after the increment. If you think break checks before x++, you’ll get the wrong answer.
  • Printing the break value. When x == 3 and break executes, nothing more is printed in that iteration.
  • Miscounting iterations. Off-by-one errors are common. Double-check your iteration count.

Questions for Reflection

Key Takeaways

  • Validate all input — assume nothing.
  • Match the architecture to the requirement: inline for one-shot checks, loop-until-valid for re-prompting.
  • while requires a priming read; do-while does not.
  • ++ and -- behave differently in prefix vs. postfix position.
  • break exits immediately — use it deliberately.
