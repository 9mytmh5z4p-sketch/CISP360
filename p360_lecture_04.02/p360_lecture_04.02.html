Function Contracts

What Does This Function Promise?

  • A function should make a promise the caller can understand.
  • The name, parameters, and return type tell most of that story.
  • The body explains how the promise is kept.

Slides 3-13

A Contract Has Three Parts

Part Question
Inputs What data does the function need?
Result or action What does the function produce or do?
Boundary What details stay inside the function?

A contract lets the caller use the function without reading every line inside it.

Reading a Prototype as a Contract

double average(vector<int> scores);
void displayScores(vector<int> scores);
  • average needs a vector and sends back a double.
  • displayScores needs a vector and performs an action.
  • The prototype tells the caller how to communicate with the function.

Caller Responsibilities

  • Choose the argument values.
  • Call the function at the right time.
  • Store or use the return value if one comes back.

The caller should not need to know how the function does its internal work.

Function Responsibilities

  • Use the data it receives.
  • Complete one clear job.
  • Keep temporary details inside the function body.

The function should not depend on hidden outside state when parameters can provide the needed data.

Example: Calculating an Average

Caller

  • Collect or already have the scores.
  • Pass the vector to average.
  • Store or display the returned value.

Function

  • Receive the vector parameter.
  • Add the scores and divide by the count.
  • Return the calculated average.

Example: Displaying a Report

Caller

  • Pass the title and calculated values.
  • Decide when the report should appear.
  • Continue after the display function finishes.

Function

  • Format the report.
  • Print the values clearly.
  • Avoid changing the data it was given.

Contract Before Code

vector<int> getScores();
double average(vector<int> scores);
void displayReport(double averageScore);
  • Prototypes can come from an IPO model.
  • They outline the program before the function bodies exist.
  • This keeps design work separate from implementation details.

Common Practices

  • Use verb-based names such as calculateTotal or displayReport.
  • Let the parameter list show what the function needs.
  • Split a function when one contract starts promising unrelated jobs.

Summary

  • A function contract is the promise a function makes to its caller.
  • Clear contracts make modular programs easier to read.
  • The prototype is the first place to check the contract.

Function Boundary Model

What Crosses the Boundary?

  • A function call creates a controlled exchange.
  • Arguments move data from the caller into the function.
  • Return values move one result back to the caller.

Slides 14-24

The Boundary Around a Function

int total(vector<int> scores) {
    int sum = 0;

    for (int score : scores) {
        sum = sum + score;
    }

    return sum;
}

The function body is a separate region with its own local details.

Data Goes In Through Parameters

int answer = total(quizScores);
  • quizScores is the caller’s argument.
  • scores is the function’s parameter.
  • The parameter gives the function a name for the received data.

Data Comes Back Through Return Values

return sum;
  • A return value crosses back to the caller.
  • The function call is replaced by that returned value.
  • The caller decides where the value goes next.

What Stays Inside

  • Local variables stay inside the function.
  • Temporary calculations stay inside the function.
  • Helper details stay inside the function.

This is why a caller can trust a small function without reading every internal step.

What the Caller Cannot See

int total(vector<int> scores) {
    int sum = 0;
    return sum;
}

int main() {
    vector<int> scores = {10, 20, 30};
    cout << total(scores) << endl;
    cout << sum << endl;  // error
    return 0;
}

sum belongs to total(), not to main().

Boundary Model: Calculate Total

Caller

  • Has a vector of scores.
  • Calls total(scores).
  • Stores the returned integer.

Function

  • Receives the vector.
  • Adds every score.
  • Returns the sum.

Boundary Model: Display Results

displayResult(totalScore);
  • The caller supplies the value.
  • No returned value is expected.
void displayResult(int total) {
    cout << "Total: " << total << endl;
}
  • The function prints the result.
  • The visible effect is output.

Design Check: Too Much Crossing

void printReport(string name, int score1, int score2,
                 int score3, int score4, int score5);
  • A long parameter list can be a design warning.
  • The function may need a vector instead of many separate values.
  • The function may be trying to do more than one job.

Summary

  • Parameters are the main path into a function.
  • Return values are the main path back to the caller.
  • Modular design keeps details inside the boundary where they belong.

Passing Vectors

Can a Function Receive a Whole Vector?

  • A vector is a value with many elements inside it.
  • A function can receive that value through a parameter.
  • The contract should say whether the function needs the whole collection.

Slides 25-36

Vector Parameters

void displayScores(vector<int> scores) {
    for (int score : scores) {
        cout << score << endl;
    }
}

The parameter type includes the vector element type.

Calling with a Vector Argument

int main() {
    vector<int> quizScores = {9, 10, 8};

    displayScores(quizScores);

    return 0;
}

The argument is the caller’s vector.

The Parameter Gets a Copy

void displayScores(vector<int> scores);
  • quizScores belongs to main().
  • scores belongs to displayScores().
  • With this syntax, the function receives its own copy.

[Graphic suggestion: show two labeled vector boxes, one in main() and one inside displayScores(), with the same values but different names.]

Reading Vector Data in a Function

int total(vector<int> scores) {
    int sum = 0;

    for (int score : scores) {
        sum = sum + score;
    }

    return sum;
}

The range-based loop reads each value from the parameter vector.

Returning a Vector

vector<int> getScores() {
    vector<int> scores;

    scores.push_back(92);
    scores.push_back(87);
    scores.push_back(95);

    return scores;
}

A function can build a vector and return it to the caller.

Vector Contract Examples

Prototype Contract
int total(vector<int> scores); receive a collection, return one number
double average(vector<int> scores); receive a collection, return one decimal
vector<int> getScores(); create a collection, return it

When Should a Function Receive a Vector?

  • Use a vector parameter when the function needs the collection as a whole.
  • Totals, averages, maximums, and reports often need the whole collection.
  • The caller should pass the vector instead of many separate variables.

Counter Example: One Element at a Time

void displayScore(int score) {
    cout << "Score: " << score << endl;
}

int main() {
    vector<int> scores = {92, 87, 95};

    for (int score : scores) {
        displayScore(score);
    }

    return 0;
}

The function receives one value because it only needs one value per call.

When Should a Function Return a Vector?

  • Return a vector when the function creates a collection.
  • Return a vector when the caller needs the completed collection.
  • Do not return a vector when the caller only needs one calculated answer.

Summary

  • Vectors cross function boundaries through parameters and return values.
  • Passing a whole vector is useful when the function needs the collection.
  • Passing one element at a time is clearer when the function only needs one value.

Scope

Now Scope Has Consequences

  • You have seen block scope, loop scope, and compiler scope errors.
  • Functions make scope affect program design.
  • The question is no longer only “Will this compile?”

Slides 37-48

Quick Recall: Scope Means Visibility

  • Scope is the region where a name can be used.
  • A name declared inside a block belongs to that block.
  • Code outside the block cannot use that name.

This rule becomes more important when a program is split across functions.

From Loop Scope to Function Scope

for (int score : scores) {
    cout << score << endl;
}

// cout << score << endl;  // score is not visible here

Loop bodies create scope. Function bodies do the same thing on a larger design boundary.

Parameters Are Local Names

void displayDouble(int number) {
    cout << number * 2 << endl;
}
  • number receives data from the caller.
  • Inside the function, number is a local name.
  • The caller cannot use number after the function call.

Same Data, Different Names

void displayScores(vector<int> numbers) {
    for (int number : numbers) {
        cout << number << endl;
    }
}

int main() {
    vector<int> scores = {90, 85, 100};
    displayScores(scores);
    return 0;
}

The caller’s name and the parameter name do not need to match.

Same Name, Different Scope

int scoreTotal(vector<int> scores) {
    int total = 0;

    for (int score : scores) {
        total = total + score;
    }

    return total;
}

int main() {
    int total = 0;
    vector<int> scores = {10, 20, 30};
    total = scoreTotal(scores);
    return 0;
}

The two total variables are in different function scopes.

Why This Matters for Modular Design

  • A function can hide local details from the caller.
  • The caller only needs the contract.
  • The function body can use helper variables without cluttering the rest of the program.

Scope is what makes that separation work.

Global Variables Break the Model

int totalScore = 0;

void addScore(int score) {
    totalScore = totalScore + score;
}
  • addScore changes data that is not listed in its parameter list.
  • The hidden dependency makes the contract incomplete.
  • Beginners should prefer parameters and return values.

Scope as a Design Tool

double average(vector<int> scores) {
    int sum = 0;

    for (int score : scores) {
        sum = sum + score;
    }

    return static_cast<double>(sum) / scores.size();
}

sum and score are protected inside the function that needs them.

Common Practices

  • Declare variables close to where they are used.
  • Pass data through parameters instead of globals.
  • Return results instead of making outside variables change invisibly.

Summary

  • Scope controls where names are visible.
  • Function scope makes modular design possible.
  • A clear contract uses scope to keep internal details internal.

Shadow Variables

Which Variable Are We Using?

  • A program can have the same name in more than one scope.
  • The closest visible declaration wins.
  • That rule can make a trace surprising.

Slides 49-59

A Shadow Variable Hides Another Name

int value = 100;

if (value > 0) {
    int value = 5;
    cout << value << endl;
}

cout << value << endl;

The inner value hides the outer value inside the block.

Loop Shadowing Review

int score = 0;
vector<int> scores = {85, 90, 95};

for (int score : scores) {
    cout << score << endl;
}

cout << score << endl;

The loop variable score is separate from the outer score.

Why the Outer Variable Does Not Change

int total = 0;
vector<int> scores = {85, 90, 95};

for (int total : scores) {
    cout << total << endl;
}

cout << "Total: " << total << endl;

The loop prints each score, but the outer total is still 0.

Shadowing in Function Code

int total(vector<int> scores) {
    int total = 0;

    for (int score : scores) {
        total = total + score;
    }

    return total;
}

This local total belongs only to the function call.

Shadowing a Parameter

void displayScore(int score) {
    int score = 0;  // error: same scope
    cout << score << endl;
}

C++ rejects this because the parameter and local declaration are in the same function scope.

Shadowing Inside a Nested Block

void displayScore(int score) {
    if (score < 0) {
        int score = 0;
        cout << score << endl;
    }
}

The nested declaration hides the parameter only inside the if block.

The Bug Pattern

  • The code compiles.
  • The name looks familiar.
  • The program updates a different variable than the one the programmer intended.

Shadowing is dangerous because it looks like ordinary variable use.

Finding Shadow Variables

  • Mark every declaration.
  • Draw a box around each scope.
  • For each use of a name, choose the closest visible declaration.

This tracing method works for loops, nested blocks, and functions.

Summary

  • A shadow variable is a new variable that hides another visible name.
  • Shadowing is a scope bug when the wrong variable receives attention.
  • Use distinct names when different variables have different jobs.

Stack Frame Model

Where Do Function Variables Go?

  • Parameters and local variables need temporary storage.
  • A function call creates that temporary workspace.
  • The workspace disappears when the function returns.

Slides 60-72

A Stack Frame Is One Function Call

  • A stack frame is the workspace for one active function call.
  • It stores that call’s parameters and local variables.
  • Each active function call has its own frame.

[Graphic suggestion: show a single labeled box named average(scores), with parameter scores and local variable sum inside.]

Main Starts the Stack

int main() {
    int score = 90;
    cout << score << endl;
    return 0;
}

At program start, main() is the active frame.

Calling a Function Adds a Frame

int main() {
    int score = 90;
    displayScore(score);
    return 0;
}

When displayScore(score) runs, a new frame is placed above main().

Returning Removes the Frame

void displayScore(int score) {
    cout << score << endl;
}
  • The function frame exists while the function runs.
  • When the function finishes, its frame is removed.
  • Execution continues in the caller.

Parameters Live in the Function Frame

void displayScore(int score) {
    cout << score << endl;
}

score is stored in the displayScore() frame, even if the caller used a different name.

Local Variables Live in the Function Frame

double average(vector<int> scores) {
    int sum = 0;

    for (int score : scores) {
        sum = sum + score;
    }

    return static_cast<double>(sum) / scores.size();
}

scores, sum, and the loop variable belong to the function call.

Return Values Leave First

int square(int number) {
    int result = number * number;
    return result;
}
  • The returned value crosses back to the caller.
  • Then the function frame is removed.
  • The caller keeps the returned value if it stores it.

Stack Frames Explain Scope

  • A caller cannot use names stored inside a function frame.
  • A local variable disappears when its frame disappears.
  • A parameter is local because it lives in the called function’s frame.

Scope is the visibility rule. Stack frames show the storage behind the rule.

Stack Frames Explain Shadowing

void displayScore(int score);

int main() {
    int score = 10;
    displayScore(score);
    return 0;
}

void displayScore(int score) {
    cout << score << endl;
}

Both frames can have a name called score, but they are separate storage.

Trace a Function Call

Step Active Frame Important Data
1 main() score = 10
2 displayScore() score = 10 copy
3 main() function frame is gone

The same value can appear in different frames under the same or different names.

Summary

  • A stack frame is the temporary workspace for one function call.
  • Parameters and local variables live in that frame.
  • Stack frames explain why function variables are temporary, local, and separate.

Key Takeaways

  • A function contract tells the caller what a function needs and what it promises to do.
  • A function boundary keeps caller responsibilities and function responsibilities separate.
  • Vectors can cross function boundaries through parameters and return values.
  • Scope controls where a name is visible.
  • Shadow variables hide another visible name and can make a trace misleading.
  • Stack frames explain why function variables are temporary, local, and separate.

Function Contracts: Good Practices

  • Write the prototype before writing the function body.
  • Use a name that describes the promised action or result.
  • Keep the parameter list limited to data the function actually needs.

Function Contracts: Common Mistakes

  • Naming a function after implementation details instead of purpose.
  • Giving a function hidden dependencies that do not appear in the prototype.
  • Mixing unrelated jobs into one function.

Shadow Variables: Good Practices

  • Use different names when two variables mean different things.
  • Watch for declarations inside nested blocks.
  • Prefer names that describe the role of the value.

  • Declaring a new variable when you meant to update an existing one.
  • Reusing a parameter name inside the same function.
  • Assuming the outer variable changed because the visible name changed.
