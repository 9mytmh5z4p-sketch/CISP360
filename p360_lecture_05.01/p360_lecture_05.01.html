Reference Variables

One Variable, Two Names?

  • Can two names refer to the same stored value?
  • A reference lets an existing variable have another name.
  • That idea matters before we use references in function parameter lists.

Slides 3-11

Last Lecture Model Check

  • Ordinary local variables live in the stack frame for one function call.
  • Ordinary pass-by-value parameters receive copies.
  • A function boundary usually separates caller storage from function storage.

[Graphic suggestion: show two stack frames, main() and changeValue(), each with its own integer box.]

What Is a Reference?

int score = 90;
int& scoreRef = score;
  • scoreRef is another name for score.
  • It does not create a second integer value.
  • Both names refer to the same stored data.

Reference Syntax

int score = 90;
int& scoreRef = score;
  • The & marks scoreRef as a reference.
  • A reference must be initialized when it is declared.
  • The initializer is the variable the reference will name.

Changing Through Either Name

int score = 90;
int& scoreRef = score;

scoreRef = 95;
cout << score << endl;

score = 100;
cout << scoreRef << endl;

Both output statements read the same stored value through different names.

A Reference Is Not Reseated

int first = 10;
int second = 20;
int& alias = first;

alias = second;

After the assignment, first becomes 20. The reference still names first.

Stack Frame View of a Reference

Name Stored Value
score 90
scoreRef alias for score

A reference is best traced as another name pointing to existing storage.

[Graphic suggestion: show one storage box labeled score with a second label scoreRef attached to the same box.]

Summary

  • A reference gives an existing variable another name.
  • Changing the value through either name changes the same storage.
  • References prepare us for parameters that work directly with caller variables.

Pass by Value and Pass by Reference

What Crosses the Boundary?

  • When an argument goes into a function, does the function get the value or the variable?
  • C++ can copy the value or alias the caller’s variable.
  • The parameter list tells us which one is happening.

Slides 12-21

Pass by Value Review

void addFive(int number);

int main() {
    int score = 10;
    addFive(score);
    cout << score << endl;
    return 0;
}

void addFive(int number) {
    number = number + 5;
}

score is still 10 because the function changed a copy.

Pass by Value Trace

Step Frame Name Value
1 main() score 10
2 addFive() number 10
3 addFive() number 15
4 main() score 10

The caller and the function have separate storage.

Pass by Reference

void addFive(int& number) {
    number = number + 5;
}
  • number is a reference parameter.
  • It becomes another name for the caller’s argument.
  • The function can change the caller’s variable.

Pass by Reference Trace

Step Active Code Important Data
1 int score = 10; score stores 10
2 addFive(score); number aliases score
3 number = number + 5; score stores 15
4 cout << score; output is 15

The parameter name and caller name refer to the same storage.

Before and After Example

void resetValue(int number) {
    number = 0;
}

void resetVariable(int& number) {
    number = 0;
}

The first function changes only its copy. The second changes the caller’s variable.

Swap Example

void swapValues(int& first, int& second) {
    int temp = first;
    first = second;
    second = temp;
}

Swapping needs reference parameters because both caller variables must change.

Boundary Model Update

  • Pass by value sends a copy across the function boundary.
  • Pass by reference sends an alias to caller-owned storage.
  • The boundary is still visible in the prototype.

[Graphic suggestion: show pass by value as two boxes and pass by reference as two names attached to one caller box.]

Summary

  • Pass by value protects the caller’s variable from changes inside the function.
  • Pass by reference lets a function change the caller’s variable directly.
  • The & in the parameter list is the signal to trace carefully.

Reference Variables as Parameters

Where Does the Ampersand Go?

  • A reference parameter is marked in the parameter list.
  • The function call still looks ordinary.
  • That makes the prototype especially important.

Slides 22-31

Prototype Syntax

void addTax(double& price);
  • The & belongs with the parameter type.
  • price will alias the caller’s variable.
  • The prototype warns the caller that the argument may change.

Definition Syntax

void addTax(double& price) {
    price = price * 1.0825;
}

The prototype and definition must agree on the reference marker.

Call Syntax Looks Ordinary

int main() {
    double mealPrice = 18.00;
    addTax(mealPrice);
    cout << mealPrice << endl;
    return 0;
}

Nothing in the call itself shows that mealPrice can change.

Reference Parameters Need Variables

addTax(mealPrice);  // works
addTax(18.00);      // error

A non-const reference parameter needs a variable it can alias.

Multiple Reference Parameters

void addTax(double& subtotal, double& tax) {
    tax = subtotal * 0.0825;
    subtotal = subtotal + tax;
}

Both reference parameters can update caller-owned variables.

Input and Output Parameters

void applyDiscount(double percent, double& price) {
    price = price - (price * percent);
}
  • percent gives the function an input value.
  • price gives the function a variable it can update.
  • The parameter list shows which data can change.

Reading Function Contracts with References

void getDimensions(double& width, double& height);
double area(double width, double height);
void displayArea(double area);

Check prototypes for & before assuming a function only reads its arguments.

Summary

  • Reference parameters are marked with & in prototypes and definitions.
  • Calls look ordinary, so the prototype carries the warning.
  • A reference parameter can let a function update caller data.

Static Local Variables

Can a Local Variable Remember?

  • Ordinary local variables are recreated each time a function runs.
  • Sometimes a function needs to keep small internal state.
  • A static local variable keeps its value between calls.

Slides 32-41

Ordinary Locals Reset Each Call

void countCall() {
    int count = 0;
    count++;
    cout << count << endl;
}

Every call creates a new count, so the output is always 1.

The static Keyword

void countCall() {
    static int count = 0;
    count++;
    cout << count << endl;
}

count is initialized once and keeps its value for the rest of the program run.

Static Local Trace

Function Call count Before ++ Output
1 0 1
2 1 2
3 2 3

The variable remembers the result of the previous call.

Scope vs. Lifetime

  • Scope asks where a name can be used.
  • Lifetime asks how long the stored value exists.
  • A static local has local scope but program-long lifetime.

This is the key difference from an ordinary local variable.

Stack Frame Model Contrast

  • Ordinary locals are part of a function call’s stack frame.
  • The frame disappears when the function returns.
  • Static locals are not recreated each time the frame appears.

[Graphic suggestion: show repeated countCall() stack frames using one shared static count box outside the frame.]

Function Call Counter Example

int nextCallNumber() {
    static int callNumber = 0;
    callNumber++;
    return callNumber;
}

The function owns the counter, but callers can receive the next value.

When Static Locals Are Useful

  • Counting how many times a function has run.
  • Remembering a small setting controlled by the function.
  • Performing one-time setup inside a helper function.

Use static locals for function-owned state, not shared program-wide data.

Summary

  • Static local variables keep their values between function calls.
  • Their names remain visible only inside the function.
  • They provide internal memory without exposing a global variable.

Default Parameters

Can an Argument Be Optional?

  • Some function calls use the same value most of the time.
  • A default parameter lets the function provide that common value.
  • The caller can still override the default when needed.

Slides 42-51

Default Parameter Idea

void printLine(char symbol = '-');

If the caller leaves out the argument, C++ uses the default value.

Prototype with Defaults

void printLine(char symbol = '-', int length = 20);
  • Put default values in the prototype.
  • The prototype is the caller-facing contract.
  • Do not repeat the defaults in the function definition.

Calling with All Arguments

printLine('=', 40);
  • symbol receives '='.
  • length receives 40.
  • Both defaults are overridden.

Calling with Some Arguments

printLine('*');
  • symbol receives '*'.
  • length uses the default value 20.
  • Supplied arguments fill parameters from left to right.

Right-to-Left Rule

void displayBox(string text, char border = '*', int width = 30);

Once a parameter has a default, the parameters after it need defaults too.

Defaults and Function Contracts

void displayTotal(double total, int decimals = 2);
  • The required data still appears first.
  • The default communicates the common case.
  • The caller sees what can be customized.

Small Formatting Example

void printLine(char symbol = '-', int length = 20);

void printLine(char symbol, int length) {
    for (int i = 0; i < length; i++) {
        cout << symbol;
    }
    cout << endl;
}

The function can be called as printLine();, printLine('=');, or printLine('*', 10);.

Summary

  • Default parameters make common calls shorter.
  • Defaults belong in the prototype.
  • Defaulted parameters must be grouped at the end of the parameter list.

Exit

Return or Stop Everything?

  • A normal return leaves one function.
  • exit() ends the whole program.
  • That difference matters when tracing control flow.

Slides 52-61

Return from a Helper Function

int doubleValue(int number) {
    return number * 2;
}

The returned value crosses back to the caller, and execution continues there.

Return from main()

int main() {
    cout << "Done" << endl;
    return 0;
}

Returning from main() is the normal way to finish a successful program.

The exit() Function

exit(0);
exit(1);
  • exit() terminates the program immediately.
  • The value passed to exit() is the program’s status code.
  • Code after the call does not continue.

Including the Header

#include <cstdlib>
using namespace std;

The exit() function is declared in <cstdlib>.

Exit Status Codes

Code Meaning
0 successful completion
nonzero abnormal termination

Status codes let the operating system know how the program ended.

Exit Inside a Function

void requirePositive(int value) {
    if (value <= 0) {
        cout << "Fatal error: value must be positive." << endl;
        exit(1);
    }
}

This function stops the program because the program cannot continue safely.

Boundary Model Contrast

  • return sends control back through the function boundary.
  • A return value can cross back to the caller.
  • exit() does not return through the boundary at all.

[Graphic suggestion: show return as an arrow back to the caller and exit() as a stop marker for the whole program.]

Summary

  • Use return for normal function flow.
  • Use return 0; from main() for normal program completion.
  • Use exit() only when the whole program should stop immediately.

Menus and Functions

Who Owns Each Menu Job?

  • A menu program has repeated responsibilities.
  • Functions can keep each responsibility small.
  • main() should coordinate the pieces instead of doing every job itself.

Slides 62-74

Menu Program Responsibilities

  • Display the choices.
  • Read the user’s choice.
  • Process the selected action.
  • Repeat until the user chooses to quit.

Each responsibility can become a function or a small group of functions.

Display Menu Function

void displayMenu() {
    cout << "1. Deposit" << endl;
    cout << "2. Withdraw" << endl;
    cout << "3. Show balance" << endl;
    cout << "4. Quit" << endl;
}

This function owns the menu text.

Get Choice Function

int getChoice() {
    int choice;
    cout << "Choice: ";
    cin >> choice;
    return choice;
}

The function reads one value and returns it to the caller.

Action Functions

void showBalance(double balance) {
    cout << "Balance: $" << balance << endl;
}

An action function should do one menu job clearly.

Main as the Coordinator

int main() {
    double balance = 0.0;
    int choice;

    displayMenu();
    choice = getChoice();

    return 0;
}

main() owns the program flow and delegates details to helper functions.

Dispatching the Choice

if (choice == 1) {
    addDeposit(balance);
} else if (choice == 2) {
    makeWithdrawal(balance);
} else if (choice == 3) {
    showBalance(balance);
}

The dispatch code connects a user choice to the matching function call.

Menu Loop Shape

do {
    displayMenu();
    choice = getChoice();
    processChoice(choice, balance);
} while (choice != 4);

A menu loop keeps showing choices until the quit option is selected.

Reference Parameters in Menu Actions

void addDeposit(double& balance) {
    double amount;
    cout << "Deposit amount: ";
    cin >> amount;
    balance = balance + amount;
}

The reference parameter lets the action update the caller’s balance.

Default Parameters in Display Helpers

void printDivider(char symbol = '-', int length = 30);

Defaults keep common formatting calls short while still allowing customization.

Complete Menu Skeleton

void displayMenu();
int getChoice();
void processChoice(int choice, double& balance);
void addDeposit(double& balance);
void makeWithdrawal(double& balance);
void showBalance(double balance);

The prototypes show the menu architecture before the function bodies are written.

Summary

  • Menus stay manageable when each function owns one responsibility.
  • Reference parameters let menu actions update caller-owned state.
  • Default parameters can simplify repeated display helper calls.

Key Takeaways

  • A reference gives an existing variable another name.
  • Pass by reference lets a function work with caller-owned storage.
  • Static local variables keep function-owned state between calls.
  • Default parameters make common function calls shorter.
  • exit() terminates the program instead of returning to a caller.
  • Menu programs are clearer when display, input, dispatch, and actions are separate functions.

Reference Variables: Good Practices

  • Initialize a reference at the point of declaration.
  • Use references when alias behavior is intentional and useful.
  • Trace the original variable and the reference as one shared stored value.

Reference Variables: Common Mistakes

  • Thinking a reference creates a second independent variable.
  • Trying to declare a reference without initializing it.
  • Expecting an assignment to make a reference point to a different variable.

Static Local Variables: Good Practices

  • Use static locals only for state owned by one function.
  • Keep the variable name local when outside code should not access it.
  • Trace the value across calls instead of resetting it each call.

Static Local Variables: Common Mistakes

  • Expecting a static local to reset every time the function runs.
  • Treating static locals as a replacement for clear data flow.
  • Confusing local scope with short lifetime.

Static Local Variables: More Examples

    id++;
    return id;

Exit: Good Practices

  • Prefer normal return flow when the program can continue.
  • Use exit() only for program-level termination.
  • Include <cstdlib> when using exit().

Exit: Common Mistakes

  • Using exit() as a shortcut instead of designing normal control flow.
  • Expecting code after exit() to run.
  • Confusing a helper function return with ending the whole program.

Exit: More Examples

        cout << "Could not open file." << endl;
        exit(1);
    }
}

The function ends the program when a critical requirement fails.
