How the Compiler Works

How the Compiler Works

What happens when you press “build” in your IDE? The compiler transforms your source code through several distinct stages into an executable program. Understanding this process helps you debug faster and write better C++ code.

Total slides: 11

The Compilation Process Overview

The path from source code to a running program involves three major stages: preprocessing, compilation, and linking. Each stage processes your code in a specific way and produces intermediate output. Together, they transform human-readable C++ into machine-executable code.

Stage 1: Preprocessing

The preprocessor runs before actual compilation and handles directives that start with #. It expands macros, inserts file contents from #include statements, and removes comments. The output is still C++ source code — just with all directives resolved.

Preprocessing Directives

  • #include <file>: Inserts the entire contents of a header file into your source. The preprocessor does not validate syntax.
  • #define NAME value: Creates a textual substitution; every occurrence of NAME is replaced with value before compilation.
  • #ifdef / #if: Conditionally includes or excludes code blocks based on whether symbols are defined.

Stage 2: Compilation

The compiler takes preprocessed source code and performs syntax checking to verify it follows C++ grammar rules. It then translates valid code into assembly language or object code specific to your target processor. A single .cpp file becomes one .o (object) file.

What the Compiler Checks

  • Syntax: Does your code follow C++ grammar rules?
  • Types: Do variable types match in assignments, function calls, and operations? C++ is statically typed — type mismatches are caught here, not at runtime.
  • Scope and declarations: Are variables and functions declared before use? Do names conflict?

Stage 3: Linking

The linker takes multiple object files and combines them into one executable. It also resolves references to library functions. If a function is called but not defined anywhere, the linker reports an “undefined reference” error.

Linking: How It Works

  • Symbol resolution: The linker matches each function call to its definition across all object files and libraries.
  • Library linking: Standard headers like <iostream> declare functions; the linker connects your calls to their implementations in the standard library.
  • If math.o is missing, linking fails with “undefined reference to add()” even though main.cpp compiled cleanly.

Why This Matters for C++

C++ performs type checking and most error detection at compile time, not at runtime. Bugs are caught before you run the program. Compilation takes longer than interpreted languages, but the result is fast native code with no hidden runtime type checks.

Compile-Time vs Runtime Errors

C++ catches type mismatches, undefined references, and syntax errors during compilation. Languages like Python defer these checks to runtime — errors only appear when that code path executes. In C++, you find problems immediately during development.

Summary

The compiler transforms source code through preprocessing, compilation, and linking. Understanding these stages helps you interpret error messages and locate bugs faster.

Appendix

Common Practices

  • Enable compiler warnings with -Wall -Wextra (g++/clang); warnings often flag real bugs that still compile.
  • Place declarations in .h files and implementations in .cpp files to keep linking straightforward.
  • Use #pragma once or header guards to prevent duplicate includes that cause linker errors.

Common Mistakes

  • Including .cpp files instead of .h files causes multiple-definition linker errors.
  • Forgetting to link against a required library produces “undefined reference” errors at link time, not compile time.
  • Assuming a program is correct because it compiled — the compiler only checks syntax and types, not logic.

More Examples

A function add() declared in math.h and defined in math.cpp: the preprocessor copies math.h into main.cpp, the compiler generates main.o and math.o, and the linker resolves the call. If math.o is omitted from the build command, linking fails — even though both source files compiled cleanly.

Practice Problems

  1. Split a simple program into two .cpp files. Compile each file separately, then link them manually.
  2. Add a #define constant and observe how the preprocessor substitutes it before compilation.
  3. Intentionally remove a function definition and observe the exact linker error message.

The Symbol Table

The Symbol Table

What problems does the compiler face when tracking every identifier in your code, and how does it solve them? The symbol table is the compiler’s internal database that records every identifier — variable, function, or type — along with its name, data type, scope, and memory location. Understanding it helps you recognize compile-time errors and write code the compiler can reason about cleanly.

Total slides: 9

What Is the Symbol Table?

The symbol table is a compiler-maintained record that maps identifiers to their properties. Every time you declare a variable, function, or type, the compiler adds an entry. This table drives type checking, scope resolution, error detection, and code generation.

What Goes in the Symbol Table?

  • Name: The identifier as written in code (count, processData).
  • Data type: The declared type (int, double, string) — used for type checking.
  • Scope: Whether the identifier is local, global, or block-scoped — determines visibility.
  • Memory location: Where the variable lives, or the entry point of a function.

How the Compiler Uses It

  • Type checking: When you assign int x = "hello", the compiler checks the table and reports a type mismatch.
  • Scope resolution: When you reference count, the compiler searches from innermost scope outward.
  • Duplicate detection: If you declare the same variable twice in the same scope, the compiler finds the first entry and rejects the second.
  • Code generation: The compiler uses addresses stored in the table to read and write each variable.

Undeclared Variable Error

int main() {
    count = 10;   // Error: 'count' was not declared in this scope
    return 0;
}

The compiler searched the symbol table and found no entry for count. Declare it before you use it.

Redeclaration Error

int main() {
    int x = 5;
    int x = 10;   // Error: redeclaration of 'int x'
    return 0;
}

The compiler found x already in the symbol table for this scope. Each identifier can have only one entry per scope.

Type Mismatch Error

int main() {
    int age = 25;
    age = "twenty-five";  // Error: invalid conversion from 'const char*' to 'int'
    return 0;
}

The symbol table records age as type int. The compiler checks the incoming value’s type and rejects the assignment.

Scope and the Symbol Table

Each scope (global, function, block) has its own region in the table. When you reference an identifier, the compiler searches from the innermost scope outward. A local variable with the same name as a global one shadows the global — the compiler finds the local entry first.

int x = 5;           // global
void func() {
    int x = 10;      // local — shadows global x
    cout << x;       // prints 10
}

Summary

The symbol table is the compiler’s record of every identifier: its name, type, scope, and location. Type mismatches, undeclared variables, and redeclarations all stem from symbol table violations — the compiler cannot build what it cannot track.

Appendix

Common Practices

  • Declare variables before use so the compiler can add them to the table before they are referenced.
  • Keep variables as local as possible — narrow scope reduces the number of entries in the table and makes code easier to reason about.
  • Avoid reusing the same name in nested scopes unless shadowing is intentional and clearly documented.

Common Mistakes

  • Writing count = 5; without a prior declaration — the symbol table has no entry and the compiler stops with an error.
  • Declaring the same variable twice in one scope — the table already has the entry and rejects the duplicate.
  • Accidentally shadowing a global variable with a local one of the same name, causing the global to silently go unused.

More Examples

void process() {
    int result = 0;     // local — own symbol table entry
}

int result = 100;       // global — separate entry; no conflict

vector<int> data;
data.push_back("x");  // Error: table says data holds int, not string

Practice Problems

  1. Write a short program with a global and a local variable of the same name. Predict what the program will print.
  2. Declare a function before defining it (forward declaration). Sketch the symbol table entries at the point of declaration.
  3. Declare two variables with the same name in the same scope. Observe and explain the compiler error message.
  4. Assign a string literal to an int variable without a cast. Read the error message and identify what type the table expected.

Overflow and Underflow

Overflow and Underflow

What happens when an integer exceeds its maximum value or drops below its minimum? These silent errors produce wrong results with no warning. This section covers how overflow and underflow work and how to avoid them.

Total slides: 9

What Is Integer Overflow?

Overflow occurs when a value exceeds the maximum representable value for its type. For unsigned integers, the value wraps back to zero — this behavior is well-defined. For signed integers, overflow is undefined behavior in C++, meaning the program may do anything.

Emphasize: signed overflow = undefined behavior. The program may appear to work, crash, or produce incorrect results silently.

What Is Integer Underflow?

Underflow occurs when a value drops below the minimum representable value for its type. For unsigned integers, decrementing past zero wraps to the maximum value — also well-defined. For signed integers, going below the minimum is again undefined behavior. Floating-point underflow is different: very small values round to zero rather than wrapping.

Integer Ranges

Type Minimum Maximum
short −32,768 32,767
int −2,147,483,648 2,147,483,647
long −9,223,372,036,854,775,808 9,223,372,036,854,775,807
unsigned int 0 4,294,967,295
unsigned short 0 65,535

These values are platform-dependent. Use <limits> rather than hardcoding them.

[Graphic suggestion: Visual number line showing where wrap-around occurs for a short.]

Querying Limits at Runtime

#include <limits>
#include <iostream>
using namespace std;

int main() {
    cout << "int max:   " << numeric_limits<int>::max()          << "\n";
    cout << "int min:   " << numeric_limits<int>::min()          << "\n";
    cout << "short max: " << numeric_limits<short>::max()        << "\n";
    cout << "uint max:  " << numeric_limits<unsigned int>::max() << "\n";
}

numeric_limits<T> from <limits> is the portable way to query exact ranges for any type.

Overflow in Action

short s = 32767;    // max value for short
s++;
cout << s;          // -32768  (wraps to minimum)

unsigned int u = 4294967295;   // max for unsigned int
u++;
cout << u;          // 0  (wraps to zero)

Unsigned wrap-around is predictable. Do not rely on signed wrap-around — the behavior is undefined and the compiler may optimize it away.

Detecting Overflow Before It Happens

#include <limits>
using namespace std;

bool wouldOverflow(int a, int b) {
    return (b > 0 && a > numeric_limits<int>::max() - b);
}

int main() {
    if (wouldOverflow(2000000000, 200000000)) {
        cout << "Overflow detected — operation skipped.\n";
    }
}

Check before the operation, not after. Once undefined behavior occurs, the result cannot be trusted.

Prevention Strategies

  • Use a larger type (long long) when calculations may exceed int range.
  • Validate user input against type limits before performing arithmetic.
  • Treat compiler warnings about integer conversion seriously — they often indicate overflow risk.
  • For unsigned loops with decrement, verify the loop terminates before reaching zero.

Summary

Unsigned integer overflow and underflow wrap around predictably. Signed integer overflow is undefined behavior and must be prevented — not merely detected after the fact. Use numeric_limits<T> to query ranges and choose the right type for your data.

Appendix

Common Practices

  • Include <limits> and use numeric_limits<T>::max() / numeric_limits<T>::min() — never hardcode type bounds.
  • Perform overflow checks before arithmetic, not after.
  • Prefer long long for accumulations and counters that may grow large.
  • Use unsigned types only when negative values are impossible and wrap-around is understood.

Common Mistakes

  • Assuming signed overflow wraps reliably — the standard gives no guarantee, and optimizing compilers exploit this.
  • Using unsigned char or unsigned short as a loop counter that decrements to zero — decrementing past zero wraps to a large value and creates an infinite loop.
  • Ignoring compiler warnings about narrowing conversions that can silently lose the high bits of a value.

More Examples

// Signed multiplication overflow
int x = 100000;
int y = 100000;
int z = x * y;   // Undefined behavior — exceeds INT_MAX

// Unsigned char wrap-around
unsigned char c = 255;
c += 2;
cout << (int)c;  // prints 1 (wraps: 255 + 2 = 257, 257 % 256 = 1)

// Float underflow
float f = 1e-50f;
cout << f;       // prints 0.0 (too small to represent)

  2. Declare an unsigned char loop counter and count from 0 to 300. Observe and explain the output.
  3. Use numeric_limits to print the min and max for short, int, long, and their unsigned variants.
  4. Demonstrate the difference in behavior between signed and unsigned overflow with a short code example.

What’s the Output? — Overflow & Underflow

Overflow and underflow produce values outside a type’s representable range. Before running each example, predict the output — then check your reasoning against the answer. Getting the prediction wrong is the point: it reveals exactly where your mental model needs work.

Total slides: 11

How to Use These Exercises

Read the code, predict what prints, then advance to the answer slide. If your prediction was wrong, focus on the answer explanation — not just the output, but why that output occurs.

What’s the Output? — Short Overflow

#include <iostream>
using namespace std;

int main() {
    short s = 32767;
    s++;
    cout << s << endl;
}

Answer: -32768

Output — Short Overflow

-32768

A short holds values from −32,768 to 32,767. Incrementing 32,767 moves past the maximum. The bit pattern rolls over to the minimum value, −32,768. This is undefined behavior for signed types — many compilers produce this result on x86, but it is not guaranteed.

Emphasize: signed overflow = undefined behavior. Students often think “it wraps, so it’s fine.” It is not.

What’s the Output? — Unsigned Underflow

#include <iostream>
using namespace std;

int main() {
    unsigned int u = 0;
    u--;
    cout << u << endl;
}

Answer: 4294967295

Output — Unsigned Underflow

4294967295

Decrementing an unsigned int past zero wraps to the maximum value. On a 32-bit system, that is 2³² − 1 = 4,294,967,295. Unlike signed overflow, unsigned wraparound is well-defined in C++ — this behavior is guaranteed.

Contrast with slide 3: unsigned wrap-around is safe and predictable; signed wrap-around is undefined. This distinction matters.

What’s the Output? — INT_MAX + 1

#include <iostream>
#include <climits>
using namespace std;

int main() {
    int x = INT_MAX;
    cout << x + 1 << endl;
}

Answer: undefined behavior — on most x86 compilers this prints -2147483648, but it is NOT guaranteed.

Output — INT_MAX + 1

-2147483648   ← common result, not guaranteed

Adding 1 to INT_MAX (2,147,483,647) causes signed integer overflow — undefined behavior. Many compilers produce −2,147,483,648 due to two’s complement wrap, but an optimizer may eliminate the operation entirely or produce something else. Never rely on this.

Good moment to ask: “If the output is undefined, why does it appear consistent?” Answer: modern x86 hardware happens to wrap, but the compiler isn’t obligated to use that behavior.

What’s the Output? — Float Underflow

#include <iostream>
#include <iomanip>
using namespace std;

int main() {
    float f = 1e-50f;
    cout << fixed << setprecision(10) << f << endl;
}

Answer: 0.0000000000

Output — Float Underflow

0.0000000000

The smallest normalized float is approximately 1.18 × 10⁻³⁸. The value 1 × 10⁻⁵⁰ is far below that range and rounds to zero. Floating-point underflow rounds to zero rather than wrapping — precision is lost silently. No error, no warning.

Float underflow is gradual: the value becomes 0.0 without crashing. Contrast with integer underflow. Practical risk: silent loss of precision in scientific or financial calculations.

Pattern: What to Watch For

  • Signed types: never exceed INT_MAX or go below INT_MIN — the behavior is undefined.
  • Unsigned types: expect wrap-around; whether that is a bug depends on intent.
  • Float: very small results silently become zero — check for underflow in precision-sensitive code.

Summary

Signed integer overflow is undefined — do not rely on wrap-around. Unsigned wrap-around is defined and predictable. Floating-point underflow rounds to zero silently. All three require awareness and preventive design.

Appendix

Common Practices

  • Use numeric_limits<T>::max() and min() to check values before operations that might overflow.
  • Treat signed and unsigned overflow differently — they have different rules and different risks.
  • Add range-checking to input validation when values will be used in arithmetic.

Common Mistakes

  • Assuming signed overflow wraps reliably across all compilers — it does not.
  • Using unsigned to “avoid” negatives but not accounting for underflow wrap when decrementing.
  • Ignoring floating-point underflow because no exception is raised — silent precision loss is a real bug.

More Examples

unsigned char c = 0;
c--;
cout << (int)c;   // 255 (wraps to max for unsigned char)

unsigned char c2 = 255;
c2 += 2;
cout << (int)c2;  // 1 (wraps: 255 + 2 = 257, 257 % 256 = 1)

  2. Write a safe increment function for int that prints an error instead of overflowing.
  3. Predict the output: unsigned int x = 1; cout << (x - 2);
  4. What is the minimum positive float your system can represent? Use <limits> to find out.

Data Type Conversion and Type Casting

Data Type Conversion and Type Casting

When should we convert between numeric types, and what are the safe and unsafe ways to do it? C++ converts between types automatically in some cases, and lets you force conversion in others. Both paths have traps.

Total slides: 11

What Is Type Conversion?

Type conversion occurs when a value of one type is used where a different type is expected. C++ handles this two ways: implicitly (automatically, without your input) and explicitly (through a cast you write). The distinction matters because implicit conversions can hide bugs.

Implicit Conversion: Promotion

Promotion happens when a smaller type is assigned to a larger type. No data is lost. C++ performs this automatically.

  • char → int → long → double (widening)
  • Occurs in assignments, arithmetic expressions, and function calls

Implicit Promotion — Examples

int x = 5;
double y = x;        // int promoted to double; y = 5.0

char c = 'A';
int val = c;         // char promoted to int; val = 65

int a = 10;
double r = a / 2.0;  // a promoted before division; r = 5.0

Implicit Conversion: Narrowing

Narrowing happens when a larger type is assigned to a smaller type. Data loss is likely. The compiler may warn but will not always stop you.

  • double → int: decimal part is truncated, not rounded
  • int → short: high bits may be lost
  • long → int: same risk

Narrowing — Examples

double x = 3.7;
int y = x;             // Warning: y = 3 (decimal discarded)

long bigNum = 100000;
short s = bigNum;      // Warning: may overflow

int a = 7;
int b = 2;
int result = a / b;    // result = 3, not 3.5

C-Style Casts

C-style syntax: (TargetType)value. Works, but is not recommended in modern C++.

  • Hard to search for in a large codebase
  • Hides intent — it’s not clear what kind of conversion is happening
  • Performs dangerous conversions without warning
  • Carried over from C; you will see it in older code

static_cast: The C++ Way

static_cast<TargetType>(value) is the preferred cast for numeric conversion. It makes your intent explicit and is easy to find with a search. Use it any time you need an intentional narrowing or explicit promotion.

static_cast — Examples

double x = 3.7;
int y = static_cast<int>(x);              // explicit truncation; y = 3

int dividend = 7;
int divisor = 2;
double r = static_cast<double>(dividend) / divisor;  // r = 3.5

char letter = 'Z';
int ascii = static_cast<int>(letter);     // ascii = 90

When to Cast and When Not To

Use a cast to express intent — when you know truncation or conversion is correct. Do not cast to silence compiler warnings without fixing the underlying mismatch. A cast that hides a narrowing bug is worse than the warning it suppressed.

Summary

Implicit promotion is safe and automatic. Narrowing loses data — use static_cast<> to make it explicit and intentional. Never cast just to silence a warning; the warning usually means there is a real issue to address.

Appendix

Common Practices

  • Use static_cast<> for all intentional numeric conversions in C++.
  • Prefer implicit promotion (small → large); be explicit and deliberate with narrowing.
  • When mixing integer and floating-point in division, cast one operand to double to preserve precision.

Common Mistakes

  • Integer division: 7 / 2 returns 3, not 3.5. Cast at least one operand.
  • Silent narrowing: assigning a double to an int loses the decimal without an error.
  • Using a C-style cast in new code — static_cast<> is preferred and easier to review.

More Examples

// Correct way to get floating-point average
int total = 15, count = 4;
double avg = static_cast<double>(total) / count;  // 3.75

// Extracting cents from a price
double price = 19.99;
int cents = static_cast<int>(price * 100) % 100;  // 99

// Explicit comparison of mixed types
int a = 5;
double b = 5.5;
if (static_cast<double>(a) < b) {
    cout << "a is smaller\n";
}

What’s the Output? — Type Casting

What’s the Output? — Type Casting

Predicting output forces you to think through what the compiler actually does with types. Type casting is where many predictions fail — integer division, implicit promotion, and explicit casts all follow precise rules. Getting a prediction wrong here is useful: it identifies exactly which rule you had wrong.

Total slides: 12

How to Use These Exercises

Read the code, predict the output, then advance. If your prediction was wrong, the explanation on the answer slide tells you which rule to revisit. Do not skip ahead.

What’s the Output? — Integer Division

int a = 7, b = 2;
cout << a / b;

Answer: 3

Output — Integer Division

3

Both operands are int, so the result is int. The decimal part is discarded — not rounded. 7 / 2 = 3.5, but integer division gives 3. To get 3.5, at least one operand must be a floating-point type.

This is the most common “gotcha” early on. Reinforce: int / int = int, always.

What’s the Output? — Mixed Arithmetic

int a = 7;
double b = 2.0;
cout << a / b;

Answer: 3.5

Output — Mixed Arithmetic

3.5

When one operand is double, the int is implicitly promoted to double before the division is performed. 7.0 / 2.0 = 3.5. One floating-point operand changes the entire expression’s type.

Contrast with exercise 1. Emphasize: the presence of double forces promotion. This is implicit conversion in action.

What’s the Output? — static_cast Narrowing

double d = 9.99;
int i = static_cast<int>(d);
cout << i;

Answer: 9

Output — static_cast Narrowing

9

static_cast<int>() truncates the decimal part — it does not round. 9.99 becomes 9. The programmer explicitly requested this conversion, so the compiler performs it without a warning.

Emphasize: truncation, not rounding. 9.99 → 9, not 10. This trips students up. Also note: the explicit cast means the programmer is accepting responsibility for the data loss.

What’s the Output? — Character to Integer

char c = 'A';
cout << (int)c;

Answer: 65

Output — Character to Integer

65

char is a small integer type. 'A' is stored as its ASCII value, 65. The C-style cast (int) reveals that underlying number. Modern C++ prefers static_cast<int>(c) over the C-style cast, but both produce the same result here.

Many students are surprised that char is numeric. Reinforce: every character is stored as a number. The ASCII table defines which number maps to which character.

Patterns to Remember

  • int / int → int (decimal dropped)
  • int / double → double (int is promoted first)
  • static_cast<int>(double) → truncates toward zero, does not round
  • char stores an integer; casting reveals its numeric value

Summary

Integer division discards the decimal. Mixed arithmetic promotes the smaller type. Explicit casts truncate, not round. Every char has a numeric value. These four rules explain the output of most type-casting exercises you will encounter.

Appendix

Common Practices

  • Cast one operand to double before dividing when you need a decimal result.
  • Use static_cast<> instead of C-style casts in new code — it communicates intent and is searchable.
  • When converting double to int, decide whether truncation is correct or whether rounding is needed (round() from <cmath>).

Common Mistakes

  • Expecting 7 / 2 to produce 3.5 — it produces 3.
  • Assuming static_cast<int>(9.9) rounds to 10 — it truncates to 9.
  • Using C-style casts on pointer types where static_cast would be rejected — C-style casts bypass safety checks.

More Examples

// Rounding vs truncation
double x = 4.7;
cout << static_cast<int>(x);          // 4 (truncated)
cout << static_cast<int>(round(x));   // 5 (rounded)

// char arithmetic
char c = 'a';
cout << static_cast<int>(c + 1);      // 98 (ASCII for 'b')

// Division with explicit cast
int p = 10, q = 3;
cout << static_cast<double>(p) / q;   // 3.3333...

  2. What does static_cast<int>(-3.9) produce? Truncation rounds toward zero — does that mean it produces -3 or -4?
  3. Write one line that prints the ASCII value of the character 'z'.
  4. Predict: int a = 3; double b = 1.5; cout << a * b;

Signed vs Unsigned Variables

Signed vs Unsigned Variables

How does a variable store negative numbers, and when should you choose signed over unsigned? The difference is not just range — it affects overflow behavior, comparison logic, and compatibility with standard library types. Most of these bugs appear silently.

Total slides: 9

Signed: How Negatives Are Stored

A signed integer uses the most significant bit (MSB) as a sign bit. In two’s complement — the format used on all modern hardware — a 1 in the MSB means negative, 0 means non-negative. A 32-bit signed int uses 31 bits for magnitude and 1 bit for the sign.

[Graphic suggestion: 32-bit layout showing bit 31 as the sign bit and bits 0–30 as the value bits.]

Unsigned: All Bits for Magnitude

An unsigned variable has no sign bit — all bits represent magnitude. A 32-bit unsigned int uses all 32 bits, doubling the positive range compared to a signed int.

Type Minimum Maximum
int (signed) −2,147,483,648 2,147,483,647
unsigned int 0 4,294,967,295
short (signed) −32,768 32,767
unsigned short 0 65,535

When to Use Unsigned

Use unsigned when:

  • The value can never be negative (sizes, counts, array indices)
  • Performing bit manipulation (flags, masks, bitwise operations)
  • Matching the type returned by .size() on containers — it returns size_t, which is unsigned
for (size_t i = 0; i < vec.size(); ++i) {
    // size_t is unsigned; matches vec.size() type exactly
}

When to Use Signed

Use signed for all general arithmetic unless there is a specific reason for unsigned. Signed is the default for int, short, and long. If a variable can ever be negative — even theoretically — use signed.

int temperature = -15;    // can be negative
int balance     = -200;   // account overdraft
int velocity    = -50;    // moving in reverse

Mixing Signed and Unsigned: The Trap

When a signed and unsigned value are compared or combined in an expression, the signed value is implicitly converted to unsigned. A negative signed value becomes a very large positive unsigned value.

Mixing Signed and Unsigned — Code

int   s =  -5;
unsigned int u = 10;

if (s < u) {
    cout << "s < u\n";   // Looks like it should print...
} else {
    cout << "NOT s < u\n";  // ...but this prints instead
}

-5 is converted to unsigned int: it becomes 4,294,967,291. The comparison 4294967291 < 10 is false. Enable -Wsign-compare to catch this at compile time.

This is a real source of security bugs in C++ — bounds checks that silently invert.

Overflow Behavior Differs

int s = INT_MAX;
s++;              // Undefined behavior — do not rely on this

unsigned int u = UINT_MAX;
u++;              // Well-defined: wraps to 0

Signed overflow is undefined. Unsigned overflow is defined modular arithmetic. This is one of the few places where the two types behave fundamentally differently at the language level.

Summary

Signed is the safe default for general arithmetic; unsigned is appropriate for sizes, indices, and bit operations. Mixing the two in comparisons or arithmetic silently converts signed to unsigned — a common source of logic bugs. Enable sign-comparison warnings and address them rather than casting them away.

Appendix

Common Practices

  • Default to int for general arithmetic; switch to unsigned only when the domain guarantees non-negative values.
  • Use size_t for container indices and loop counters over containers — it matches .size() and avoids sign-comparison warnings.
  • Enable -Wsign-compare (or equivalent) in your compiler settings and treat the warnings as errors.

Common Mistakes

  • Comparing int to the result of .size() — .size() returns size_t (unsigned), and a negative int silently becomes a huge positive value.
  • Decrementing an unsigned counter past zero in a loop — the loop never terminates because the underflow wraps to a large positive value.
  • Assuming signed and unsigned overflow behave the same way — they follow different rules with different consequences.

More Examples

// Infinite loop due to unsigned underflow
for (unsigned int i = 10; i >= 0; i--) {
    cout << i << "\n";  // When i is 0, i-- wraps to UINT_MAX
}

// Safe: use signed int instead
for (int i = 10; i >= 0; i--) {
    cout << i << "\n";  // Terminates correctly
}

Fixed Width Integers and size_t

Fixed Width Integers and size_t

If int is 4 bytes on one system and 2 bytes on another, how do you write code that behaves identically everywhere? C++ provides fixed-width integer types that guarantee an exact size regardless of platform. This section also introduces size_t, the standard type for anything that measures a size in memory.

Total slides: 9

The Problem: Platform-Dependent Sizes

The standard types int, short, and long have minimum guaranteed sizes but no maximum. In practice, int is 4 bytes on most modern systems, but embedded and legacy platforms differ. Code that assumes a fixed size silently breaks when compiled elsewhere.

Fixed-Width Types from <cstdint>

<cstdint> provides types with guaranteed sizes:

Type Signed Unsigned
8-bit int8_t uint8_t
16-bit int16_t uint16_t
32-bit int32_t uint32_t
64-bit int64_t uint64_t

If the platform cannot provide that exact size, the type does not exist — you get a compile error rather than silent data corruption.

Using Fixed-Width Types

#include <cstdint>
#include <iostream>
using namespace std;

int main() {
    int32_t  count   = 2000000000;   // guaranteed 32-bit signed
    uint8_t  flags   = 0b10101010;   // guaranteed 8-bit unsigned
    int64_t  balance = -900000000000LL;

    cout << count   << "\n";
    cout << (int)flags << "\n";
    cout << balance << "\n";
}

When to Use Fixed-Width Types

Use fixed-width types when:

  • The exact bit width matters: network protocols, binary file formats, hardware registers
  • Code must be portable across platforms with different word sizes
  • You need bit manipulation with a known field width

Do not use them for general arithmetic where the exact size does not matter — the standard int and long are fine there.

What Is size_t?

size_t is the unsigned integer type returned by sizeof() and by .size() on any container. It is guaranteed to be large enough to represent the size of any object in memory on the current platform — typically 4 bytes on 32-bit systems and 8 bytes on 64-bit systems.

cout << sizeof(int)   << "\n";  // returns size_t
cout << sizeof(double) << "\n";

vector<int> v = {1, 2, 3};
cout << v.size() << "\n";       // also returns size_t

Using size_t for Loop Indices

vector<string> names = {"Alice", "Bob", "Carol"};

for (size_t i = 0; i < names.size(); ++i) {
    cout << names[i] << "\n";
}

Comparing int to size_t triggers a sign-comparison warning. Using size_t for the loop counter eliminates the warning and is semantically correct — an index into a container cannot be negative.

Fast and Least Variants (Brief)

<cstdint> also defines:

  • int_fast32_t — at least 32 bits; the fastest available type of that width on the platform
  • int_least8_t — the smallest type with at least 8 bits

These exist for performance-sensitive code. At the introductory level, use the exact-width types (int32_t) unless you have a specific reason to choose otherwise.

Summary

Fixed-width integer types guarantee an exact size on any platform. Use them when bit width matters — protocols, file formats, hardware. Use size_t for sizes, indices, and anything that measures memory or container length. For everything else, int is fine.

Appendix

Common Practices

  • Include <cstdint> whenever you use fixed-width types.
  • Use size_t for loop indices over containers to avoid sign-comparison warnings.
  • Choose int32_t over int when you are writing cross-platform code that assumes a specific range.
  • Never hardcode sizes like 4 for int — use sizeof() or the appropriate size_t instead.

Common Mistakes

  • Using int8_t for small numbers and then printing it directly — it may print as a character since int8_t is often signed char.
  • Assuming size_t is always 64 bits — it is platform-dependent (32 bits on 32-bit systems).
  • Mixing int and size_t in comparisons without a cast — the signed int is silently converted to unsigned.

More Examples

#include <cstdint>
#include <iostream>
using namespace std;

int main() {
    uint32_t pixel = 0xFF8800FF;    // RGBA color packed in 32 bits
    uint8_t  r = (pixel >> 24) & 0xFF;
    uint8_t  g = (pixel >> 16) & 0xFF;
    uint8_t  b = (pixel >> 8)  & 0xFF;

    cout << "R: " << (int)r << "\n";  // 255
    cout << "G: " << (int)g << "\n";  // 136
    cout << "B: " << (int)b << "\n";  // 0
}

Packing and unpacking data into exact-width fields is one of the primary use cases for fixed-width types.

Determining Variable Size

Determining Variable Size

How can you programmatically find out how much memory a variable or type occupies? C++ provides the sizeof operator for exactly this purpose. It evaluates at compile time and works on any type, variable, or array — and it is the only portable way to measure memory.

Total slides: 8

The sizeof Operator

sizeof is a compile-time operator that returns the size of a type or variable in bytes. It does not execute code or evaluate the operand’s value — it determines size from type information alone. The return type is size_t.

sizeof(type)      // size of a type
sizeof variable   // size of a declared variable
sizeof(expr)      // size of the type an expression would produce

Typical Platform Sizes

Sizes are not mandated by the standard and vary by platform and compiler. These are typical on a 64-bit system with a modern C++ compiler:

Type Typical Size
char 1 byte
short 2 bytes
int 4 bytes
long 8 bytes
float 4 bytes
double 8 bytes
bool 1 byte
pointer 8 bytes

Always use sizeof rather than assuming these values.

Why Sizes Vary

  • Architecture: 32-bit systems use 4-byte pointers; 64-bit systems use 8-byte pointers.
  • Compiler choices: long is 4 bytes on Windows (MSVC) and 8 bytes on Linux (GCC/Clang).
  • Alignment: the compiler inserts padding between struct members to align data on word boundaries for performance.

sizeof with Arrays

int arr[10];
cout << sizeof(arr);              // 40  (10 * 4 bytes)
cout << sizeof(arr[0]);           // 4   (size of one int)
cout << sizeof(arr) / sizeof(arr[0]);  // 10  (element count)

The idiom sizeof(arr) / sizeof(arr[0]) gives the number of elements in a statically allocated array. It does not work for pointer parameters — the array decays to a pointer and you get pointer size, not array size.

sizeof with Structs: Padding

struct Point {
    char x;   // 1 byte
    // 3 bytes padding inserted by compiler
    int  y;   // 4 bytes
};

cout << sizeof(Point);  // 8 (not 5)

The compiler aligns int members on 4-byte boundaries. Padding is invisible in code but real in memory. Reordering members (largest first) can reduce total struct size.

[Graphic suggestion: Memory layout diagram showing 1-byte x, 3-byte padding gap, 4-byte y.]

Printing All Type Sizes

#include <iostream>
using namespace std;

int main() {
    cout << "char:   " << sizeof(char)   << " byte(s)\n";
    cout << "short:  " << sizeof(short)  << " byte(s)\n";
    cout << "int:    " << sizeof(int)    << " byte(s)\n";
    cout << "long:   " << sizeof(long)   << " byte(s)\n";
    cout << "float:  " << sizeof(float)  << " byte(s)\n";
    cout << "double: " << sizeof(double) << " byte(s)\n";
    cout << "bool:   " << sizeof(bool)   << " byte(s)\n";
}

Run this on your machine. The output is specific to your compiler and platform.

Summary

sizeof is the portable, compile-time way to determine memory size. Use it instead of hardcoding byte counts. Remember that struct size includes alignment padding — the total is not always the sum of member sizes.

Appendix

Common Practices

  • Always use sizeof rather than hardcoding byte sizes — platforms differ.
  • Use sizeof(arr) / sizeof(arr[0]) to count elements in a statically allocated array.
  • When designing structs for memory efficiency, order members from largest to smallest to minimize padding.
  • Store sizeof results in a size_t variable if reused multiple times in a calculation.

Common Mistakes

  • Passing an array to a function and calling sizeof inside the function — the array has decayed to a pointer; you get the pointer size, not the array size.
  • Assuming sizeof(struct) equals the sum of its member sizes — padding almost always makes it larger.
  • Confusing sizeof(vector<int>) with the number of elements — it returns the size of the vector object itself, not its contents.

More Examples

// Pointer size is independent of pointed-to type
char* p1 = nullptr;
int*  p2 = nullptr;
cout << sizeof(p1) << " " << sizeof(p2);  // Same size (8 on 64-bit)

// Array of structs
struct Point { int x; int y; };
Point pts[100];
cout << sizeof(pts);                     // sizeof(Point) * 100

// sizeof does not evaluate its argument
int i = 0;
cout << sizeof(i++);  // i is still 0 after this
