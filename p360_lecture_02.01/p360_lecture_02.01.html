How the Compiler Works

What happens when you press “build”?

  • The compiler transforms your source code through several distinct stages into an executable program.
  • Understanding this process helps you read error messages, locate bugs faster, and write code that compiles cleanly.

Total slides: 16 Frame this as practical knowledge, not theory. Every error message a student sees — “undeclared identifier,” “undefined reference,” “no matching function” — comes from one of the three stages. Knowing which stage produced the error tells you where to look.

The three stages

  • Stage 1 — Preprocessing: resolves #include, #define, and macro directives before the compiler ever sees your code.
  • Stage 2 — Compilation: type-checks the source and translates it to object code.
  • Stage 3 — Linking: combines object files and resolves references to library functions.

[Graphic suggestion: horizontal pipeline diagram — source → preprocessor → expanded source → compiler → object file → linker (+ library) → executable.]

Stage 1: Preprocessing

  • Runs before compilation; processes all directives that start with #.
  • Expands #include by inserting the full contents of the header file verbatim.
  • Removes comments; output is still C++ source, not object code.

Students sometimes think the preprocessor is part of the compiler — it is not. It is a separate text-processing step that knows nothing about C++ syntax. The preprocessor cannot catch type errors or undefined variables. It simply performs textual substitution.

Preprocessing directives

  • #include <file> — inserts a header file’s contents verbatim at that point in the source.
  • #define NAME value — textual substitution; every occurrence of NAME is replaced before compilation.
  • #pragma once — prevents a header from being inserted more than once per translation unit.

Without pragma once (or header guards), including the same header twice from different files causes redeclaration errors at compile time. Students will see both pragma once and the classic #ifndef/#define/#endif pattern in code they read.

Stage 2: Compilation

  • Takes preprocessed source and checks syntax and type correctness.
  • Translates valid C++ into processor-specific object code.
  • One .cpp file → one .o object file.

This is the stage students interact with most — syntax errors and type errors both appear here. The compiler processes one .cpp file at a time; it has no knowledge of other .cpp files. Definitions are resolved at link time.

What the compiler checks: syntax errors

  • Syntax — does your code follow C++ grammar rules? Missing braces, semicolons, or mismatched delimiters are caught here.
int main()
    cout << "hello";   // ERROR: missing { before function body
    return 0;
}
error: expected '{' before 'cout'

What the compiler checks: type errors

  • Types — do variable types match in assignments and operations? C++ is statically typed — mismatches are caught at compile time, not at runtime.
int age = 25;
age = "twenty-five";   // ERROR: can't store a string in an int
error: invalid conversion from 'const char*' to 'int'

Static typing — a C++ strength

Static typing is a key C++ strength: an entire class of bugs that would appear only at runtime in Python or JavaScript are caught at compile time in C++.

What the compiler checks: scope and declaration errors

  • Scope and declarations — are all variables and functions declared before use? Do names conflict within the same scope?
int main()
{
    count = 10;   // ERROR: 'count' was never declared
    return 0;
}
error: 'count' was not declared in this scope

Static typing is a key C++ strength: an entire class of bugs that would appear only at runtime in Python or JavaScript are caught at compile time in C++.

Stage 3: Linking

  • Combines multiple .o object files into a single executable.
  • Resolves references to library functions (e.g., cout from <iostream>).
  • Reports “undefined reference” when a called function has no definition anywhere.

The most common linker error beginners see: “undefined reference to ‘main’” (forgot to write main) or “undefined reference to ‘sqrt’” (forgot to link the math library). These are not compile errors — the code compiled fine; the linker just cannot find the implementation.

What the compiler checks

  • Syntax errors — missing braces, semicolons, or mismatched delimiters.
  • Type errors — mismatched types in assignments or operations.
  • Scope and declaration errors — using identifiers that are undeclared or redeclared.

Translation units

  • Each .cpp file is compiled independently — it is called a translation unit.
  • The compiler sees one .cpp file at a time and has no knowledge of other .cpp files until linking.
  • Declarations go in .h files so the preprocessor can copy them into every .cpp that needs them.
// math.h — declaration only (shared via #include)
int add(int a, int b);

// math.cpp — definition in its own translation unit
int add(int a, int b) { return a + b; }

// main.cpp — sees the declaration via #include, not the definition
#include "math.h"
int main() { return add(2, 3); }

This explains why a function defined in another .cpp causes an “undeclared” error: the compiler only sees the .cpp it is currently compiling. The declaration in the header gives the compiler what it needs for type checking; the linker handles connecting the call to the definition.

Translation units — for your general knowledge

  • The concept of translation units is part of how C++ works under the hood.
  • You do not need to worry about this for this course.
  • You may come across the term in your readings or online — now you know what it means.

This is a gentle landing after a fairly dense slide. Students at this level do not need to write multi-file programs yet. The goal of the translation unit slide was to explain why declarations go in headers; this slide releases the pressure and lets students know the concept is background knowledge, not something they will be tested on.

Reading error messages by stage

  • Errors that mention line numbers in your source file → compile-time errors (Stage 2).
  • Errors that say “undefined reference” or “symbol not found” → linker errors (Stage 3).
  • Errors that say “no such file” or “invalid preprocessing directive” → preprocessor errors (Stage 1).

Teaching students to categorize errors by stage is a practical debugging skill. A linker error means the code compiled correctly — the problem is a missing definition, not a syntax mistake.

Summary

  • The compiler transforms source code through preprocessing, compilation, and linking.
  • Understanding these stages helps you interpret error messages and locate bugs faster.
  • Compile errors → Stage 2. Linker errors → Stage 3. File-not-found → Stage 1.

The Symbol Table

What problems does the compiler face tracking every identifier?

  • The symbol table is the compiler’s internal database that records every identifier — variable, function, or type — along with its name, type, scope, and location.
  • Every type mismatch, undeclared variable, and redeclaration error traces back to this table.

Total slides: 7 Frame this as the mechanism behind error messages students already encounter. “Undeclared identifier” means: the compiler searched the symbol table and found no entry. “Redeclaration” means: the compiler found an entry already. Once students understand the table, the error messages make sense.

What goes in the symbol table?

Two-panel diagram: source code on the left, symbol table entries on the right with arrows connecting each identifier to its row. Name, type, scope, and address are the four fields. This image replaces the text description — the visual makes the mapping between code and table entries concrete.

How the compiler uses it

  • Type checking — when you write int x = "hello", the compiler looks up x in the table, sees int, and rejects the string.
  • Scope resolution — when you reference count, the compiler searches from the innermost scope outward.
  • Duplicate detection — if you declare the same name twice in the same scope, the compiler finds the first entry and rejects the second.
  • Code generation — addresses stored in the table tell the compiler where to read and write each variable.

Undeclared identifier error

int main()
{
    count = 10;   // error: 'count' was not declared in this scope
    return 0;
}

The compiler searched the symbol table — starting in main()‘s scope, then global scope — and found no entry for ’count’. The fix is always the same: declare the variable before you use it. The error message tells students exactly what went wrong and where to look.

Redeclaration error

int main()
{
    int x = 5;
    int x = 10;   // error: redeclaration of 'int x'
    return 0;
}

The compiler found ‘x’ already in the symbol table for this scope and cannot add a second entry with the same name. Each identifier can have only one entry per scope. The solution: either rename the second variable or remove the redundant declaration.

Type mismatch error

int main()
{
    int age = 25;
    age = "twenty-five";  // error: invalid conversion from 'const char*' to 'int'
    return 0;
}

The symbol table records ‘age’ as type int. When the compiler evaluates the assignment, it checks the incoming value’s type (const char*) against the table entry (int) and rejects it. This is static type checking at work.

Data Type Conversion and Casting

Type coercion — the compiler converts automatically

  • The compiler automatically promotes the lower-ranking type to match the higher-ranking type.
  • Promotion — moving up the ranking. No data is lost.
  • Demotion — moving down the ranking. Data may be lost.

Data type ranking

When two operands of different types appear in an expression, the lower-ranking type is promoted to the higher before the operation executes.

Rank Type
1 (highest) long double
2 double
3 float
4 unsigned long long
5 long long
6 unsigned long
7 long
8 unsigned int
9 (lowest) int

Data Type Ranking table (Table 3-7 from the original lecture). The result of any mixed-type expression takes the type of the higher-ranking operand. Note abbreviated names: long = long int, long long = long long int, etc.

Three rules of coercion

  • Rule 1: char, short, and unsigned short are automatically promoted to int in any expression.
  • Rule 2: When two operands of different types appear in an operation, the lower-ranking type is promoted to the higher-ranking type.
  • Rule 3: The final value of an expression is converted to the type of the variable it is being assigned to.

Rule 3 is where demotion happens. If you compute a double result and assign it to an int, the decimal part is silently dropped — no error, no warning by default. This is the most common source of unexpected results in beginner programs.

Promotion — moving up

  • Promotion happens when a smaller type is used where a larger type is expected.
  • No data is lost. C++ performs this automatically.
int x = 5;
double y = x;        // int promoted to double; y = 5.0

char c = 'A';
int val = c;         // char promoted to int; val = 65

int a = 10;
double r = a / 2.0;  // a promoted before division; r = 5.0

Demotion — moving down

  • Demotion (narrowing) happens when a larger type is assigned to a smaller type.
  • Data loss is likely. The compiler may warn but will not always stop you.
double x = 3.7;
int y = x;             // Warning: y = 3 (decimal discarded — not rounded)

long bigNum = 100000;
short s = bigNum;      // Warning: may overflow

int a = 7, b = 2;
int result = a / b;    // result = 3, not 3.5

The problem: integer division gives the wrong answer

// 3-7x.cpp — casting & expressions
#include <iostream>
using namespace std;

int main()
{
    int books, months;
    double booksPerMonth;

    cout << "How many books: ";
    cin >> books;
    cout << "How many months to read them? ";
    cin >> months;

    booksPerMonth = books / months;   // integer division — decimal dropped!
    cout << "That is " << booksPerMonth << " books per month.\n";
    return 0;
}

Input: books = 10, months = 6. Expected: 1.66667. Actual: 1. Why? books and months are both int. Integer division runs first and produces 1. That 1 is then assigned to a double — so booksPerMonth becomes 1.0, not 1.66667. The type of the variable receiving the result does not affect the division. The fix is on the next slide.

The fix: cast one operand to double

// Without cast — integer division, wrong answer
booksPerMonth = books / months;
// Input: 10 books, 6 months → output: 1

// With static_cast — floating point division, correct answer
booksPerMonth = static_cast<double>(books) / months;
// Input: 10 books, 6 months → output: 1.66667

Casting one operand to double forces the entire division to use floating point. Only one operand needs to be cast — the other is promoted automatically by coercion rule 2. The pattern static_cast(intVariable) is the idiomatic C++ way to get a decimal result from integer operands.

Type casting — you control the conversion

  • Type casting is explicitly telling the compiler to convert from one type to another.
  • It is better to be explicit with your code than to rely on the compiler doing something automatically.
  • Three syntactic forms exist in C++. Only one is recommended.
static_cast<to_type>(value)    // C++ static cast — preferred
(to_type)value                 // C style — prefix notation — avoid
to_type(value)                 // pre-standard C++ functional — avoid

static_cast — the C++ way

  • static_cast<TargetType>(value) is the preferred form for numeric conversion.
  • Makes your intent explicit and is easy to find in a codebase.
  • Use it any time you need an intentional narrowing or explicit promotion.
double x = 3.7;
int y = static_cast<int>(x);              // explicit truncation; y = 3

int dividend = 7, divisor = 2;
double r = static_cast<double>(dividend) / divisor;  // r = 3.5

char letter = 'Z';
int ascii = static_cast<int>(letter);     // ascii = 90

C-style casts — avoid

  • Syntax: (TargetType)value — carried over from C.
  • Works, but is not recommended in modern C++.
  • Hard to search for in a large codebase; hides intent; performs dangerous conversions without warning.
double x = 3.7;
int y = (int)x;    // C-style cast — works, but avoid in new C++ code

Students will encounter C-style casts in textbooks and older code. They should be able to read them but should write static_cast in their own programs. The main problem with C-style casts: they are difficult to distinguish visually from other constructs and they bypass safety checks that static_cast enforces.

Functional notation — avoid

  • Pre-standard C++ syntax: to_type(value) — looks like a constructor call.
  • Equivalent to a C-style cast for primitive types. Avoid in new code.
int(3.7)       // converts double to int; result: 3
double(7)      // converts int to double; result: 7.0
char(65)       // converts int to char; result: 'A'

When to cast and when not to

  • Use a cast to express intent — when you know truncation or conversion is correct.
  • Do not cast just to silence a compiler warning without fixing the underlying mismatch.
  • A cast that hides a narrowing bug is worse than the warning it suppressed.

This is the discipline point. Students sometimes discover they can make warnings disappear with a cast and treat that as solving the problem. It is not — it is hiding the symptom. The right response to a narrowing warning is to understand why the narrowing occurs and whether it is safe. If it is safe, cast with intent. If it is not safe, fix the types.

Implicit vs. explicit at a glance

Implicit (Coercion) Explicit (Cast)
Who initiates Compiler Programmer
Syntax required None static_cast<>
Promotion Automatic, safe Rarely needed
Narrowing Automatic, may lose data Intentional, visible
Risk Hidden data loss Visible — you own the result

Summary

  • Implicit coercion promotes the lower-ranking type automatically; narrowing silently loses data.
  • Use static_cast<> to make conversions explicit and intentional.
  • Never cast just to silence a warning — the warning usually means there is a real issue to address.

Signed vs Unsigned Variables

How does a variable store negative numbers?

  • The difference between signed and unsigned is not just range — it affects overflow behavior, comparison logic, and compatibility with standard library types.
  • Most bugs in this area appear silently — no compile error, wrong answer.

Total slides: 10 Connect to the overflow topic: students now know unsigned overflow wraps and signed overflow is undefined. This section explains why — it is in the bit representation. The mixing-types trap is the most practically important part.

Signed: the MSB is the sign bit

  • A signed integer uses the most significant bit (MSB) as a sign bit.
  • In two’s complement — the format used on all modern hardware — a 1 in the MSB means negative, 0 means non-negative.
  • A 32-bit signed int uses 31 bits for magnitude and 1 bit for sign.

[Graphic suggestion: 32-bit layout diagram showing bit 31 labeled “sign bit” and bits 0–30 labeled “magnitude.”] Two’s complement makes addition work without special-casing negative numbers — the hardware uses the same addition circuit for both. Students do not need to understand two’s complement arithmetic in detail, but they should know the MSB encodes sign.

Unsigned: all bits for magnitude

  • An unsigned variable has no sign bit — all bits represent magnitude.
  • A 32-bit unsigned int uses all 32 bits, doubling the positive range.
Type Minimum Maximum
int (signed) −2,147,483,648 +2,147,483,647
unsigned int 0 +4,294,967,295
short (signed) −32,768 +32,767
unsigned short 0 +65,535

When to use unsigned

  • Use unsigned when the value can never be negative: sizes, counts, array indexes.
  • Use unsigned for bit manipulation: flags, masks, bitwise operations.
  • Use size_t (unsigned) when matching the return type of .size() on containers.
for (size_t i = 0; i < vec.size(); ++i) {
    // size_t matches vec.size() type — no sign-comparison warning
}

When to use signed

  • Use signed for all general arithmetic unless there is a specific reason for unsigned.
  • Signed is the default for int, short, and long.
  • If a variable can ever be negative — even theoretically — use signed.
int temperature = -15;   // can be negative
int balance     = -200;  // account overdraft
int velocity    = -50;   // moving in reverse

Mixing signed and unsigned — the trap

  • When a signed and unsigned value appear in the same expression, the signed value is implicitly converted to unsigned.
  • A negative signed value becomes a very large positive unsigned value.
int          s =  -5;
unsigned int u =  10;

if (s < u) {
    cout << "s < u\n";      // looks like it should print...
} else {
    cout << "NOT s < u\n";  // ...but this prints instead
}

-5 is converted to unsigned int: it becomes 4,294,967,291. The comparison 4294967291 < 10 is false. This is a real source of security bugs in C++ — bounds checks that silently invert. Enable -Wsign-compare in g++ to catch this at compile time. The fix: use a consistent type throughout, or cast explicitly.

Overflow behavior differs

int s = INT_MAX;
s++;              // undefined behavior — signed overflow

unsigned int u = UINT_MAX;
u++;              // well-defined: wraps to 0

Revisit the overflow topic here in the signed/unsigned context. The behavior difference is fundamental: it comes from the language standard, not from hardware. Signed overflow being undefined is intentional — it lets the compiler assume it never happens and optimize aggressively.

Enabling sign-comparison warnings

  • The compiler can warn you when signed and unsigned values are compared.
  • Add -Wsign-compare to your build flags (included in -Wall).
g++ -Wall -Wextra myprogram.cpp
warning: comparison of integer expressions of different signedness:
'int' and 'unsigned int' [-Wsign-compare]

-Wall enables most common warnings including -Wsign-compare. Students should compile with -Wall from the start — it catches a wide class of common mistakes. The goal is zero warnings, not just zero errors.

Summary

  • Signed is the safe default for general arithmetic; unsigned is appropriate for sizes, indexes, and bit operations.
  • Mixing the two in comparisons or arithmetic silently converts signed to unsigned — a common source of logic bugs.
  • Enable -Wall and address sign-comparison warnings rather than casting them away.

Overflow and Underflow

What happens when a value exceeds its type’s range?

  • Every numeric type has a fixed range. Assign a value outside that range and the result is not what you expect.
  • Overflow — value exceeds the maximum.
  • Underflow — value drops below the minimum (or, for floating point, below the smallest representable positive value).
  • The compiler does not always warn you, and the behavior differs between signed and unsigned types.

Total slides: 14 Frame this as a consequence of the type system. Students now understand that types have fixed sizes. This section shows what happens at the boundaries. The signed/unsigned distinction is critical here — one behavior is undefined, the other is well-defined.

Signed overflow — undefined behavior

  • For signed integers, overflow is undefined behavior in C++.
  • The compiler is free to assume it never happens — which can produce counter-intuitive optimizations.
  • In practice on most platforms it wraps around, but you cannot rely on this.
#include <climits>

int x = INT_MAX;   // 2,147,483,647
x++;               // undefined behavior — do NOT rely on any result

Undefined behavior means the C++ standard places no requirement on what happens. The compiler may assume signed overflow never occurs and optimize accordingly — which can eliminate safety checks, infinite loops, and other constructs that depend on overflow behavior. This is not theoretical: real compilers (GCC, Clang) aggressively exploit undefined behavior for optimization.

Unsigned overflow — well-defined wrapping

  • For unsigned integers, overflow is well-defined: the value wraps modulo 2ⁿ (where n is the bit width).
  • An unsigned value that exceeds the maximum wraps back to zero.
  • An unsigned value that goes below zero wraps to the maximum.
#include <climits>

unsigned int u = UINT_MAX;  // 4,294,967,295
u++;                        // wraps to 0 — well-defined

unsigned int v = 0;
v--;                        // wraps to 4,294,967,295 — well-defined

This is one of the few cases where signed and unsigned types have genuinely different semantics at the language level. Unsigned overflow being well-defined makes it usable in hash functions, checksums, and modular arithmetic. Signed overflow being undefined is a source of real bugs in security-critical code.

numeric_limits — querying type boundaries

  • std::numeric_limits<T> from <limits> provides the exact min and max for any numeric type.
  • More reliable than INT_MAX macros from <climits> — works with any type including user-defined ones.
#include <limits>
#include <iostream>
using namespace std;

int main()
{
    cout << "int max: "    << numeric_limits<int>::max()    << "\n";
    cout << "int min: "    << numeric_limits<int>::min()    << "\n";
    cout << "uint max: "   << numeric_limits<unsigned int>::max() << "\n";
    cout << "double max: " << numeric_limits<double>::max() << "\n";
}

Worked example: short overflow

#include <iostream>
using namespace std;

int main()
{
    short s = 32767;   // SHRT_MAX
    cout << s << "\n"; // 32767
    s++;
    cout << s << "\n"; // -32768 on most platforms (undefined for signed)
}

On most platforms with most compilers, signed overflow wraps two’s complement — 32767 + 1 = -32768. But this is implementation-defined behavior, not guaranteed. The example illustrates why you should never write code that depends on this. Use unsigned types and explicit modular arithmetic when you need wrap-around behavior.

Worked example: unsigned underflow

#include <iostream>
using namespace std;

int main()
{
    unsigned int u = 0;
    cout << u << "\n";  // 0
    u--;
    cout << u << "\n";  // 4,294,967,295 — wraps (well-defined)
}

Worked example: INT_MAX + 1

#include <iostream>
#include <climits>
using namespace std;

int main()
{
    int x = INT_MAX;
    cout << "x     = " << x     << "\n";  // 2147483647
    cout << "x + 1 = " << x + 1 << "\n"; // undefined behavior

    unsigned int u = INT_MAX;
    u++;
    cout << "u     = " << u << "\n";      // 2147483648 — well-defined
}

Floating point underflow

  • Floating point underflow occurs when a positive value becomes too small to represent — the result rounds to zero.
  • Unlike integer overflow, floating point underflow is generally not catastrophic — it produces zero rather than garbage.
#include <limits>
#include <iostream>
using namespace std;

int main()
{
    double tiny = numeric_limits<double>::min();  // ~2.2e-308
    tiny /= 2.0;
    cout << tiny << "\n";  // may print 0 or a denormalized value
}

Floating point has two special cases: overflow (result becomes infinity) and underflow (result becomes zero or a denormalized number). Neither crashes the program, but both produce incorrect results. This is different from integer overflow, which is undefined behavior for signed types.

Detecting overflow before it happens

  • The safest approach: check whether the operation would overflow before performing it.
  • For addition: if a > INT_MAX - b, then a + b would overflow.
#include <climits>
#include <iostream>
using namespace std;

bool safeAdd(int a, int b, int &result)
{
    if (b > 0 && a > INT_MAX - b) return false;  // would overflow
    if (b < 0 && a < INT_MIN - b) return false;  // would underflow
    result = a + b;
    return true;
}

This is the defensive programming pattern for integer arithmetic. Checking after the addition is too late — the overflow has already produced undefined behavior. The check must happen before the operation using rearranged algebra that avoids overflow in the check itself.

What to do in practice

  • For general arithmetic: use int or long long and validate inputs to ensure they stay in range.
  • For counters and indexes that can never be negative: use unsigned and know that decrement below zero wraps.
  • When exact range matters: use numeric_limits<T>::max() rather than hardcoded values.
  • When overflow is a real risk: detect before the operation, not after.

The compiler will not save you

int x = 2000000000;
int y = 2000000000;
int z = x + y;     // undefined behavior — no compile error, no warning
cout << z << "\n"; // may print -294967296 or anything else

Compile this and run it to show students what undefined behavior looks like in practice. The output depends on the compiler, optimization level, and platform. Some compilers emit a warning with -ftrapv (which traps on signed overflow). The lesson: overflow does not announce itself — you have to think about whether it can happen.

Summary

  • Signed integer overflow is undefined behavior — never rely on it.
  • Unsigned integer overflow wraps modulo 2ⁿ — well-defined and sometimes intentional.
  • Use numeric_limits<T> to query type boundaries portably.
  • Detect potential overflow before the operation, not after.

Fixed-Width Integers and size_t

What if int is a different size on a different machine?

  • The standard types int, short, and long have minimum guaranteed sizes but no maximum.
  • Code that assumes a fixed size silently breaks when compiled on a different platform.
  • C++ provides fixed-width integer types that guarantee an exact size everywhere.

Total slides: 9 This is a practical portability topic. Students writing code for embedded systems, network protocols, or binary file formats will encounter this immediately. The key lesson: never hardcode byte counts — use fixed-width types when the width matters and sizeof() when you need to measure.

The problem: platform-dependent sizes

  • int is 4 bytes on most modern desktops, but 2 bytes on some embedded platforms.
  • long is 8 bytes on 64-bit Linux but 4 bytes on 64-bit Windows.
  • Writing int x and assuming it is always 32 bits is a portability bug.

Real-world scenario: a network protocol field is exactly 32 bits. Using int on a platform where int is 16 bits corrupts the data silently. The solution is to name the width explicitly using the fixed-width types from .

Fixed-width types from <cstdint>

Signed Unsigned Width
int8_t uint8_t 8 bits
int16_t uint16_t 16 bits
int32_t uint32_t 32 bits
int64_t uint64_t 64 bits

If the platform cannot provide that exact size, the type does not exist — you get a compile error rather than silent data corruption.

Using fixed-width types

#include <cstdint>
#include <iostream>
using namespace std;

int main()
{
    int32_t  count   = 2000000000;   // guaranteed 32-bit signed
    uint8_t  flags   = 0b10101010;   // guaranteed 8-bit unsigned
    int64_t  balance = -900000000000LL;

    cout << count        << "\n";
    cout << (int)flags   << "\n";
    cout << balance      << "\n";
}

When to use fixed-width types

  • Use fixed-width types when:
    • The exact bit width matters: network protocols, binary file formats, hardware registers.
    • Code must be portable across platforms with different word sizes.
    • You need bit manipulation with a known field width.
  • For general arithmetic where exact size does not matter, plain int and long are fine.

What is size_t?

  • size_t is the unsigned integer type returned by sizeof() and by .size() on any container.
  • Guaranteed to be large enough to represent the size of any object in memory on the current platform.
  • Typically 4 bytes on 32-bit systems and 8 bytes on 64-bit systems.
cout << sizeof(int)    << "\n";   // returns size_t
cout << sizeof(double) << "\n";

vector<int> v = {1, 2, 3};
cout << v.size() << "\n";         // also returns size_t

Using size_t for loop indexes

  • Comparing int to size_t triggers a sign-comparison warning.
  • Using size_t for the loop counter eliminates the warning and is semantically correct — an index into a container cannot be negative.
vector<string> names = {"Alice", "Bob", "Carol"};

for (size_t i = 0; i < names.size(); ++i) {
    cout << names[i] << "\n";
}

Fast and least variants

  • <cstdint> also defines performance-oriented variants:
    • int_fast32_t — at least 32 bits; the fastest available type of that width on the platform.
    • int_least8_t — the smallest type with at least 8 bits.
  • These exist for performance-sensitive code.
  • At the introductory level, use the exact-width types (int32_t) unless you have a specific reason to choose otherwise.

Summary

  • Fixed-width integer types guarantee an exact size on any platform.
  • Use them when bit width matters — protocols, file formats, hardware.
  • Use size_t for sizes, indexes, and anything that measures memory or container length.
  • For everything else, int is fine.

Determining Variable Size

How can you measure how much memory a variable uses?

  • C++ provides the sizeof operator — the only portable way to measure memory.
  • It evaluates at compile time and works on any type, variable, or array.
  • The return type is size_t.

Total slides: 6 Keep this focused and practical. sizeof with arrays and sizeof with structs are deferred — arrays to lecture 11.02, struct padding to lecture 15.02.

The sizeof operator

sizeof(type)      // size of a type
sizeof variable   // size of a declared variable (parentheses optional)
sizeof(expr)      // size of the type an expression would produce
  • Does not execute code or evaluate the operand’s value — it determines size from type information alone.
  • Result is always in bytes.

Typical platform sizes

Sizes are not mandated by the standard and vary by platform and compiler. These are typical on a 64-bit system:

Type Typical Size
char 1 byte
short 2 bytes
int 4 bytes
long 8 bytes
float 4 bytes
double 8 bytes
bool 1 byte
pointer 8 bytes

Always use sizeof rather than hardcoding these values.

Why sizes vary

  • Architecture — 32-bit systems use 4-byte pointers; 64-bit systems use 8-byte pointers.
  • Compiler choices — long is 4 bytes on Windows (MSVC) and 8 bytes on Linux (GCC/Clang).
  • Alignment — the compiler inserts padding between struct members to align data on word boundaries for performance.

Printing all type sizes

#include <iostream>
using namespace std;

int main()
{
    cout << "char:   " << sizeof(char)   << " byte(s)\n";
    cout << "short:  " << sizeof(short)  << " byte(s)\n";
    cout << "int:    " << sizeof(int)    << " byte(s)\n";
    cout << "long:   " << sizeof(long)   << " byte(s)\n";
    cout << "float:  " << sizeof(float)  << " byte(s)\n";
    cout << "double: " << sizeof(double) << " byte(s)\n";
    cout << "bool:   " << sizeof(bool)   << " byte(s)\n";
    return 0;
}

Students should run this on their own machine. The output is specific to their compiler and platform. Comparing results with classmates on different systems is an effective way to make platform dependence concrete.

Summary

  • sizeof is the portable, compile-time way to determine memory size.
  • Use it instead of hardcoding byte counts.
  • Sizes vary by architecture and compiler — always measure, never assume.

Key Takeaways

  • The compiler works in three stages: preprocessing → compilation → linking. Error messages tell you which stage failed.
  • The symbol table tracks every identifier’s name, type, scope, and address — type mismatches and undeclared variables are symbol table violations.
  • Implicit coercion promotes silently; narrowing loses data silently. Use static_cast<> to make conversions explicit.
  • Signed integer overflow is undefined behavior. Unsigned overflow wraps — well-defined.
  • Mixing signed and unsigned in comparisons silently converts signed to unsigned — enable -Wall to catch it.
  • Use int32_t and friends when exact width matters; use size_t for sizes and indexes.
  • sizeof is the only portable way to measure type size; struct padding means size ≠ sum of members.

The Symbol Table: Good Practices

  • Declare variables at their first point of use, not at the top of the function — reduces the scope visible in the table.
  • Use distinct, meaningful names to avoid accidental shadowing.
  • Enable -Wshadow to catch cases where a local variable shadows a global.

The Symbol Table: Common Mistakes

  • Using a variable before declaring it — the compiler searches the table and finds nothing.
  • Declaring the same variable twice in the same scope — the compiler finds a duplicate entry.
  • Assuming that a variable in one function is visible in another — each function has its own scope region.

The Symbol Table: More Examples

// Demonstrating scope regions
int x = 100;           // global scope

void show() {
    int x = 200;       // local scope — shadows global
    cout << x << "\n"; // prints 200
}

int main() {
    cout << x << "\n"; // prints 100 — global, no local x here
    show();
    return 0;
}

Data Type Conversion: Good Practices

Overflow and Underflow: Good Practices

  • Always validate user input before using it in arithmetic — never assume the user enters a value within range.
  • Use numeric_limits<T>::max() rather than magic numbers like INT_MAX in production code.
  • For any loop counter or accumulator that could grow large, estimate the maximum possible value and choose a type accordingly.

Overflow and Underflow: Common Mistakes

  • Assuming signed overflow wraps to a negative number — it is undefined behavior.
  • Decrementing an unsigned counter to zero and then decrementing again — it wraps to a large positive number.
  • Using int for a loop that accumulates values beyond 2 billion.

Overflow and Underflow: More Examples

// Unsigned counter underflow trap
unsigned int count = 0;
while (count >= 0) {   // always true — unsigned is never < 0
    count--;           // wraps from 0 to UINT_MAX
}
// Infinite loop — classic unsigned underflow bug

Signed vs Unsigned: Good Practices

  • Use int by default; switch to unsigned only when the value genuinely cannot be negative.
  • Match the type of your loop counter to the type returned by .size() — use size_t or an explicit cast.
  • Compile with -Wall and address every sign-comparison warning.

Signed vs Unsigned: Common Mistakes

  • Comparing a signed loop counter to an unsigned size — sign-comparison warning, potential logic error.
  • Storing a negative return value (like -1 for “not found”) in an unsigned variable.
  • Assuming that unsigned int u = -1 causes a compile error — it compiles and stores UINT_MAX.

Signed vs Unsigned: More Examples

// (size_t)0 - 1 wraps to a huge number — loop runs forever

Determining Variable Size: Good Practices

  • Always use sizeof instead of hardcoded byte counts — even if you are “sure” of the size.
  • Use sizeof(arr) / sizeof(arr[0]) for static array element count, but prefer std::array or std::vector in new code.
  • Reorder struct members from largest to smallest to minimize padding.

Determining Variable Size: Common Mistakes

  • Passing an array to a function and using sizeof on the parameter — you get pointer size, not array size.
  • Assuming sizeof(struct) equals the sum of its member sizes — padding makes this wrong.
  • Using sizeof on a pointer and thinking you measured the pointed-to data.
