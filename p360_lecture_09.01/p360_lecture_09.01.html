Increment and Decrement Operators

Why Does i = i + 1 Have A Shortcut?

Loop code often changes a control variable by exactly one. C++ gives that pattern its own notation because it is so common.

Slides 3-11

The Operators

Operator Meaning Equivalent statement
x++ increase x by 1 x = x + 1
++x increase x by 1 x = x + 1
x-- decrease x by 1 x = x - 1
--x decrease x by 1 x = x - 1

When the operator is alone on a line, prefix and postfix forms leave the variable with the same final value.

The Loop Use Case

for (int i = 1; i <= 5; i++) {
    cout << i << endl;
}

The update expression i++ means “move to the next value.” In loop headers, i++ is the conventional form students will see most often.

Counting Down

for (int lives = 3; lives > 0; lives--) {
    cout << lives << " lives remaining" << endl;
}

cout << "Game over" << endl;

Use -- when each pass moves the control variable one step downward.

Prefix vs Postfix

The difference matters only when the expression’s value is used.

int x = 5;

cout << x++ << endl;  // prints 5, then x becomes 6
cout << ++x << endl;  // x becomes 7, then prints 7

Avoid mixing increment operators into larger expressions until the order is clear.

Compound Assignment

Use compound assignment when the step is not exactly one.

Statement Meaning
x += 5; x = x + 5;
x -= 2; x = x - 2;
x *= 3; x = x * 3;
x /= 4; x = x / 4;

++ and -- are special cases of this broader shorthand family.

Readability Rule

Prefer the form that makes the loop movement obvious.

for (int i = 0; i < count; i++) {
    // standard one-step loop
}

for (int i = 0; i <= 100; i += 10) {
    // clearer than ten separate increments
}

Shorthand should make the code easier to scan, not harder to reason about.

Summary

Use ++ and -- for one-step movement. Use compound assignment for larger steps, and keep prefix/postfix differences out of complex expressions unless the order matters.

Break

When Should A Loop Stop Immediately?

Most loops stop when the condition becomes false. break gives the loop a second exit point from inside the body.

Slides 12-20

What break Does

break exits the nearest loop immediately. Execution continues after the loop.

for (int i = 1; i <= 10; i++) {
    if (i == 5) {
        break;
    }

    cout << i << endl;
}

cout << "Done" << endl;

The output stops at 4 because the loop exits before printing 5.

Break On A Sentinel

int value;

while (true) {
    cout << "Enter a value (-1 to quit): ";
    cin >> value;

    if (value == -1) {
        break;
    }

    cout << "You entered " << value << endl;
}

This pattern is useful when the loop must read before it knows whether to stop.

Break During A Search

int target = 72;
int foundAt = -1;

for (int i = 0; i < scores.size(); i++) {
    if (scores.at(i) == target) {
        foundAt = i;
        break;
    }
}

Once the target is found, the remaining loop passes cannot improve the answer.

Break Exits One Level

In nested loops, break exits only the loop it is directly inside.

for (int row = 1; row <= 3; row++) {
    for (int col = 1; col <= 3; col++) {
        if (col == 2) {
            break;
        }

        cout << row << "," << col << endl;
    }
}

The outer loop still continues with the next row.

Break vs Loop Condition

Prefer a loop condition when the stopping rule is simple.

while (choice != 'Q') {
    displayMenu();
    cin >> choice;
}

Use break when the clearest stopping moment happens inside the loop body.

Break In A switch

Inside a switch, break exits the switch, not the surrounding loop.

while (choice != 'Q') {
    cin >> choice;

    switch (choice) {
    case 'A':
        cout << "Add" << endl;
        break;
    }
}

This break prevents fall-through to the next case.

Summary

break stops the nearest loop or switch immediately. Use it when stopping from inside the body is clearer than forcing the exit into the loop condition.

Continue

When Should A Loop Skip The Rest Of This Pass?

continue does not stop the loop. It skips the remaining statements in the current pass and moves to the next condition check.

Slides 21-28

What continue Does

for (int i = 1; i <= 5; i++) {
    if (i == 3) {
        continue;
    }

    cout << i << endl;
}

The output is 1, 2, 4, and 5. The loop keeps running, but the print statement is skipped when i is 3.

Continue vs Break

Statement Effect
continue skip the rest of this pass
break exit the loop immediately

Use continue when this value should be ignored but the loop should keep checking later values.

Skipping Invalid Items

int total = 0;

for (int score : scores) {
    if (score < 0) {
        continue;
    }

    total += score;
}

Negative scores are ignored. Valid scores still contribute to the total.

Continue In A While Loop

Update the loop-control variable before continue.

int i = 0;

while (i < 10) {
    i++;

    if (i % 2 == 0) {
        continue;
    }

    cout << i << endl;
}

If the update comes after continue, the loop may never move forward.

When To Use Continue

  • Skip data that should not be processed.
  • Keep the main path less indented.
  • Make the “ignore this case” rule explicit.

Use it sparingly. A simple if statement is often clearer.

Summary

continue skips the rest of the current loop pass and returns to the next condition check or update step. It is useful for ignoring one case while allowing the loop to keep working.

Input Validation Strategy

What Can We Trust?

  • Client input begins outside the program’s control.
  • Validation decides when that input is safe enough to use.

Slides 3-17

Why Validate Input?

  • Bad input can corrupt program state.
  • It can break calculations or send the program down the wrong branch.
  • Validation protects the boundary before the value is trusted.

Minimum Validation Behavior

  • Identify the problem.
  • Give a specific correction message.
  • Reprompt so the client can fix the value.

Where Validation Belongs

  • Validate right after input is read.
  • Do it before calculations, branches, vectors, or function calls use the value.
  • The rest of the program should receive already-checked data.

Two General Strategies

Strategy Meaning
Allow by default Accept most input and reject known bad values
Deny by default Reject everything except known good values

[Graphic suggestion: two doors labeled “default open” and “default closed” with a guard checking names.]

Deny By Default

  • Deny by default works well when the valid set is small.
  • Menus, yes/no prompts, and known commands are strong fits.
  • The program accepts only the choices it understands.

Allow By Default

  • Allow by default can reject a short list of known bad inputs.
  • It is better than no validation when the invalid set is small.
  • It is weak when the program needs evidence that the data is correct.

The Gauntlet Of Trust

Check Question
Presence Is there data?
Shape Is it the kind of input we need?
Range or set Is it an expected value?

[Graphic suggestion: three gates labeled presence, shape, and range, with trust increasing after each gate.]

Start With Presence

  • Empty input is different from wrong input.
  • For full-line input, use getline() so the program can see the whole response.
  • A blank string should be handled before checking more specific rules.

Check The Shape

  • Shape means the input has the form the program expects.
  • A whole number, a nonblank string, and a yes/no answer are different shapes.
  • This check comes before asking whether the value is in range.

Check The Range Or Set

  • A readable value can still be unacceptable.
  • Range checks ask whether a number is between legal bounds.
  • Set checks ask whether a value is one of the legal choices.

Why The Error Message Matters

  • “Invalid input” tells the client almost nothing.
  • “Enter a whole number from 0 to 100” names the correction.
  • Precise feedback reduces repeated mistakes.

Broad To Narrow

  • First ask whether anything was entered.
  • Then ask whether the data has the right shape.
  • Then ask whether it meets the specific rule for this program.

Summary

  • Validation builds trust in stages.
  • Check broad failures first, explain clearly, and reprompt until the program has usable data.

Input Validation

What Makes Input Safe To Use?

Input validation checks client data before the rest of the program trusts it. The goal is to reject bad input early, explain the problem, and ask again.

Slides 44-57

Validate At The Boundary

The best time to validate input is immediately after reading it.

  • The client enters raw data.
  • The validation code decides whether the data is acceptable.
  • The rest of the program receives data that has already been checked.

Do not let questionable data travel deeper into the program.

The Minimum Behavior

A validation loop should do three things.

  • State what is wrong.
  • Tell the client what valid input looks like.
  • Reprompt until the input is usable.

“Invalid input” is rarely enough. A useful message teaches the client how to recover.

Range Validation

int score;

cout << "Enter a score (0-100): ";
cin >> score;

while (score < 0 || score > 100) {
    cout << "Score must be between 0 and 100. Try again: ";
    cin >> score;
}

The loop condition describes bad data. The loop stops when the value is acceptable.

Set Validation

char answer;

cout << "Continue? (Y/N): ";
cin >> answer;

while (answer != 'Y' && answer != 'N') {
    cout << "Enter Y or N: ";
    cin >> answer;
}

Use set validation when the legal values are specific choices rather than a continuous range.

The Priming Read

A while validation loop needs a first value to test.

cin >> value;                 // priming read

while (value is invalid) {
    cout << "Try again: ";
    cin >> value;             // replacement read
}

Without the priming read, the first condition check has no meaningful input value.

Validate Before Processing

int count;

cout << "How many scores (1-30)? ";
cin >> count;

while (count < 1 || count > 30) {
    cout << "Enter a count from 1 to 30: ";
    cin >> count;
}

for (int i = 1; i <= count; i++) {
    cout << "Score " << i << ": ";
}

The loop that uses count should not run until count has been validated.

Type Failure Is Different

Range validation assumes the extraction succeeded. If the client types abc for an integer, cin enters a failed state.

while (!(cin >> age)) {
    cin.clear();
    cin.ignore(10000, '\n');
    cout << "Enter a whole-number age: ";
}

Recover from type failure before applying ordinary range checks.

Simple Validation

int age;

cout << "Enter your age (1-120): ";

while (!(cin >> age) || age < 1 || age > 120) {
    cin.clear();
    cin.ignore(10000, '\n');
    cout << "Enter a whole-number age from 1 to 120: ";
}

This combines type recovery with range validation in one loop.

Summary

Input validation protects the program boundary. Read the value, reject invalid data with a specific message, reprompt, and only process the value after it passes the check.

  • continue skips the rest of the current loop pass but keeps the loop running.
  • Validation strategy decides what a program must check before it trusts input.
  • Input validation protects the program boundary before client data is trusted.

Increment and Decrement Operators: Good Practices

  • Use i++ in ordinary for-loop update expressions.
  • Use += or -= when the loop moves by more than one.
  • Keep increment and decrement statements easy to see.

Increment and Decrement Operators: Common Mistakes

  • Assuming x++ and ++x produce the same value inside a larger expression.
  • Incrementing the wrong variable in a nested or multi-variable loop.
  • Forgetting that ++ always changes the value by exactly one.

Break: Good Practices

  • Keep the reason for break close to the break statement.
  • Prefer a clear loop condition when it expresses the exit naturally.
  • Use break in searches when the answer is already found.

Break: Common Mistakes

  • Assuming one break exits every nested loop.
  • Confusing break in a switch with break in a loop.
  • Using many scattered break statements when one loop condition would be clearer.

Break: More Examples

int value;
int count = 0;

while (cin >> value) {
    if (value < 0) {
        break;
    }

    count++;
}

Input Validation Strategy: Good Practices

  • Validate from broadest condition to narrowest condition.
  • Give a specific error message and reprompt.
  • Keep invalid input from reaching the rest of the program.

Input Validation Strategy: Common Mistakes

  • Checking range before checking whether the input is the right type.
  • Printing a generic error message when the client needs a specific correction.
  • Using a value before the program has decided it is safe.

Input Validation Strategy: More Examples
