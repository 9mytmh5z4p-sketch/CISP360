Advanced cin

Input Beyond One Word

  • What happens when user input is messier than the example?
  • cin is predictable, but we need to know what it consumes and what it leaves behind.

Slides 3-15

Two Input Problems

  • cin >> stops when it reaches whitespace.
  • cin fails when the input does not match the receiving variable’s type.

These are different problems with different fixes.

Whitespace Stops Extraction

cin >> reads one token at a time.

string fullName;

cout << "Full name: ";
cin >> fullName;

If the user types Maya Chen, only Maya is stored.

Whitespace Output

getline Reads the Whole Line

getline(cin, variable) reads through the end of the line.

string fullName;

cout << "Full name: ";
getline(cin, fullName);

Use getline when spaces are part of the input.

cin and getline Conflict

cin >> often leaves the newline character behind.

int age;
string name;

cin >> age;
getline(cin, name);

The getline may read the leftover newline instead of waiting for a name.

cin.ignore Clears the Line

Use cin.ignore() after cin >> when the next read is getline.

int age;
string name;

cin >> age;
cin.ignore();
getline(cin, name);

The ignored newline is removed before the full-line read.

Wrong Data Type

If the user types text when the program expects a number, extraction fails.

int age;

cout << "Age: ";
cin >> age;

Typing ten does not produce an integer.

Mismatch Output

Clearing a Failed Read

After a failed read, clear the failure and remove the bad input.

cin.clear();
cin.ignore(1000, '\n');

clear resets the stream state; ignore discards the leftover characters.

Common Practices

  • Use cin >> for one-word or one-number input.
  • Use getline when spaces are meaningful.
  • After failed numeric input, clear the stream before reading again.

Summary

  • Whitespace is a separator for cin >>.
  • Wrong data types put cin into a failed state.
  • Good input handling means knowing what remains in the input stream.

Dates and Random Numbers

Programs Need Changing Values

  • Some programs need values that change while the program runs.
  • Date/time values and random numbers let programs respond to the current moment.

Slides 16-29

Date and Time

Date and time functions let a program ask the computer for the current moment.

This is useful for timestamps, elapsed time, and seeding random numbers.

Getting the Current Time

time(0) returns the current time as a numeric timestamp.

#include <iostream>
#include <ctime>        // library for time
using namespace std;

int main() {
    time_t now = time(0);
    cout << now << endl;
    return 0;
}

Date Illustration

Calendar time:     June 4, 2026, 10:30 AM
Timestamp value:   seconds since a fixed starting point
Program use:       store, compare, seed, or display

[Graphic suggestion: a calendar label connected to a numeric timestamp, then to a program output line.]

Time as a Number

The timestamp is useful even before we format it as a human-readable date.

  • It changes every second.
  • It can be stored in a variable.
  • It can make random-number sequences start differently.

Random Numbers

Random numbers let a program choose from several possible values.

They are useful for games, simulations, sampling, and practice programs.

Why Random Needs a Seed

rand() follows a repeatable sequence.

The seed chooses where that sequence starts.

srand(5);

The same seed produces the same sequence.

Seeding with Time

Use the current time as a changing seed.

#include <cstdlib>
#include <ctime>

srand(time(0));

Seed once near the beginning of the program.

Generating a Random Number

rand() returns a large integer.

int raw = rand();

The raw value is usually too large for the program’s actual need.

Limiting the Range

Use modulus to reduce the raw value to a smaller range.

int roll = rand() % 6 + 1;

This produces values from 1 through 6.

Die Roll Example

#include <iostream>
#include <cstdlib>
#include <ctime>
using namespace std;

int main() {
    srand(time(0));

    int roll = rand() % 6 + 1;
    cout << "Roll: " << roll << endl;

    return 0;
}

Common Practices

  • Seed once, before calling rand.
  • Use a range formula that matches the values you want.
  • Use random numbers only when unpredictability helps the program.

Summary

  • Time gives a changing value from the system.
  • A seed controls where the random sequence begins.
  • A range formula turns raw random numbers into useful program values.

IPO Models

Before Code, Name the Shape

  • How can we plan a program before writing statements?
  • IPO separates a problem into inputs, processing, and outputs.

Slides 30-43

What Is an IPO Model?

An IPO model names the three major parts of a program.

  • Input: data the program receives.
  • Processing: work the program does.
  • Output: result the program produces.

Input

Inputs are the values the program needs before it can do useful work.

They may come from the user, a file, a constant, or a system function.

Processing

Processing is the calculation, decision, or transformation applied to the input.

This is where the program changes data into a useful result.

Output

Outputs are the values, messages, or reports the program produces.

Good outputs answer the problem the program was written to solve.

A Small IPO Table

Input Processing Output
quantity, price total = quantity * price total price

The table turns a word problem into a program plan.

From IPO to Variables

int quantity;
double price;
double total;

total = quantity * price;

Inputs often become variables; processing becomes statements; output becomes cout.

Common Practices

  • Identify the output first.
  • Work backward to the required inputs.
  • Name the processing step as a verb or formula.

Date and Time IPO

Input Processing Output
current system time store timestamp printed timestamp

The program does not ask the user for the time; it asks the system.

Date and Time Variables

time_t now;   // a time type variable

now = time(0);
cout << now << endl;

The input comes from time(0), the variable stores it, and cout displays it.

Random Number IPO

Input Processing Output
seed, raw random value raw % 6 + 1 die roll

The processing step turns a large raw value into the range the program needs.

Random Number Variables

int raw;
int roll;

raw = rand();
roll = raw % 6 + 1;

Separating raw and roll makes the design easier to trace.

Summary

  • IPO identifies what comes in, what happens, and what comes out.
  • The model helps choose variables before coding.
  • A clear IPO plan makes code easier to write and test.

Execution Trace Models

Follow the Program, One Line at a Time

  • How do we predict what code will do before running it?
  • An execution trace records statements, variables, and output in order.

Slides 44-55

What Is an Execution Trace?

An execution trace is a table or notes showing program state after each statement runs.

It slows the program down enough for humans to inspect it.

Trace Columns

Useful trace columns include:

  • line number
  • statement
  • variable values
  • output

Only update the values that change on that line.

Simple Assignment Trace

int a = 4;
int b = 6;
int total = a + b;
Line a b total
1 4
2 4 6
3 4 6 10

Output in a Trace

When cout runs, record exactly what appears.

Statement total Output
cout << total; 10 10

Output is part of the program’s observable behavior.

Input in a Trace

When cin runs, choose an assumed user input and record the stored value.

User types Statement age
19 cin >> age; 19

The trace depends on the input you assume.

Trace vs. IPO

  • IPO is a design model used before writing code.
  • An execution trace is a reading model used after code exists.
  • IPO asks “what should happen?”; tracing asks “what does happen?”

Date and Time Trace

Statement now Output
now = time(0); 1780594200
cout << now; 1780594200 1780594200

The exact timestamp changes, but the trace pattern stays the same.

Random Number Trace

Statement raw roll Output
raw = rand(); 41
roll = raw % 6 + 1; 41 6
cout << roll; 41 6 6

Tracing separates the raw value from the final bounded value.

Common Practices

  • Trace statements in execution order.
  • Update only the values that change.
  • Record output exactly.
  • Do not skip lines that seem obvious.

Summary

  • Execution traces show how program state changes.
  • They help catch mistaken assumptions before or after running code.
  • A good trace makes invisible variable changes visible.

Terminal Charts and Graphs

When Output Becomes a Display

  • How can plain terminal output become something easier to scan?
  • Terminal charts use labels, spacing, and repeated characters to show simple data.

Slides 56-65

Chart as Design Problem

An output display still starts with IPO.

Input Processing Output
labels, values choose scale and spacing chart or table

The chart is planned before it is printed.

Bar Charts With Characters

A terminal bar chart uses repeated characters to represent quantity.

Apples  #####
Oranges ###

The longer bar represents the larger value.

One Bar by Hand

cout << left << setw(10) << "Apples"
     << "#####" << endl;

Start with one correct row before building a larger display.

Several Bars by Repetition

cout << left << setw(10) << "Apples"  << "#####" << endl;
cout << left << setw(10) << "Oranges" << "###"   << endl;
cout << left << setw(10) << "Pears"   << "####"  << endl;

Each row follows the same output pattern.

Tables as Charts

Tables are charts when their layout helps the reader compare values.

Item        Qty
Apples        5
Oranges       3
Pears         4

Alignment turns separate lines into one display.

A Small Sales Display

Sales Display
Item        Qty  Bar
Apples        5  #####
Oranges       3  ###
Pears         4  ####

The table gives exact values; the bars make comparison fast.

Common Practices

  • Keep labels aligned.
  • Choose a simple scale.
  • Use the same row pattern throughout the display.
  • Make the chart readable before making it clever.

Summary

  • Terminal charts are planned output.
  • Labels, spacing, and repeated symbols carry the meaning.
  • A good display makes the important comparison visible.

Key Takeaways

  • cin >> stops at whitespace; getline reads a whole line.
  • Failed numeric input must be cleared before the stream can be used again.
  • Time can provide a changing value, and random numbers need a seed and range formula.
  • IPO models plan a program before code; execution traces inspect code after it exists.
  • Terminal charts are structured output designed for quick comparison.

Advanced cin: Good Practices

  • Use cin >> for simple single-token values.
  • Use getline when spaces are part of the input.
  • Clear failed input before reading again.

Advanced cin: Common Mistakes

  • Expecting cin >> to read a full name.
  • Mixing cin >> and getline without handling the leftover newline.
  • Trying another read after a failed numeric extraction without clearing the stream.

Advanced cin: More Examples

Dates and Random Numbers: Good Practices

  • Seed the random-number generator once near the start of the program.
  • Use clear variable names such as raw, roll, and now.
  • Keep range formulas visible until the pattern is familiar.

Dates and Random Numbers: Common Mistakes

  • Calling srand(time(0)) repeatedly before every random number.
  • Forgetting the + 1 when the desired range starts at 1.
  • Treating a timestamp as a formatted calendar date.

IPO Models: Good Practices

  • Start with the output the program must produce.

  • Treating every variable as an input.
  • Skipping processing because the calculation feels obvious.

Execution Trace Models: Good Practices

  • Trace statements in the order they execute.
  • Record output exactly when it happens.
  • Update only the variable values that change.

Execution Trace Models: Common Mistakes

  • Writing what you expect instead of what the code says.
  • Skipping assignments because they look simple.
  • Recording final output before the cout statement runs.

  • Choosing a scale that exaggerates or hides differences.
  • Mixing exact values and visual bars without clear labels.
