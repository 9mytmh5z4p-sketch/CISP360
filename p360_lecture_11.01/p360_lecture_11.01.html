Introduction to Arrays

A fixed-size collection on the stack

  • Same idea as a vector — different contract
  • Size is locked in at compile time; no growing, no shrinking

Total slides: 5

What is an array?

  • A sequence of elements, all the same type, stored contiguously in memory
  • Size is fixed at compile time — you state it once in the declaration
  • Elements are accessed by index, starting at 0

[Graphic suggestion: a row of five labeled boxes showing int arr[5] with indices 0–4 marked below each box and the array name pointing to the first box]

Arrays vs. vectors

Array Vector
Size Fixed at compile time Grows at runtime
Storage Stack Heap
Bounds check None None (.at() throws)
Overhead Zero Pointer + size + capacity

Declaring an array

int scores[5];        // uninitialized — garbage values
double temps[7];      // uninitialized — garbage values
char letters[26];     // uninitialized — garbage values
  • Type comes first, then the name, then the size in brackets
  • Size must be a compile-time constant — a literal or a const integer
  • Without an initializer list, elements hold whatever was in memory

Summary

  • An array is a fixed-size, contiguous block of same-type elements allocated on the stack
  • Use an array when size is known at compile time and the collection does not need to grow

Array Initialization

Declaring vs. initializing — what’s the difference?

  • Declaration allocates the space; initialization sets the values
  • Arrays give you several ways to do both at once

Total slides: 6

Initializer list — full

int scores[5] = {91, 78, 84, 95, 62};  // all 5 values
  • Brace-enclose the values separated by commas
  • The count of values must not exceed the declared size
  • If the count matches the size, you can omit the size entirely:
int scores[] = {91, 78, 84, 95, 62};   // size inferred as 5

Initializer list — partial

int scores[5] = {91, 78};   // first 2 set; rest zero-initialized
  • Provide fewer values than the declared size
  • Remaining elements are zero-initialized automatically
  • This is the idiomatic way to zero out an array:
int scores[5] = {};          // all elements zero
int scores[5] = {0};         // same effect

Size rules

const int SIZE = 5;
int scores[SIZE];             // OK — const is compile-time

int n = 5;
int scores[n];                // not standard C++ — avoid
  • Array size must be a compile-time constant
  • A plain int variable is not a compile-time constant, even if you assign it immediately
  • Prefer const int or a named constant for the size

Variable-length arrays (VLAs) are a GCC extension and exist in C99, but they are not standard C++ and should not be taught as portable practice.

What if you don’t initialize?

int scores[5];          // uninitialized — garbage values
  • Local arrays without an initializer hold whatever was in memory at that address
  • Reading an uninitialized element is undefined behavior
  • Always initialize, or fill every element before reading it

Summary

  • Provide an initializer list to set values at declaration; = {} zeros the entire array
  • Size must be a compile-time constant — use const int for named sizes
  • Uninitialized local arrays hold garbage; always initialize before reading

Accessing Array Elements in Loops

Getting to each element — index or range?

  • Two loop styles work with arrays, each with different tradeoffs
  • Choosing wrong loses the index or risks going out of bounds

Total slides: 6

The subscript operator

int scores[5] = {91, 78, 84, 95, 62};

scores[0];   // 91 — first element
scores[4];   // 62 — last element
scores[5];   // out of bounds — undefined behavior
  • array[i] reads or writes element at index i
  • Valid indices: 0 through size - 1
  • C++ does not check bounds — going past the end is undefined behavior

Index-based for loop

const int SIZE = 5;
int scores[SIZE] = {91, 78, 84, 95, 62};

for (int i = 0; i < SIZE; i++) {
    cout << scores[i] << "\n";  // access by index
}

Use SIZE (a named constant) as the loop bound — not a magic number. The loop variable i is available inside the body, which allows computing positions, comparing neighbors, or writing back to the array.

Range-for loop

int scores[5] = {91, 78, 84, 95, 62};

for (int score : scores) {
    cout << score << "\n";   // no index needed
}

Range-for reads the array size from the declaration automatically — no SIZE constant needed. The element variable (score) is a copy by default; use int& score to modify elements in place.

Which loop to use?

Situation Loop
Need the index (position, neighbor, write back) for (int i = 0; i < SIZE; i++)
Read every element, no index needed for (int x : arr)
Modify elements in place for (int& x : arr)
  • When the index is irrelevant, range-for is cleaner and less error-prone
  • When you need the position or must write back by index, use the index-based form

Summary

  • Use array[i] to read or write a single element; valid indices run from 0 to size - 1
  • Index-based for gives you the position; range-for hides it — choose based on whether the index matters

Inputting and Displaying Array Data

Reading values from the user into an array

  • The pattern is a loop — same index mechanics, now with cin
  • Fill first, display second: two separate passes

Total slides: 5

Input loop — filling the array

const int SIZE = 5;
int scores[SIZE];

for (int i = 0; i < SIZE; i++) {
    cout << "Enter score " << i + 1 << ": ";
    cin  >> scores[i];              // fill each element
}

i + 1 in the prompt gives users a 1-based label (“Enter score 1”) while the array remains 0-indexed. This is a common display convention worth pointing out.

Display loop — printing the array

for (int i = 0; i < SIZE; i++) {
    cout << "Score " << i + 1 << ": " << scores[i] << "\n";
}

This is deliberately separate from the input loop. Mixing input and output in a single loop works but makes programs harder to read and test. Teach the two-pass pattern early.

Combined example

#include <iostream>
using namespace std;
const int SIZE = 5;

int main() {
    int scores[SIZE];

    for (int i = 0; i < SIZE; i++) {       // fill
        cout << "Enter score " << i + 1 << ": ";
        cin  >> scores[i];
    }

    for (int i = 0; i < SIZE; i++) {       // display
        cout << "Score " << i + 1 << ": " << scores[i] << "\n";
    }
}

Summary

  • Fill the array in one loop, display it in a second loop — keep the two passes separate
  • The same index-based for pattern applies to both; i + 1 in the prompt gives users a 1-based label

Vectors vs. Arrays

Same idea — different contract

  • You already know vectors; arrays trade flexibility for simplicity
  • Knowing both means choosing the right tool for the job

Total slides: 5

Side-by-side comparison

Array

const int SIZE = 5;
int scores[SIZE] = {};

scores[2] = 84;
// can't grow
// no .size()

Vector

vector<int> scores(5, 0);

scores[2] = 84;
scores.push_back(100);  // grows
scores.size();          // available

Bounds safety

int scores[5] = {91, 78, 84, 95, 62};
scores[5] = 99;          // out of bounds — undefined behavior
vector<int> scores = {91, 78, 84, 95, 62};
scores.at(5) = 99;       // throws out_of_range
scores[5] = 99;          // still undefined behavior
  • Neither [] on arrays nor [] on vectors checks bounds
  • Vectors expose .at() as a checked alternative; arrays have no equivalent

When to use each

Use an array when… Use a vector when…
Size is fixed and known at compile time Size is unknown or changes at runtime
You need zero overhead You need push_back, size, or algorithms
Interfacing with C APIs or hardware General-purpose collection work
  • Default to vector — it is safer and more flexible
  • Reach for arrays only when overhead or API constraints require it

Summary

  • Arrays are fixed-size and stack-allocated; vectors are dynamic and heap-allocated
  • Neither [] operator checks bounds — use .at() on vectors when safety matters, and be deliberate with array indices

Performance — Vectors vs. Arrays

Stack vs. heap — does the difference matter?

  • Arrays live on the stack; vectors allocate on the heap
  • The difference is measurable — but rarely the deciding factor

Total slides: 5

Why arrays are faster — in theory

  • Stack allocation: the compiler reserves space at function entry; no system call, no bookkeeping
  • Heap allocation: new (what vector uses internally) requests memory from the OS; involves overhead and possible cache misses
  • Arrays also avoid storing a pointer, size, and capacity — three extra words that vectors carry

[Graphic suggestion: two side-by-side memory diagrams — left shows the call stack with an array block sitting directly inside the frame; right shows the call stack with a vector object (pointer + size + capacity) pointing out to a heap block]

What the numbers look like

  • For small arrays (< a few hundred elements), the practical difference is nanoseconds
  • The bottleneck in most programs is I/O, algorithm complexity, or cache misses — not stack vs. heap
  • Modern allocators are fast; vector’s heap overhead is often negligible

If you have a timing demo ready, show it here. A tight loop summing 1,000 ints as an array vs. vector will show a small but real difference. The point is not that arrays are always faster — it’s that the difference matters only in constrained contexts.

When the difference actually matters

  • Embedded systems and microcontrollers with no heap (or a tiny one)
  • Real-time systems where allocation latency must be bounded
  • Hot inner loops called millions of times in numerical or signal-processing code
  • Interfacing with C libraries that expect raw arrays

Summary

  • Arrays are faster than vectors because stack allocation has no overhead and no indirection
  • The gap matters in constrained or performance-critical contexts; for general programming, choose based on safety and flexibility, not speed

Key Takeaways

  • An array is a fixed-size, contiguous block of same-type elements on the stack — declare size with a const int, initialize with = {}
  • Access elements with array[i]; valid indices run 0 through size - 1 — C++ does not check bounds
  • Use index-based for when you need the position; use range-for when you don’t
  • Default to vector; reach for an array when size is fixed at compile time or overhead must be zero
