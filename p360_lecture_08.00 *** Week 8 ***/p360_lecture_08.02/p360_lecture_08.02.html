Validating User Input

Validating User Input

Programs trust the user to provide correct data. That trust is almost always misplaced. Input validation is how we make sure only good data reaches the rest of the program.

Total slides: 7

Why Validate?

  • Programs that accept bad data produce bad results — or crash entirely.
  • The user may make typos, misread a prompt, or deliberately enter garbage.
  • Validation is not optional. It is part of the contract between the program and the client.

Two Validation Strategies

Every validation approach falls into one of two categories.

Let everything in

  • Default: accept all input.
  • Explicitly reject what you don’t want.
  • Example: accept any name except “Bob”.

Keep everything out

  • Default: reject all input.
  • Explicitly allow what you want.
  • Example: accept only “Sue” or “Dan”.

Let Everything In — Example

string name;
cout << "Enter your name: ";
cin >> name;

while (name == "Bob") {
    cout << "Access denied. Enter your name: ";
    cin >> name;
}

cout << "Welcome, " << name << "\n";

Keep Everything Out — Example

string name;
cout << "Enter your name: ";
cin >> name;

while (name != "Sue" && name != "Dan") {
    cout << "Access denied. Enter your name: ";
    cin >> name;
}

cout << "Welcome, " << name << "\n";

Which Strategy to Use?

  • Let everything in works when the invalid set is small and well-defined (e.g., one banned user).
  • Keep everything out works for small, fixed valid sets (e.g., a menu with 3 choices).
  • For general input validation — zip codes, account numbers, ranges — neither is enough on its own. You need a finer-grained approach.

Instructor note: This slide sets up the Gauntlet of Trust topic directly. General validation (zip codes, numbers in range) requires layered checks — which is exactly what the Gauntlet addresses.

Minimum Validation Behaviors

Every validation loop must do two things.

  • Give the client a specific error message. “Invalid input” is useless. “Entry must be between 1 and 10” is actionable.
  • Reprompt. Never silently accept bad data or silently exit. Give the client a chance to correct the mistake.

Summary

Input validation ensures only acceptable data reaches the program. The two core strategies — let everything in, or keep everything out — each have a natural home. Always give a specific error message and always reprompt.

Appendix

Common Practices

  • Default to “keep everything out” for menus — the valid set is small and fixed.
  • Default to “let everything in” only when the invalid set is explicitly smaller than the valid set.
  • Always test your validation loop with boundary values: the lowest valid input, the highest valid input, and one value just outside each boundary.

Common Mistakes

  • Accepting input outside the valid range without informing the user.
  • Giving a vague error message like “Error” or “Invalid” — the client cannot correct what they do not understand.
  • Forgetting to reprompt after rejecting input, forcing the user to restart the program.

More Examples

// Keep everything out: only accept 1, 2, or 3
int choice;
cout << "Enter 1, 2, or 3: ";
cin >> choice;

while (choice != 1 && choice != 2 && choice != 3) {
    cout << "Invalid. Enter 1, 2, or 3: ";
    cin >> choice;
}

  3. Why is “keep everything out” impractical for validating a user’s age (1–120)?

Range Checking

Range Checking

How do you know whether to trust a piece of input? Trust is not binary — it is earned in layers. Each layer filters out a different class of bad data.

Total slides: 6

The Gauntlet of Trust

Input passes through a series of checks, from broadest to narrowest. Each check earns a higher level of trust.

Low trust

Is there data?

Is it the right type?

High trust

Is it in range?

  • Empty or whitespace input fails the first check immediately.
  • Wrong type (a letter where a number is expected) fails the second.
  • Out-of-range values (age = 500) fail the third.

[Graphic suggestion: Vertical funnel or gauntlet diagram with three gates labeled “Has data?”, “Right type?”, “In range?” — input flows down and gets filtered at each gate.]

Level 1 — Is There Data?

The broadest check: did the user enter anything at all?

  • Empty input or whitespace-only input should fail immediately.
  • Use string and getline() — not cin >> — so you can catch empty strings before type conversion.
  • A blank entry with cin >> simply blocks, waiting for real input. You never see it as blank.

Level 2 — Is It the Right Type?

Did the user enter the kind of data the program expects?

  • Entering “abc” when an integer is expected is a type mismatch.
  • cin >> does not protect you — it silently fails and leaves garbage in the stream.
  • Robust type checking requires reading as a string first, then converting and validating.

Instructor note: Full type-safe validation is beyond the scope of this lecture. The key takeaway is awareness — students should know this layer exists. The textbook’s Figure 5-3 shows the full pattern.

Level 3 — Is It in Range?

The narrowest check: is the value within the set of values the program can handle?

int age;
cout << "Enter your age (1-120): ";
cin >> age;

while (age < 1 || age > 120) {
    cout << "Age must be between 1 and 120. Try again: ";
    cin >> age;
}

This is range checking — the most common form of input validation in introductory programs.

Rules of Thumb

  • Use a string data type when reading input that needs full validation.
  • Use getline(), not cin >>, to catch empty and whitespace input.
  • Move from broadest to narrowest check — empty first, type second, range last.
  • Display as specific an error message as possible.
  • Always allow the client to reenter incorrect data.

Summary

The Gauntlet of Trust is a layered approach to input validation: check for data first, then type, then range. Move from broadest to narrowest. Each layer filters a different class of bad input.

Appendix

Common Practices

  • Always perform range checking when the valid values form a continuous interval (e.g., 1–100, 0.0–1.0).
  • Combine range checking with a specific error message that states the valid range explicitly.

Common Mistakes

  • Checking range before checking type — a type mismatch corrupts the value before the range check runs.
  • Using cin >> to read input intended for full validation — it cannot detect empty strings or type errors cleanly.
  • Writing age > 1 && age < 120 instead of age >= 1 && age <= 120 — off-by-one on the boundary values.

More Examples

// Validate a percentage (0.0 to 100.0)
double pct;
cout << "Enter a percentage (0-100): ";
cin >> pct;

while (pct < 0.0 || pct > 100.0) {
    cout << "Must be between 0 and 100. Try again: ";
    cin >> pct;
}

  3. Why is range checking alone insufficient for validating a credit card number?

Input Validation with the While Loop

Input Validation with the While Loop

You know the concepts. Now here is the standard pattern for putting them into code — a while loop that enforces the Gauntlet before the rest of the program sees any data.

Total slides: 8

The Standard Pattern

// 1. Read input once before the loop (priming read)
cout << "Enter a value: ";
cin >> value;

// 2. Loop while the input is invalid
while (/* invalid condition */) {
    cout << "Specific error message. Try again: ";
    cin >> value;
}

// 3. Input is guaranteed valid here
  • Read before the loop so the condition has something to test.
  • Loop while the data is bad — not while it is good.
  • After the loop, proceed with confidence.

Range Validation

int score;
cout << "Enter a score (0-100): ";
cin >> score;

while (score < 0 || score > 100) {
    cout << "Score must be between 0 and 100. Try again: ";
    cin >> score;
}

cout << "Score accepted: " << score << "\n";

Multiple Conditions

A single validation loop can enforce multiple rules simultaneously.

int month;
cout << "Enter a month (1-12): ";
cin >> month;

while (month < 1 || month > 12) {
    cout << "Month must be between 1 and 12. Try again: ";
    cin >> month;
}

Combine conditions with || — the loop continues as long as any condition is violated.

Validating Against a Set

When valid values are not a continuous range, list them explicitly.

char grade;
cout << "Enter a grade (A, B, C, D, or F): ";
cin >> grade;

while (grade != 'A' && grade != 'B' && grade != 'C'
    && grade != 'D' && grade != 'F') {
    cout << "Invalid grade. Enter A, B, C, D, or F: ";
    cin >> grade;
}

This is the “keep everything out” strategy in code.

The Error Message Matters

Vague — unhelpful

cout << "Invalid input.\n"
     << "Try again: ";

Specific — actionable

cout << "Age must be between "
     << "1 and 120.\n"
     << "Try again: ";

The client cannot fix what they do not understand. Always state what the valid range or set is.

Validation Before Processing

int n;
cout << "How many students (1-30)? ";
cin >> n;

while (n < 1 || n > 30) {
    cout << "Must be between 1 and 30. Try again: ";
    cin >> n;
}

// n is guaranteed 1-30 — safe to use in a for loop
for (int i = 1; i <= n; i = i + 1) {
    int score;
    cout << "Score for student " << i << ": ";
    cin >> score;
}

Instructor note: This connects input validation directly to the for loop from 08.01. Validating n before the loop prevents a for loop from running a negative or zero number of times.

Summary

The standard input validation pattern: prime the read before the loop, loop while input is invalid, give a specific error message and reprompt. After the loop, the data is safe to use.

Appendix

Common Practices

  • Always prime the read — read once before the loop so the condition is meaningful on the first check.
  • State the valid range or valid set explicitly in every error message.
  • Validate all user input before it reaches any calculation or loop that depends on it.

Common Mistakes

  • Placing cin >> value only inside the loop body without a priming read — the condition has nothing to test on the first pass.
  • Using && instead of || in the loop condition — score < 0 && score > 100 is never true; use ||.
  • Forgetting to re-read the variable inside the loop body, causing an infinite loop.

More Examples

// Validate a password length (6-20 characters)
string password;
cout << "Enter a password (6-20 characters): ";
cin >> password;

while (password.length() < 6 || password.length() > 20) {
    cout << "Password must be 6-20 characters. Try again: ";
    cin >> password;
}
cout << "Password accepted.\n";

Practice Problems

  1. Write a validation loop that accepts a temperature in Fahrenheit between -50 and 150.
  2. Write a validation loop that accepts only ‘Y’ or ‘N’ from the user.
  3. What is wrong with this validation? while (x > 0 && x < 10) — when would this loop never exit?
  4. Add input validation to a program that asks for the number of items in a shopping cart (1–99).

Increment & Decrement Operators

Increment & Decrement Operators

You have been writing i = i + 1 since Week 8. C++ has a shorthand for exactly this — and it shows up in almost every loop you will ever read.

Total slides: 7

Note: Students have been writing i = i + 1 and i = i - 1 throughout 08.01. This topic gives them the shorthand they’ve been doing the long way. Explicitly connect to prior usage.

The Operators

Increment

++ adds 1 to a variable.

int x = 5;
x++;      // x is now 6
// same as: x = x + 1

Decrement

-- subtracts 1 from a variable.

int x = 5;
x--;      // x is now 4
// same as: x = x - 1

In a Loop

Every i = i + 1 in your prior loops can now be written as i++.

Before

for (int i = 1; i <= 5;
     i = i + 1) {
    cout << i << "\n";
}

After

for (int i = 1; i <= 5;
     i++) {
    cout << i << "\n";
}

The behavior is identical. i++ is the standard form — you will see it in every C++ textbook and codebase.

Pre vs Post

++ can go before or after the variable. In a standalone statement, the result is the same.

Post-increment

x++

Uses the current value, then increments.

Pre-increment

++x

Increments first, then uses the new value.

Instructor note: The difference only matters when the expression is used as a value (e.g., inside cout or as a function argument). In a for loop update position, both are equivalent. Keep this simple for now — the distinction matters more in later topics.

Where It Matters

In a standalone statement, x++ and ++x are equivalent. The difference appears when the expression produces a value.

int x = 5;
cout << x++;   // prints 5, then x becomes 6
cout << ++x;   // x becomes 7, then prints 7

For loop update expressions (i++ in a for header), the distinction never matters — use i++ by convention.

Compound Assignment

++ and -- are special cases of a broader family of shorthand operators.

Shorthand Equivalent
x++ x = x + 1
x-- x = x - 1
x += 5 x = x + 5
x -= 3 x = x - 3
x *= 2 x = x * 2

Summary

++ and -- are shorthand for adding or subtracting 1. Use i++ in for loop headers — it is the standard form. Post-increment and pre-increment differ only when the expression is used as a value.

Appendix

Common Practices

  • Use i++ (post-increment) in for loop headers — it is the conventional form and matches what you’ll find in textbooks and examples.
  • Use += and -= for step sizes other than 1 (e.g., i += 2 to count by twos).

Common Mistakes

  • Using ++i and i++ interchangeably inside expressions — they produce different values when used as part of a larger expression.
  • Applying ++ to a floating-point variable expecting a fractional increment — ++ always adds exactly 1.
  • Writing x++ = 5 — you cannot assign to the result of a post-increment expression.

More Examples

// Count down using --
for (int i = 10; i >= 1; i--) {
    cout << i << " ";
}
cout << "\nLiftoff!\n";

// Step by 2 using +=
for (int i = 0; i <= 20; i += 2) {
    cout << i << " ";
}

Practice Problems

  1. Rewrite this loop using ++: int i = 0; while (i < 10) { cout << i; i = i + 1; }
  2. What is the output? int x = 3; cout << x++ << " " << x;
  3. What is the output? int x = 3; cout << ++x << " " << x;
  4. Write a for loop that counts from 100 to 0 in steps of 10 using -- or -=.

Breaking Out of a Loop

Breaking Out of a Loop

Every loop exits when its condition becomes false. But what if you need to exit immediately — from the middle of the loop body, before the condition is checked?

Total slides: 7

The break Statement

break exits the nearest enclosing loop immediately. Execution continues with the first statement after the loop.

for (int i = 1; i <= 10; i++) {
    if (i == 5)
        break;          // exit the loop immediately
    cout << i << "\n";
}
cout << "Loop ended.\n";
// Output: 1, 2, 3, 4, Loop ended.

The loop runs until i == 5, then stops — even though the condition i <= 10 is still true.

break in a While Loop

int value;
while (true) {          // loop runs forever...
    cout << "Enter a value (-1 to quit): ";
    cin >> value;

    if (value == -1)
        break;          // ...until break fires

    cout << "You entered: " << value << "\n";
}
cout << "Goodbye.\n";

while (true) is an intentional infinite loop. break is the only exit. This pattern is common for interactive programs.

break Exits One Loop

break only exits the loop it is directly inside. In nested loops, it exits the inner loop only.

for (int i = 1; i <= 3; i++) {
    for (int j = 1; j <= 3; j++) {
        if (j == 2)
            break;      // exits inner loop only
        cout << i << "," << j << "\n";
    }
}
// Output: 1,1  2,1  3,1

Instructor note: Nested loops are not a focus of this lecture. This slide is awareness-only. Students don’t need to write nested loop break statements yet.

When to Use break

  • When the exit condition is complex and awkward to express in the loop header.
  • When the loop runs until the user signals done (sentinel pattern).
  • When searching a collection and you want to stop the moment you find a match.
// Stop searching when target is found
for (int i = 0; i < scores.size(); i++) {
    if (scores[i] == target) {
        cout << "Found at position " << i << "\n";
        break;
    }
}

break vs Condition-Based Exit

Both approaches work. The right choice depends on which is clearer.

Condition in header

while (value != -1) {
    cin >> value;
    process(value);
}

break inside body

while (true) {
    cin >> value;
    if (value == -1) break;
    process(value);
}

Prefer the condition-in-header form when possible — it makes the exit visible at the top of the loop.

Summary

break exits the nearest enclosing loop immediately. Use it when the exit condition cannot be cleanly expressed in the loop header, or when you need to stop a loop from inside the body.

Appendix

Common Practices

  • Prefer expressing the exit condition in the loop header when possible — while (value != -1) is clearer than while (true) with a break.
  • Use break intentionally — it should be obvious to the reader why the loop is exiting at that point.

Common Mistakes

  • Assuming break exits all loops — it only exits the innermost one.
  • Using break in a switch statement inside a loop expecting it to exit the loop — break exits the switch, not the loop.
  • Placing break where the loop condition would suffice, making the logic harder to follow.

More Examples

// Search a vector for the first negative number
vector<int> nums = {4, 7, -2, 9, -5, 1};
int pos = -1;

for (int i = 0; i < nums.size(); i++) {
    if (nums[i] < 0) {
        pos = i;
        break;
    }
}

if (pos != -1)
    cout << "First negative at position " << pos << "\n";
else
    cout << "No negatives found.\n";

What’s the Output?

Desk Check — Trace the Output

DO NOT type this in and run it.

#include <iostream>
using namespace std;

int main() {
    int total = 0;
    int i = 1;

    while (i <= 10) {
        if (i % 2 == 0) {
            i++;
            continue;
        }
        if (total > 15)
            break;
        total += i;
        i++;
    }

    cout << "Total: " << total << "\n";
    cout << "i: " << i << "\n";
    return 0;
}

Trace: i=1: odd, total>15? no, total=0+1=1, i=2 i=2: even → i=3, continue i=3: odd, total>15? no, total=1+3=4, i=4 i=4: even → i=5, continue i=5: odd, total>15? no, total=4+5=9, i=6 i=6: even → i=7, continue i=7: odd, total>15? no, total=9+7=16, i=8 i=8: even → i=9, continue i=9: odd, total>15? YES (16>15) → break

Output: Total: 16 i: 9

Instructor note: This exercise introduces continue (skips the rest of the body) without a formal slide — students will encounter it in the trace and ask about it. That is intentional. Brief explanation: continue sends execution back to the top of the loop immediately, skipping the rest of the body.

Debrief

  • What output did your pair get?
  • What does continue do? How is it different from break?
  • At what value of i did break fire — and why?

Walk through the trace. Key teaching moments: (1) continue skips the rest of the body and loops back — students often confuse it with break. (2) The break fires when total exceeds 15, not when i reaches some limit. (3) Note that i is still 9 after the break — the loop did not finish normally.
