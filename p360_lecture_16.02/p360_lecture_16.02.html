Object-Oriented Programming

Procedural vs. object-oriented

  • Procedural: organize code around actions — functions that receive data and produce results.
  • Object-oriented: organize code around things — objects that own their own data and know how to operate on it.

Total slides: 4 Bank account example: procedural thinks in verbs (withdraw, deposit, transfer); OOP thinks in nouns (Customer, Account, Money). Same program, completely different organizing lens. This paradigm shift is the core of what students are stepping into. [Graphic suggestion: Two-column diagram — left shows person silhouette with action arrows labeled withdraw/deposit/transfer; right shows same person surrounded by Customer, Account, Money objects circling her.]

The class is the blueprint

  • A class defines a new type — it specifies what data it holds and what operations it supports.
  • An object is one specific instance created from that class — like a cookie cut from a cutter.

Cookie cutter analogy: one class definition (the cutter) stamps out as many objects (cookies) as you need. Each object is independent — same shape, its own data. [Graphic suggestion: Cookie cutter labeled “the class” pointing to two cookie shapes labeled “the objects.”]

One class, many objects

class Car {
    // members defined here
};

Car mustang;   // Green, Ford, Mustang, Gasoline
Car prius;     // Red, Toyota, Prius, Electric
Car golf;      // Blue, Volkswagen, Golf, Diesel

Same class, three distinct objects — each holding different data. This mirrors what students already know from scalars: int x; int y; is the same type with different values. The class is declared once; objects are created as needed. [Graphic suggestion: Car class at top with arrows down to three car icons (green/red/blue), each labeled with make/model/fuel.]

Summary

  • OOP organizes programs around objects — things that own their own data and behavior.
  • Declare a class once; create as many objects as needed — just like any other type.

Introduction to Classes

A struct that knows how to act

  • A class is like a struct — it groups related data under one type name.
  • The key differences: class members are private by default, and a class can contain functions alongside its data.

Total slides: 5 Students know struct from 16.01. Class is the next step: same composite type concept, but with access control and member functions built in. The private-by-default rule is the single most important structural difference to hammer.

Class anatomy

class Point {
private:
    int x;
    int y;

public:
    void setX(int val);
    void setY(int val);
    int  getX() const;
    int  getY() const;
};
  • class — keyword; creates a new type
  • Point — the type name; use it like int or string
  • private: — these members are hidden from outside code
  • public: — these members are visible to callers
  • }; — closing brace must have a semicolon

Walk through each labeled part deliberately. The private/public split is the new concept relative to struct. Convention: data goes in private; functions external code needs go in public.

private vs public in action

Point p;

p.x = 10;       // compile error — x is private
p.setX(10);     // OK — setX() is public

Private means the outside world cannot read or write that member directly. Public methods are the controlled interface. This is encapsulation: the object decides how its data is accessed. Struct members are public by default — that’s the only syntactic difference between struct and class in C++.

A complete example

#include <iostream>
using namespace std;

class Point {
private:
    int x = 15;
    int y = 33;
public:
    void Print() {
        cout << "Object is located at " << x << ", " << y << "\n";
    }
};

int main() {
    Point widget1;
    widget1.Print();   // Object is located at 15, 33
    return 0;
}

x and y are private — outside code cannot touch them directly. Print() is public — callers reach it with dot notation, same as struct member access. widget1 is one object; declare widget2 and it gets its own independent x and y.

Summary

  • A class groups data (private) and behavior (public) under one type name.
  • Outside code interacts only through the public interface — the class controls its own data.

Creating and Using Objects

Three decisions before you write a class

  • What data does this type need? — becomes the private members.
  • What operations should it support? — becomes the public member functions.
  • What should be visible to outside code and what should stay hidden? — the public/private split.

Total slides: 4 This is 7.4, framed as a review extension of 7.3. Students have seen the Point class; now they think like designers. The three-question framework scales to any class they’ll ever write. Walk through it before showing code.

Creating and using objects

#include <iostream>
#include <string>
using namespace std;

class Book {
public:
    string title;
    void setTitle(string t)  { title = t; }
    string getTitle() const  { return title; }
};

Book is a class with one private-ish member (left public here for simplicity) and two inline methods. Students have seen this shape before with structs — the difference is methods instead of plain member access.

Objects interacting

class Library {
public:
    Book books[10];
    int  count = 0;
    void addBook(const Book& b)  { books[count++] = b; }
    void showBooks() const {
        for (int i = 0; i < count; i++)
            cout << "- " << books[i].getTitle() << "\n";
    }
};

int main() {
    Book b1, b2;
    b1.setTitle("C++ for Beginners");
    b2.setTitle("Learn Programming with C++");

    Library lib;
    lib.addBook(b1);
    lib.addBook(b2);
    lib.showBooks();
    return 0;
}

Library holds an array of Book objects and a count. addBook stores a Book; showBooks loops and calls getTitle() on each element. This is the interacting-objects pattern: one class uses another as a building block. main() creates both book objects and the library in the same scope — no cross-slide dependency.

Summary

  • Create objects the same way you create any variable: ClassName variableName;
  • Objects interact by passing each other as arguments and calling each other’s public methods.

Defining Member Functions

Inline or outside the class?

  • Inline: the function body is written inside the class definition — compact, good for short functions.
  • Outside: the class lists a prototype only; the body is defined below using ClassName::methodName().

Total slides: 5 Both forms compile and run identically — this is a style and organization choice. Small getters and setters are typically inline. Longer methods are defined outside to keep the class definition readable as an interface summary.

Inline definition

class StopLight {
private:
    bool isPowered = false;
public:
    bool getState() const { return isPowered; }  // inline — body here
    void powerOn();                              // prototype only
    void powerOff();                             // prototype only
};

getState() is short enough to live inline. powerOn() and powerOff() are declared with prototypes — their bodies will be written outside. The class definition now reads like a clean public interface.

Outside definition

void StopLight::powerOn() {
    isPowered = true;
}

void StopLight::powerOff() {
    isPowered = false;
}

int main() {
    StopLight bidwell;
    bidwell.powerOn();
    cout << bidwell.getState() << "\n";   // 1
    bidwell.powerOff();
    cout << bidwell.getState() << "\n";   // 0
    return 0;
}

The :: is the scope resolution operator. It tells the compiler which class this function belongs to. Without it, powerOn() would be a free function, not a member — and it wouldn’t have access to isPowered.

Accessors and mutators

class Temperature {
private:
    double degrees = 0.0;
public:
    double getDegrees() const  { return degrees; }    // accessor (getter)
    void   setDegrees(double d){ degrees = d; }       // mutator  (setter)
};

One key purpose of member functions is controlled access to private data. Accessor (getter): reads a private member — marked const because it doesn’t change the object. Mutator (setter): modifies a private member — can add validation logic before storing. Combined, they are the public interface to private data.

Summary

  • Member functions can be defined inline (inside the class body) or outside using ClassName::method().
  • Accessors (getters) read private data; mutators (setters) modify it — they are the controlled gateway to private members.

Arrays of Objects

An array where each element is an object

  • An array of objects uses the same bracket syntax as any other array — the element type is your class.
  • Each element gives you access to all of that object’s public methods via dot notation.

Total slides: 4 Students know arrays of structs from 16.01. This is the same pattern: Student roster[5] becomes Student roster[5] whether Student is a struct or a class. The new wrinkle is roster[i].method() — same two-step access, but now you’re calling a function, not reading a member directly.

Declaring and accessing

class Student {
public:
    string name;
    int    grade = 0;

    void   setName(string n)  { name = n; }
    void   setGrade(int g)    { grade = g; }
    string getName()  const   { return name; }
    int    getGrade() const   { return grade; }
};

int main() {
    Student roster[3];              // array of 3 Student objects

    roster[0].setName("Alice");     // [index].method()
    roster[0].setGrade(92);

    cout << roster[0].getName() << ": " << roster[0].getGrade() << "\n";
    return 0;
}

roster[0] is a Student object; roster[0].getName() calls a method on that object. The subscript selects the element; the dot selects the method. Identical pattern to arrays of structs — the method call replaces direct member access.

Looping through an array of objects

const int SIZE = 4;
Student   roster[SIZE];

roster[0].setName("Alice");  roster[0].setGrade(92);
roster[1].setName("Bob");    roster[1].setGrade(85);
roster[2].setName("Carol");  roster[2].setGrade(97);
roster[3].setName("Dan");    roster[3].setGrade(78);

int best = 0;
for (int i = 1; i < SIZE; i++)
    if (roster[i].getGrade() > roster[best].getGrade()) best = i;

cout << "Top student: " << roster[best].getName()
     << " with " << roster[best].getGrade() << "\n";

Max-tracker pattern is identical to arrays of structs — just replace roster[i].grade with roster[i].getGrade(). The loop logic is unchanged; only the syntax for reading the grade differs because it’s now behind an accessor.

Summary

  • ClassName array[n] — same syntax as any array; the class is the element type.
  • Access: array[i].method() — subscript first, then dot to call the method.

Fill In the Missing Code

Exercise: complete the BankAccount method

  • The BankAccount class and main are complete — the printSummary body is missing.
  • Fill in lines 33–35 so the program compiles and prints both accounts correctly.

Total slides: 3 Students need to write three cout lines using private members directly (they’re inside the class) plus one call to getBalance(). About 8 minutes — this is the capstone exercise for the last lecture. Encourage students to notice that printSummary is defined outside the class using ::, and that it can access private members because it’s a member function.

The code

#include <iostream>
#include <string>
using namespace std;

class BankAccount {
private:
    string owner;
    double balance = 0.0;
public:
    void   setOwner(string name)   { owner = name; }
    void   setBalance(double amt)  { balance = amt; }
    string getOwner()  const       { return owner; }
    double getBalance() const      { return balance; }
    void   deposit(double amt)     { balance += amt; }
    void   printSummary() const;
};

void BankAccount::printSummary() const {
    // === YOUR CODE HERE (lines 33–35) ===
    // Print: "Account: [owner]"
    // Print: "Balance: $[balance]"
    // Print: "After $500 deposit: $[balance + 500]"
    // =====================================
}

int main() {
    BankAccount a, b;
    a.setOwner("Alice");   a.setBalance(1200.00);
    b.setOwner("Bob");     b.setBalance(850.50);

    a.printSummary();
    cout << "\n";
    b.printSummary();

Solution

void BankAccount::printSummary() const {
    cout << "Account: "          << owner             << "\n";
    cout << "Balance: $"         << balance           << "\n";
    cout << "After $500 deposit: $" << balance + 500  << "\n";
}

Output for Alice: Account: Alice Balance: $1200 After $500 deposit: $1700

Output for Bob: Account: Bob Balance: $850.5 After $500 deposit: $1350.5

Key points to discuss: (1) printSummary accesses private members owner and balance directly — it’s a member function, so it has access. (2) The :: tells the compiler this belongs to BankAccount. (3) const after the signature means this method promises not to modify any members.

Key Takeaways

  • A class groups private data and public behavior under one user-defined type name
  • Outside code interacts only through the public interface — the class owns its data
  • Member functions defined outside the class use ClassName::method() syntax
  • Arrays of objects: ClassName array[n]; call methods with array[i].method()
