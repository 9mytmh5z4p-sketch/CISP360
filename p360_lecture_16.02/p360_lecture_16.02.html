What is a Constructor

The problem with setters

After writing a class, every object creation looks the same: declare, then call a chain of setters. Is there a better way to initialize an object at the moment it is born?

Total slides: 4 Students have been writing Student s; s.setName(“Alice”); s.setGrade(92); since 16.01. The constructor is the answer to the question this pattern raises: why can’t initialization happen at the declaration itself? Frame this as solving a real annoyance they have already felt.

The setter chain problem

// Current pattern — three lines to get one usable object
Student alice;
alice.setName("Alice");
alice.setGrade(92);

// With a constructor — one line, object ready immediately
Student alice("Alice", 92);
  • Every setter call is a chance for the object to exist in a partially initialized state.
  • A constructor guarantees all members are set before any other code can touch the object.

The partially-initialized object is a real bug source: if code between the declaration and the last setter call reads from the object, it sees garbage or defaults. The constructor closes that window entirely.

Constructor anatomy

class Student {
private:
    string name;
    int    grade;
public:
    Student(string n, int g) {   // constructor: same name as class, no return type
        name  = n;
        grade = g;
    }
};
  • Same name as the class — no exceptions.
  • No return type — not even void.
  • Called automatically — the moment an object is declared, the matching constructor runs.

The no-return-type rule trips students up. It is not void — void means the function exists but returns nothing. A constructor has no return type specification at all. The compiler enforces this: a function with the class name and a return type is not a constructor.

Summary

  • A constructor is a special member function with the same name as the class and no return type — it runs automatically when an object is created.
  • It replaces the setter chain by guaranteeing full initialization at the moment of declaration.

Default Constructor

What happens with no constructor at all?

If you write a class and define zero constructors, C++ quietly provides one for you. What does it do — and when does it disappear?

Total slides: 4 This topic exists to prevent a specific surprise: students add their first parameterized constructor and then find that Student s; no longer compiles. The implicit default is invisible until it vanishes. Name it, describe it, and show exactly when it goes away.

The implicit default constructor

class Student {
private:
    string name;
    int    grade = 0;
public:
    string getName()  const { return name; }
    int    getGrade() const { return grade; }
    // No constructor written — C++ provides: Student() {}
};

int main() {
    Student s;   // OK — implicit default runs, members get default values
}
  • C++ generates a zero-argument constructor automatically when no constructors are defined.
  • Members with in-class initializers (int grade = 0) use those values; others get zero-initialized for primitives or default-constructed for objects.

The implicit default does nothing special — it does not set name to “Unknown” or grade to -1. Whatever the member’s in-class initializer says (or zero/empty if none) is what the object contains after the implicit default runs.

The disappearing default

class Student {
private:
    string name;
    int    grade;
public:
    Student(string n, int g) { name = n; grade = g; }
    // C++ no longer provides Student() — you defined a constructor
};

int main() {
    Student alice("Alice", 92);   // OK
    Student blank;                // compile error — no default constructor
}
  • The moment you define any constructor, C++ stops providing the implicit default.
  • To keep zero-argument construction, write it yourself: Student() { name = "Unknown"; grade = 0; }

This is the single most common constructor-related compile error students will see. Show the exact error message so they recognize it: “no matching function for call to Student::Student()”. The fix is always the same — add an explicit default constructor.

Summary

  • C++ provides a free zero-argument constructor only when you define no constructors at all.
  • Once you write any constructor, you own the default — write it explicitly or lose zero-argument construction.

Parameterized Constructor

Can an object know its own data at birth?

A parameterized constructor accepts arguments at the moment of declaration — the object is fully initialized in a single line before any other code runs.

Total slides: 5 This is the main event of the lecture. Students already understand function parameters; applying that knowledge to constructors is the leap. Use the Student class they have been working with since 16.01 so the upgrade from setter chain to constructor call is immediately visible.

Syntax

class ClassName {
public:
    ClassName(type param1, type param2) {
        member1 = param1;
        member2 = param2;
    }
};

ClassName obj(value1, value2);   // arguments match constructor parameters
  • The parameter list works exactly like any function — types, names, order.
  • Arguments are passed at the declaration: ClassName obj(arg1, arg2).

Full example — upgrading Student

#include <iostream>
#include <string>
using namespace std;

class Student {
private:
    string name;
    int    grade;
public:
    Student(string n, int g) { name = n; grade = g; }
    string getName()  const  { return name; }
    int    getGrade() const  { return grade; }
};

int main() {
    Student alice("Alice", 92);
    Student bob("Bob",     85);

    cout << alice.getName() << ": " << alice.getGrade() << "\n";
    cout << bob.getName()   << ": " << bob.getGrade()   << "\n";
    return 0;
}

Copy and run. alice and bob are fully initialized at their declarations — no setter calls, no partially constructed state. The class interface is cleaner too: getName and getGrade are still the only public surface, the constructor just handles setup.

Before and after

// Before — setter chain; object exists but is incomplete on line 1
Student alice;
alice.setName("Alice");
alice.setGrade(92);

// After — constructor; object is complete on line 1
Student alice("Alice", 92);
// Works the same with arrays and vectors
Student roster[3] = {
    Student("Alice", 92),
    Student("Bob",   85),
    Student("Carol", 97)
};

vector<Student> v;
v.push_back(Student("Dan", 78));

The array initialization list shows that constructors work inside aggregate initializers — students can initialize an entire roster in one block. The vector push_back with a constructor call creates a temporary object inline, which is a natural pattern with vectors of objects.

Summary

  • A parameterized constructor takes arguments at the point of declaration and initializes all members before any other code runs.
  • Constructors compose naturally with arrays and vectors — objects can be fully initialized at the point of creation regardless of how they are stored.

Constructor Overloading

What if you want two ways to create the same object?

Sometimes you have all the data upfront; sometimes you want a default object to fill in later. Constructor overloading lets a single class support both — the compiler picks the right constructor from the argument list.

Total slides: 4 Students learned function overloading in lecture 5.02. Constructor overloading is the same mechanic applied to constructors. Keep this brief — the concept transfer is fast. Focus on the default-plus-parameterized pattern they will use constantly.

Same rules as function overloading

  • A class can have any number of constructors as long as each has a unique parameter list.
  • The compiler matches the call to the constructor whose parameters fit the arguments — same resolution rules as any overloaded function.
  • A zero-argument constructor and a parameterized constructor in the same class is the most common pattern.

No new syntax — just apply what students already know about overloading. The only wrinkle is the disappearing default from the previous topic: if you define a parameterized constructor but still need Student s;, you must write the default explicitly.

Default and parameterized together

class Student {
private:
    string name;
    int    grade;
public:
    Student()                    { name = "Unknown"; grade = 0; }  // default
    Student(string n, int g)     { name = n; grade = g; }         // parameterized
    Student(string n)            { name = n; grade = 0; }         // name only

    string getName()  const      { return name; }
    int    getGrade() const      { return grade; }
};

int main() {
    Student blank;                    // default
    Student alice("Alice", 92);       // parameterized
    Student named("Bob");             // name only

    Student roster[3];                // array of 3 default-constructed Students
}

Three constructors, three calling patterns. The array Student roster[3] uses the default constructor — each element starts as “Unknown”/0. This is why the default constructor matters even after you add parameterized ones.

Summary

  • A class can define multiple constructors with different parameter lists — the compiler selects the matching one at each declaration.
  • The most common pattern is a default constructor paired with one or more parameterized constructors.

Destructor

If a constructor runs when an object is born, what runs when it dies?

Every object has a defined moment when it goes out of scope or is explicitly deleted. The destructor runs at that exact moment — automatically, with no call needed.

Total slides: 4 Students have done new/delete in lectures 14.01–14.02, so object lifetime is not a new concept. The destructor names the automatic cleanup mechanism they have been invoking implicitly. Keep this introductory — CISP 400 covers destructors in depth (resource management, rule of three/five). The goal here is recognition and basic understanding.

Destructor syntax

class ClassName {
public:
    ClassName()  { /* constructor */ }
    ~ClassName() { /* destructor  */ }   // tilde prefix, no parameters, no return type
};
  • Tilde prefix (~) — the only syntactic difference from a constructor.
  • No parameters, no return type — a class can have only one destructor; it cannot be overloaded.
  • Called automatically — you never call it directly; the runtime calls it when the object’s lifetime ends.

When destructors run

class Logger {
public:
    Logger()  { cout << "Logger created\n"; }
    ~Logger() { cout << "Logger destroyed\n"; }
};

int main() {
    Logger a;              // "Logger created"
    {
        Logger b;          // "Logger created"
    }                      // "Logger destroyed" — b goes out of scope here
    return 0;              // "Logger destroyed" — a goes out of scope here
}

// With dynamic allocation:
Logger* p = new Logger;    // "Logger created"
delete p;                  // "Logger destroyed" — destructor runs at delete

Run this live if possible — watching “Logger destroyed” print at exactly the closing brace is memorable. The key message: for stack objects, you never have to think about cleanup. For heap objects (new/delete), delete triggers the destructor before releasing the memory. This is why forgetting delete causes resource leaks even when a destructor is defined.

Summary

  • A destructor is defined with a tilde prefix, takes no parameters, and runs automatically when an object’s lifetime ends — at scope exit for stack objects, at delete for heap objects.
  • Use a destructor to release any resources the constructor acquired; for simple data classes, you usually need none.

Key Takeaways

  • A constructor has the same name as the class, no return type, and runs automatically at object creation
  • C++ provides a free default constructor only when you define no constructors — once you define any, write the default explicitly if you still need it
  • Parameterized constructors replace the setter chain and guarantee full initialization at the point of declaration
  • A destructor (~ClassName()) runs automatically when an object goes out of scope or is deleted

Appendix

What is a Constructor: Good Practices

  • Always initialize every member in the constructor body — an object should never leave its constructor in a partially initialized state.
  • Name constructor parameters to make call sites readable: Student("Alice", 92) is clear only if the constructor signature documents that order.
  • Keep the constructor body short; if initialization logic is complex, delegate to a private helper method.

What is a Constructor: Common Mistakes

  • Writing a return type on a constructor — even void — turns it into a regular member function with the same name; it will not run at object creation.
  • Confusing the constructor parameter names with the member names — if both are name, the assignment name = name does nothing useful; use distinct names or the this-> prefix.
  • Assuming the constructor is called again when you assign one object to another — assignment is not construction; the constructor only runs at the point of declaration.

What is a Constructor: More Examples

class Rectangle {
private:
    double width;
    double height;
public:
    Rectangle(double w, double h) { width = w; height = h; }
    double area()   const         { return width * height; }
    double perimeter() const      { return 2 * (width + height); }
};

int main() {
    Rectangle r(4.0, 7.5);
    cout << "Area: "      << r.area()      << "\n";   // 30
    cout << "Perimeter: " << r.perimeter() << "\n";   // 23
}

Default Constructor: Good Practices

  • Always write an explicit default constructor when you also define parameterized constructors — the implicit one disappears and ClassName obj; will stop compiling.
  • Use in-class member initializers (int grade = 0;) as a first line of defense; the default constructor body can then remain empty.
  • Name the default state meaningfully — "Unknown" or 0 — so a default-constructed object is recognizably incomplete rather than silently garbage.

Default Constructor: Common Mistakes

  • Defining a parameterized constructor and then being surprised that ClassName arr[n]; no longer compiles — array declarations call the default constructor for every element.
  • Writing Student s(); to call the default constructor — this is parsed as a function prototype, not an object declaration; write Student s; with no parentheses.
  • Relying on zero-initialization of string members in the implicit default without understanding that int and double members may contain garbage on some compilers without in-class initializers.

Default Constructor: More Examples

class Point {
private:
    double x;
    double y;
public:
    Point()             { x = 0.0; y = 0.0; }      // explicit default
    Point(double a, double b) { x = a; y = b; }    // parameterized

    double getX() const { return x; }
    double getY() const { return y; }
};

Point origin;           // default  — (0, 0)
Point p(3.0, 4.0);      // parameterized — (3, 4)
Point grid[10];         // ten default-constructed Points — all (0, 0)

Parameterized Constructor: Good Practices

  • List parameters in the same order that callers will naturally think about them — Student(name, grade) reads left-to-right like a form field.
  • Validate inputs in the constructor body the same way a setter would — the constructor is the first place to enforce class invariants.
  • When creating arrays or vectors of objects with parameterized constructors, use initialization lists to keep declarations compact and readable.

Parameterized Constructor: Common Mistakes

  • Passing arguments in the wrong order — Student(92, "Alice") compiles if the types match a different constructor, silently storing the grade as a name.
  • Forgetting that vector<Student> v(5) requires a default constructor — it tries to create 5 default-constructed elements, not 5 uninitialized slots.
  • Shadowing member names with parameter names: void Student(string name, int grade) { name = name; } assigns the parameter to itself; use distinct names.

Parameterized Constructor: More Examples

// Constructor with validation
class Temperature {
private:
    double degrees;
public:
    Temperature(double d) {
        degrees = (d < -273.15) ? -273.15 : d;   // clamp to absolute zero
    }
    double get() const { return degrees; }
};

// Vector populated with constructor calls
vector<Student> roster;
roster.push_back(Student("Alice", 92));
roster.push_back(Student("Bob",   85));
roster.push_back(Student("Carol", 97));

Constructor Overloading: Good Practices

  • Always pair a parameterized constructor with an explicit default constructor — code that creates arrays or default-initializes objects will break otherwise.
  • Keep the number of constructors small; if you find yourself writing many variations, consider whether default parameter values would reduce duplication.
  • Document what each constructor represents — a zero-argument constructor should produce a meaningful “empty” state, not silent garbage.

Constructor Overloading: Common Mistakes

  • Defining two constructors that are ambiguous for a given call — Student(string) and Student(string, int = 0) both match Student("Alice") and the compiler will reject the ambiguity.
  • Writing a default constructor that leaves members uninitialized — Student() {} with no assignments means name is empty and grade is garbage on some compilers.
  • Expecting constructor overloading to work differently from function overloading — the same resolution rules apply; when in doubt, trace through the argument types manually.

Constructor Overloading: More Examples

class Point {
private:
    double x, y;
public:
    Point()                   { x = 0; y = 0; }        // origin
    Point(double a, double b) { x = a; y = b; }        // specific location
    Point(double a)           { x = a; y = a; }        // diagonal (x == y)

    double getX() const { return x; }
    double getY() const { return y; }
};

Point origin;         // (0, 0)
Point p(3.0, 4.0);    // (3, 4)
Point diag(5.0);      // (5, 5)

Destructor: Good Practices

  • If a class allocates heap memory with new, always write a destructor that calls delete — forgetting it leaks memory every time an object is destroyed.
  • For classes that only hold value-type members (strings, ints, vectors), the compiler-provided implicit destructor is sufficient; do not write an empty one.
  • Never call the destructor directly — let the runtime invoke it at the correct moment; calling it manually can cause a double-free crash.

Destructor: Common Mistakes

  • Forgetting to write a destructor for a class that heap-allocates — the memory is leaked every time an object goes out of scope or is deleted.
  • Writing a destructor that calls delete on a pointer that was never initialized or was already freed — always set pointers to nullptr after deletion.
  • Trying to overload the destructor — there is exactly one destructor per class; no parameters, no overloads.

Destructor: More Examples

// Destructor releasing heap memory
class DynamicArray {
private:
    int* data;
    int  size;
public:
    DynamicArray(int n) { size = n; data = new int[n]; }
    ~DynamicArray()     { delete[] data; data = nullptr; }
    int  getSize() const { return size; }
};

{
    DynamicArray arr(100);   // allocates 100 ints on the heap
}                            // ~DynamicArray() runs here — memory freed automatically
